# MLID-2417 — Notifications & SignalR for Maintenance Orders (Pre-Plan)

> This is a **pre-plan**, not the implementation plan. It is the briefing that the real plan (and the resulting Jira sub-tasks) will be built from once approved. We are crafting this file collaboratively with the user; the Jira description is intentionally not the source of truth here.

## 1. Epic summary

- **Jira**: [MLID-2417](https://localinfusion.atlassian.net/browse/MLID-2417)
- **Scope**: Port the notifications + SignalR event implementation from new orders to **maintenance orders**.
- **Collection in scope**: `maintenanceorders` (not `neworders`, not `lead`).

## 2. Context from the user

### 2.1 Relationship to MLID-2011

MLID-2417 is **almost identical to MLID-2011**. The relevant differences:

- **MLID-2011** was the **first** implementation of notifications + SignalR events in the app. Because it was first, it required creating the foundation:
  - The SignalR service layer.
  - The notifications schema.
  - All the supporting plumbing for both.
  - Its collection target was `neworders`.
- **MLID-2417** is the **same feature** (notifications + SignalR events) but applied to **`maintenanceorders`**. The foundation (SignalR service, notifications schema, etc.) is already in place from MLID-2011 and should be reused, not rebuilt.

In other words: 2417 is the second consumer of the foundation that 2011 created. The work is mostly wiring maintenance-order surfaces into the existing notification/SignalR machinery, not net-new infrastructure.

### 2.2 Caveat — QA edge cases from MLID-2011

After MLID-2011 was released, QA found **unexpected edge cases that were not covered in the original implementation**. MLID-2417 must address those edge cases as well — both in the new maintenance-order wiring and, where applicable, retroactively in the new-order wiring that MLID-2011 produced.

The specific edge cases still need to be enumerated in this pre-plan (TBD with the user).

### 2.3 Delivery strategy — D0 is a hotfix, D1–D4 is a feature

The deliverables split into two distinct streams with different urgencies and engineering shapes:

- **D0 — hotfix on new orders**. Bug fixes against code that already shipped in MLID-2011. May need to land in the **current sprint** to repair live behavior. Self-contained scope: reassignment and unlink edge cases on `neworders` only. Should be deliverable independently of D1–D4.
- **D1–D4 — new feature for maintenance orders**. Extends existing code paths (the propagation orchestrator, `getOrdersByPatient`, the orders-tracker table column factories, the order-detail layout, the link-orders route) so they cover both `neworders` and `maintenanceorders`. Expect some refactoring as part of this — for example, parameterizing collection scope on shared helpers rather than branching on collection name at each call site.

Three practical consequences for the implementation plan:

1. **D0 branches off `develop`, not off an epic branch.** D0-T1 (MLID-2391) and D0-T2 (MLID-2413) each branch directly from `develop`, ship as independent hotfixes, and merge back to `develop` on their own timeline. They can't wait for the rest of MLID-2417 to be ready. (Jira parenting under MLID-2417 is a tracking/organization choice; it does not imply git branch parenting.)
2. **D0 ships first**, even though the shared helpers (`handleOrderReassignment`, `handleOrderUnlink`) get a second pass during D1–D4. The hotfix introduces them in their `neworders`-only form; D1–D4 widens the scope. Even where D0 and D4 share a helper name (D0-T2 ↔ D4-T4, D0-T1 ↔ D4-T5), they remain separate Jira tickets and separate PRs because the timelines diverge — D0 is an urgent hotfix and D4 is feature/refactor work.
3. **Refactor budget for D1–D4** is part of the task estimates — these tasks are not pure additions; they touch existing code paths the new-orders feature already depends on. Tests on `neworders` behavior must continue to pass after each maintenance-orders extension.

## 6. Notification propagation — definition (from MLID-2011 code)

This section describes what "notification propagation" is in the existing implementation (new orders), so the maintenance-orders port has a clear reference.

### 6.1 What it is

**Notification propagation** is the act of, for a given `(order, document)` pair, writing a durable notification, writing an audit log entry on the order, and broadcasting a SignalR event — atomically per pair, with isolated error handling for each step.

The orchestrator is `propagateDocumentNotificationToOrder` in `apps/web/services/notifications/documentNotification.ts:156`. Three public entry points wrap it for the three fan-out shapes that exist today.

### 6.2 What it does — per `(order, document)` pair

Each propagation call performs these four steps in order; each step has its own `try/catch` so a failure in one does not abort the others:

1. **Eligibility gate** — `isDocumentEligibleForOrder(document, order)`. Documents with `category === 'Order'` are blocked unless `document.linkedOrderIds` includes this orderId. All other categories pass through. (This is what makes "Order"-category uploads silent — they only propagate when explicitly linked.)

2. **Durable notification write** — only when `order.assignedTo` is truthy. Calls `createNotification(...)` writing a `notifications` row with:
   - `type: 'user'`, `priority: 'normal'`
   - `targetUserId: order.assignedTo` (captured at write time — can go stale on reassignment)
   - `title: "New Document: <category>"`
   - `message: "A new document was added via <Upload|E-Order|Fax>."`
   - `entities: [{ entityType: 'order', entityId }, { entityType: 'document', entityId }]`
   - `createdBy: 'system'`

3. **Audit log write** — `writeOrderDocumentAuditEntry` records `action: 'update'`, `field: 'documents'`, `from: null`, `to: { _id: documentId, category, source }` on the order entity.

4. **SignalR broadcast** — `signalr.broadcastToAll('documentAdded', { documentId, orderId, source, category })`. Goes to **every connected client** (no per-user filtering server-side); the client hooks drop events that aren't relevant to the current view.

### 6.3 Public entry points (the three fan-out shapes)

| Entry point | Fan-out direction | Used by |
|---|---|---|
| `notifyDocumentCreated(document, { source, actorId, auditSource })` | One document → many orders for the same patient | Upload paths (intakes review, e-order, fax) |
| `notifyOrderOfRecentDocuments(order, { auditSource, actorId })` | One new order → all documents from the last 30 days for that patient | `POST /api/orders/new` backfill |
| `firePropagationForNewlyLinkedOrders(document, newlyLinkedOrderIds, { actorId, auditSource })` | One document → the *delta* set of newly-linked orders | `PUT /api/documents/[documentId]/link-orders` |

All three early-return when the `ORDER_DOCUMENT_NOTIFICATIONS` feature flag is off.

### 6.4 Events that trigger propagation today (new orders)

Five call sites total, all in the new-orders surface:

| # | Trigger event | Code location | Entry point used |
|---|---|---|---|
| 1 | Document upload via **intakes review** (split-pdf path A) | `apps/web/app/api/documents/split-pdf/route.ts:358` | `notifyDocumentCreated` (`source: 'upload'`) |
| 2 | Document upload via **intakes review** (split-pdf path B) | `apps/web/app/api/documents/split-pdf/route.ts:624` | `notifyDocumentCreated` (`source: 'upload'`) |
| 3 | Document upload via **e-order / fax** (background job) | `apps/web/services/jobs/definitions/jotFormSubmission/weinfuseUploader.ts:343` | `notifyDocumentCreated` (`source: 'e-order' \| 'fax'`) |
| 4 | **New order created** — backfill recent docs | `apps/web/app/api/orders/new/route.ts:154` | `notifyOrderOfRecentDocuments` |
| 5 | **Document linked** to one or more orders (delta only) | `apps/web/app/api/documents/[documentId]/link-orders/route.ts:90` | `firePropagationForNewlyLinkedOrders` |

### 6.5 What is *not* triggered today (gaps to address in MLID-2417)

These are the gaps that MLID-2417 closes — they have **no code path** today (mapped to deliverables in section 4 / implementation in section 5):

- **Order reassignment** (D0-T1 for new orders, D4-T5 for maintenance) — `PUT /api/orders/new` updates `assignedTo` but does not touch existing notifications. `targetUserId` remains the previous assignee. (The `isCallerOrderAssignee` fallback in read-side authorization papers over this for mark-as-read, but the inbox-by-user query still shows stale ownership.)
- **Document unlink** (D0-T2 for new orders, D4-T4 for maintenance) — `PUT /api/documents/[documentId]/link-orders` only fires for the *added* delta. Removed orderIds leave their notifications orphaned in `notifications` (and counted in badges).
- **Maintenance-orders coverage** (D4-T1, T2, T3) — the upload path's `getOrdersByPatient({ expandRefs: false })` queries only the `neworders` collection (`ordersTracker.ts:1152-1159`). Maintenance orders linked to the same patient receive no notification on upload. The link-orders path is the only code that touches `maintenanceorders` today (falls back to it after trying `new`).

### 6.6 SignalR events the system emits

| Event name | Emitted from | Payload | Audience |
|---|---|---|---|
| `documentAdded` | `propagateDocumentNotificationToOrder` (step 4) | `{ documentId, orderId, source, category }` | All connected clients |
| `notification:read` | `PATCH /api/notifications/:id/read` | `{ notificationId, userId }` | All connected clients |
| `notification:read` | `PATCH /api/notifications/bulk-read` | `{ notificationIds, userId }` | All connected clients |

Hub name: `lihub`. Connection string: Azure Key Vault, secret `li-prod-signalr-primary-connection-string` (prod) or `li-stage-signalr-primary-connection-string` (everything else), overridable via `SIGNALR_KV_SECRET_NAME`.

### 6.7 Feature flag

`FeatureFlag.ORDER_DOCUMENT_NOTIFICATIONS` (`apps/web/utils/featureFlags/featureFlags.ts:138`), default `false`. Gates all five API routes, all three propagation entry points, and all UI surfaces (table badges, tab badge, "New" chips, mark-as-read buttons).

**Decision (2026-05-25):** **reuse `ORDER_DOCUMENT_NOTIFICATIONS`** for maintenance-orders work. Do not introduce a parallel `ORDER_DOCUMENT_NOTIFICATIONS_MAINTENANCE`. The existing flag already gates the orchestrator, all API routes, and all UI surfaces — extending its scope to cover maintenance is sufficient and avoids the operational cost of managing two flags whose lifecycles are effectively the same.
