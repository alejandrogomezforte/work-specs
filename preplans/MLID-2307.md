### What the PO is asking for

Add a new sub-tab **"Financial Review"** inside the Dashboard area of the Order Details v2 page. The tab is a **patient-responsibility calculator** with two sub-views:

1. **List** — saved calculations for the current order, latest first. Each row is a `CalculationCard` that **expands in place** to reveal the breakdown (inputs grid + summary panel + drug/admin split bar). Includes a copy-summary action per row.
2. **New** — a calculator form (deductible / coinsurance / OOP / drug+admin costs / copay assistance) with a live "Live Estimate" sidebar, save action.

> **Note**: a `CalculationDetail.tsx` component and a `/financial/[calcId]` route handler exist in the playground source, but **nothing in the playground UI navigates to them** — verified via grep. They are unreachable artifacts, not a designed flow. **Out of scope** for this ticket. If we later need deep-linking to a saved calc, that's a separate ticket.

**Scope guardrail (explicit from PO)**: build the UI only with fake/mock data. Real data wiring comes later as a separate ticket. Do **not** add Mongoose models, API routes, or pulse jobs.

### Key types (from `financialCalculations.ts`)

```ts
interface CalculationInputs {
  deductibleMet: number; deductibleMax: number;
  coinsurance: number;                    // percent, 0–100
  oopMet: number; oopMax: number;
  drugCost: number; adminCost: number;
  drugAssist: number; adminAssist: number;
}

interface CalculationResult {
  drugResp: number; adminResp: number; totalResp: number;
  summaryLines: string[];
}

interface FinancialCalculation {
  id: string; orderId: string;
  createdAt: string;                      // MM/DD/YYYY (string, not Date)
  author: string;                         // user email
  inputs: CalculationInputs;
  result: CalculationResult;
}
```

### Calculation algorithm (verbatim from playground, must port unchanged)

1. `remainingDeductible = max(0, deductibleMax - deductibleMet)`
2. Apply remaining deductible to drug cost first, then admin cost.
3. Apply `coinsurance%` to whatever cost remains after deductible (per leg).
4. If pre-assistance total > remaining OOP, scale both legs proportionally to cap at OOP.
5. Subtract `drugAssist` / `adminAssist` per leg, clamp to ≥ 0.
6. Generate `summaryLines[]` based on what was applied.

The function is pure and trivially unit-testable — this is where TDD bites first.

## 4. Architecture decisions for the real plan

### 4.1 Routes — use nested folders (mirror clinical-reviews)

Path is `/orders-tracker/[category]/[orderId]/financial-review/{,/new}`. Two `page.tsx` files (`'use client'`), each pulling `useParams()`, `useOrderDetails()`, building `basePath`, and rendering a `_components/` partial. No catch-all segments.

**Justification**: This matches the established clinical-reviews pattern (minus the `[reviewId]` segment — Financial Review has no standalone detail route per playground design). Catch-all `[[...slug]]` would diverge from precedent for no benefit.

### 4.2 Fake data strategy — local hook, no service, no API

Create one module under `_components/` exporting:

- `seed.ts` — exported `FinancialCalculation[]` (typed); same structure as playground's seed.
- `useFakeFinancialCalculations.ts` — a tiny hook with:
  - `useFinancialCalculationsForOrder(orderId)` → memoized filter of the seed (re-export shape so list/detail components stay decoupled).
  - `useAddFinancialCalculation()` → mutates a `useState`-backed array (in-memory, lost on refresh — that's fine for mock).
  - Optionally wrap in a tiny React Context if `Calculator`'s save needs to be visible to `Index` without a refetch.

**Important**: Do **not** store anything in localStorage or sessionStorage. Keeping it pure in-memory makes the eventual swap to a real service trivial (just replace the hook body).

**Important**: Do **not** create a Mongoose schema, API route, or types file under `apps/web/types/`. The types live next to the hook in `_components/`. When a future ticket wires real data, that's when types graduate.

### 4.3 Feature flag — `ORDER_FINANCIAL_CALCULATION`, default `false`

Belt-and-suspenders gating:
- **Tab visibility** in `OrderClinicalReviewsPanel.tsx` (UI hide).
- **Route gating** in `layout.tsx` `{children}` block (URL hide — typing the URL when flag is off renders nothing).

Both must be wired; UI-only gating would leave the URL reachable in production. This matches the RBAC guidance in `CLAUDE.md`: never rely solely on hiding UI elements.

### 4.4 Component structure — port playground, adapt to our conventions

| Playground file/symbol | Our destination |
|---|---|
| `FinancialReviewIndex.tsx` | `_components/FinancialReviewIndex.tsx` |
| `CalcRowCard` (inside `FinancialReviewIndex.tsx`) | **Extract** to `_components/CalculationCard.tsx` (rename) — it owns the collapsed row **and** mounts `<CalculationBreakdown compact />` when expanded |
| `Calculator.tsx` | `_components/Calculator.tsx` |
| `CalculationBreakdown.tsx` | `_components/CalculationBreakdown.tsx` — keep only the compact-mode code path; drop the dark-teal headline card branch (no longer used since detail page is out of scope) |
| `CalculationDetail.tsx` | **Skipped** — not in playground UX flow |
| `UserLine.tsx` (in `components/`) | Inline a 5-line equivalent in `_components/`, or import existing if a sibling already exists (verify during planning) |
| `data/financialCalculations.ts` | Split: `_components/calculate.ts` (pure fn + types) + `_components/seed.ts` (data) + `_components/useFakeFinancialCalculations.ts` (hook) |

The playground's `Calculator.tsx` uses an `ESC to go back` keyboard handler — keep it. The save-and-navigate flow uses `useNavigate()` from react-router; ours uses `useRouter` from `next/navigation`.

The playground's `addCalculation()` mutates a module-level array — that pattern is **not portable** to Next.js (SSR-safe state must live in React). Replace with a hook-backed `useState` shared via Context within the `financial-review/` tree.

### 4.5 TDD ordering — start with `calculate.ts`

This is the only pure function in the task. The 4 seed scenarios in the playground give us 4 ready-made test fixtures with expected outputs (e.g. zero-coinsurance no-deductible case → `totalResp: 0`; deductible-still-pending case → `drugResp: 0, adminResp: 64, totalResp: 64`). Write those as red tests first, then port the function, then it goes green. Refactor afterward (extract sub-functions if readable).

After `calculate.ts` is green, do component tests with React Testing Library:
- List shows the right count and the "Latest" chip on the first row.
- Expand-toggle reveals/hides breakdown.
- Copy button puts the right multi-line string on `navigator.clipboard` (mock it).
- Calculator "Save" button is disabled when all inputs are empty.
- Calculator's live estimate matches `calculate()` output for a representative input set.
- Empty-state shows when no calcs for the order.

### 4.6 Edge cases to bake in (from re-reading the playground)

- **Decimals only**: input mask strips everything except `\d.` — keep that to avoid Locale parse bugs.
- **OOP cap scaling**: when pre-assistance total exceeds remaining OOP, both legs are scaled proportionally. This is non-obvious and needs a test.
- **Assistance overflow**: if admin assistance > admin responsibility, the overflow is **not** moved to drug — it's just lost. Test this.
- **Zero-deductible + zero-coinsurance + non-zero costs** → `totalResp: 0`, special summary lines. Test this.
- **Drug-vs-admin split bar**: only rendered when both legs > 0. Don't crash on `0/0` divisions.

## 6. Risks / non-obvious things

- **Layout owns dashboard rendering**: `OrderClinicalReviewsPanel` is rendered *by the layout*, not via `children`. Adding a "Financial Review" tab is two changes (panel tab list **and** layout child gate), not one. Easy to miss.
- **`activeReviewTab` is hardcoded** to `'clinical'` in the current panel — selecting Financial Review requires URL-driven tab state (`usePathname`-based derivation). Otherwise the tab visually never highlights.
- **Calculator state lifetime**: if user clicks "Save", we navigate back to the list — but in-memory state from a hook doesn't survive a route change unless lifted to a Context provider mounted higher than the routes. Either mount the provider in `financial-review/layout.tsx` (new file) or accept that "fake save" disappears on navigation. Recommend: tiny `layout.tsx` with a context provider.
- **Husky pre-commit requires `MLID-XXXX`** — fine, ticket key is real.
