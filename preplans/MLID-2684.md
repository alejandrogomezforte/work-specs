# [MLID-2684] fix(prior-auth): preview shows whole fax instead of parsed prior-auth document

Pre-plan / effort analysis. Not an approved implementation plan yet.

## 1. The actual bug (from the ticket)

MLID-2684 is a **High priority bug**, not a data-model migration:

- On the Patient card Prior Auth tab, clicking the Preview icon shows **the whole fax**.
- On Order Details (Documents tab), the same auth document shows **only the parsed prior-auth document**.
- Expected: both the Patient card and the Order Details Auth Review preview must show **only the parsed prior-auth document**, matching what the Documents tab already shows.

So the request "adapt the code to show and edit objects from documents instead of intakes" is really the proposed **fix approach** for a preview-source bug. This document analyzes what that would take and recommends the smallest correct change.

## 2. Confirmed root cause

When a fax is split into individual documents, `POST /api/documents/split-pdf` (`apps/web/app/api/documents/split-pdf/route.ts`):

1. Creates a **new PDF blob containing only the parsed prior-auth pages** at a path like `intakes/{yyyy}/{mm}/{dd}/{intakeId}/authorization_pages-2-3.pdf`.
2. Pushes that split blob as a **new entry** in `intake.documents[]` (with `type: documentType`, `pdfDocument` = the split blob URL).
3. Creates a row in the **`documents` collection** with `blobPath` = the split blob and `category` = "Prior Authorization" (mapped from the `authorization` document type).
4. Leaves `intake.documents[0]` as the **original whole/consolidated fax** (the referral copy).

The prior-auth transform, `transformIntakeToPriorAuthLetter` in `apps/web/services/priorAuth/transforms.ts`, builds the preview URL from:

- `intake.documents?.[0]?.pdfDocument` → this is index 0 = **the whole fax**.
- `documentPage` from `additionalInfo.documentDefinitions[].pageRanges[0].from` → a page number relative to the whole fax.

The Documents tab instead reads the `documents` collection, whose prior-auth row `blobPath` points at the split file — hence it correctly shows only the parsed document.

**Both prior-auth surfaces share this one transform**, so the same wrong preview appears on the Patient card list, the Patient card detail wizard, and the Order Details Auth Review screens.

## 3. Where the preview URL flows (shared code, both screens)

The Patient card and the Order Details Auth Review reuse the same service layer, so a fix in the shared transform corrects both screens at once:

- Transform (single source of the preview URL): `transformIntakeToPriorAuthLetter` in `apps/web/services/priorAuth/transforms.ts` sets `documentUrl` and `documentPage` on `PriorAuthLetter`.
- Detail service: `getPriorAuthDetail` in `apps/web/services/priorAuth/detail.ts` returns `documentUrl` / `documentPage` (from the transform) plus already fetches the matching `documents` rows via `getDocumentsByIntakeId(intakeId)` for the description. This means the `documents` collection is already joined in the detail path.
- Patient list route: `apps/web/app/api/patient/[patientId]/prior-auth/route.ts` (maps intakes through the transform).
- Patient detail route: `apps/web/app/api/patient/[patientId]/prior-auth/[intakeId]/route.ts`.
- Order list route: `apps/web/app/api/orders/new/[orderId]/auth-review/route.ts`.
- Order detail route: `apps/web/app/api/orders/new/[orderId]/auth-review/[letterId]/route.ts`.
- Consuming UI (no change needed if the DTO field stays the same):
  - Patient list preview: `PatientPriorAuth` in `apps/web/components/Modules/patient/components/patientPriorAuth/PatientPriorAuth.tsx` (uses `letter.documentUrl` / `letter.documentPage` with `EmbeddedFileViewer`).
  - Order list/detail: `AuthReviewIndex.tsx` and `AuthReviewDetail.tsx` under `apps/web/app/orders-tracker/[category]/[orderId]/auth-review/_components/` (consume the same `PriorAuthLetter` / `PriorAuthDetailResponse` DTOs).

## 4. What "show from documents" vs "show and edit from documents" really costs

The key finding: **the `documents` collection carries the correct parsed-file pointer (`blobPath`), but it does NOT carry any of the prior-auth review data.** All of the OCR data (`additionalInfo.ocrData`) and all of the user-edited verify data (`additionalInfo.priorAuthData`, including the 19 `VerifyDataFormPersisted` fields, letter type, review status, reprocessing state) live only on the Intake.

`documents` (`apps/web/models/Document.ts`) has: `intakeId`, `patientId`, `category`, `description`, `blobPath`, `pageCount`, OCR/embedding status, `linkedOrderIds`, `createdAt`. It has **no** review status, letter type, verify-data fields, OCR letter schema, reprocessing block, or workflow status.

That splits the request into two very different scopes:

- **Show** the parsed document → only needs the correct `blobPath`, which `documents` already has. Small.
- **Edit** from documents → would require moving the entire prior-auth read/write model onto `documents` (schema additions, pipeline rewrite, Postgres dual-write story, reprocess-job re-keying). Large and unnecessary for this bug.

## 4b. Data evidence (queried against `local-infusion-db`, 2026-07-08)

Confirmed against the live staging database:

- Distinct `documents.category` values that are prior-auth: **"Prior Authorization"** (288 rows) and "Insurance Authorization" (30 rows). **Decision: use `category === "Prior Authorization"` as the single match** (the canonical value the split flow writes via `mapDocumentTypeToWeInfuse`).
- Cardinality by `intakeId` for `category === "Prior Authorization"` / "Insurance Authorization": **316 intakes have exactly one** prior-auth document, **1 intake has two**. So the join is effectively one-to-one, with a single edge case that needs a tiebreaker.
- Live proof on intake `6971551cdced1dbf89f5926a`:
  - `intake.documents[0]` = `type: "Fax"`, `.../Fax.pdf` — the whole fax (what the preview shows today).
  - `intake.documents[3]` = `type: "Insurance Authorization"`, `.../insurance-authorization_pages-4-4.pdf` — the parsed letter.
  - `documents` row for the same `intakeId` = `category: "Prior Authorization"`, `blobPath: .../insurance-authorization_pages-4-4.pdf` — the same parsed file. `documentDefinitions` places the letter on page 4 of the fax, matching the `pages-4-4` split.
- Validated on the ticket's own order (`6a424dc1febab4f6ff88dfd0`, TREMFYA #05wr — 8 documents, exactly one is `category: "Prior Authorization"`, description "Auth test"):
  - `documents.findOne({ intakeId: "6a42478cc910f40776039e19", category: "Prior Authorization" })` returns that one row, `blobPath: .../6a42478cc910f40776039e19/insurance-authorization_pages-1-1.pdf`, `pageCount: 1` — exactly the file the preview must show.

### Join key: `intakeId`, NOT `linkedOrderIds`

On that order, only the `Order` document carries `linkedOrderIds` for the order; the other 7 (including the "Auth test" prior-auth doc) surface on the Order Documents tab purely because they share the same `intakeId` (`6a42478cc910f40776039e19`). So the resolver must join `documents` by the row's intake `_id`, never by `linkedOrderIds`. Each prior-auth list/detail row already corresponds to one intake, so `String(intake._id)` is the join value on both the patient and the order surfaces.

### "Prior Authorization" vs "Insurance Authorization" — same document, different label

They are the same underlying document (the prior-auth letter), not two kinds. In `apps/web/app/api/documents/split-pdf/route.ts`, the raw document type "Insurance Authorization" both becomes the filename slug `insurance-authorization` and maps through `mapDocumentTypeToWeInfuse` to the canonical category **"Prior Authorization"**. That is why the "Auth test" row above has `category: "Prior Authorization"` yet a filename of `insurance-authorization_pages-1-1.pdf`. The 30 rows stored with category literally "Insurance Authorization" are the same document mislabeled with the raw type (the review UI can pass a category through verbatim). **Locked decision: match `category === "Prior Authorization"` only** — the canonical, current value. The consequence is measured below; the ~25 "Insurance Authorization"-labeled rows hit the fallback rather than being shown parsed, and can be relabeled separately.

### Fallback sizing (measured across all 295 list intakes)

- 260 (88.1%) have a `category === "Prior Authorization"` document → join hit, parsed preview.
- 35 (11.9%) have no such row → fallback. Of those: 25 have a parsed doc labeled "Insurance Authorization" (would be recovered if the match were broadened), 4 are unreviewed with 0 documents (exactly the not-yet-split case — no parsed doc can exist), 6 are reviewed with other docs but no authorization category.
- So under the locked "Prior Authorization"-only rule, ~12% of list rows fall back to the whole fax; the genuinely unavoidable fallback (no parsed doc exists at all) is ~10 rows (3.4%).

## 5. Options and effort

### Option A — Targeted preview-source fix via the `documents` collection (recommended, locked). Est. 2-3 SP

Point the preview at the parsed prior-auth document by joining the `documents` collection on `intakeId`, instead of reading `intake.documents[0]`. Leave all edit/read logic on the Intake.

Resolution rule (the algorithm):

1. Join: for the intake, find `documents` rows where `intakeId === String(intake._id)` and `category === "Prior Authorization"`.
2. One hit (the norm, 316/317 intakes): use its `blobPath` as the preview URL.
3. Multiple hits (the one known intake with two): tiebreaker — prefer the row whose split page range matches the prior-auth `documentDefinitions` range, else the most recent `createdAt`, else the first.
4. No hit (older records with no split row): fall back to today's behavior — the whole fax (`intake.documents[0].pdfDocument`) opened at the `documentDefinitions` prior-auth page. Nothing regresses for those.
5. `documentPage`: when the preview points at a matched split `blobPath`, set `documentPage` to `1`/undefined (the split PDF contains only the parsed pages, so the whole-fax page offset would jump to a wrong or non-existent page). Keep the existing page number only in the no-hit fallback.

Scope of A:
- Change `transformIntakeToPriorAuthLetter` in `apps/web/services/priorAuth/transforms.ts` to accept the resolved `documents` rows (or a pre-resolved `blobPath`) and set `documentUrl` / `documentPage` from them, with the fallback above.
- Detail path (`getPriorAuthDetail` in `apps/web/services/priorAuth/detail.ts`): it already calls `getDocumentsByIntakeId(intakeId)` for the description — reuse that result, filter to `category === "Prior Authorization"`, and pass it into the transform. No new query.
- List paths (`apps/web/app/api/patient/[patientId]/prior-auth/route.ts` and `apps/web/app/api/orders/new/[orderId]/auth-review/route.ts`): add one batched `documents` query per page — `documents.find({ intakeId: { $in: [...pageIntakeIds] }, category: "Prior Authorization" })` — and index the rows by `intakeId` in memory before mapping through the transform. One extra query per page (25-100 intakes), not one per row.
- No UI changes (the `PriorAuthLetter.documentUrl` / `documentPage` contract is unchanged).
- Update tests: `transforms` unit tests, the patient and order route tests, and any assertion on `documentUrl` / `documentPage`.
- Fixes both screens simultaneously because the transform is shared.

### Option B — Broader "show from documents" re-source. Est. 3-5 SP

Superset of Option A: also source the description and received date from the `documents` row (not just the preview file), and formalize the selection/fallback rules as a reusable resolver used by every prior-auth read path. Larger surface and more test coverage than A, without changing any edit behavior. Only worth it if the product wants the whole displayed row (not just the preview) to come from `documents`.

### Option C — "Show AND edit from documents" (full migration). Est. 13+ SP, high risk. Not recommended for this bug

Move the prior-auth read and write model from `intakes` to `documents`. This is a large architectural change:

- Add to `Document` schema: review status, verified letter type, all 19 verify-data fields (`VerifyDataFormPersisted`), the OCR letter schema (`ocrData` today is lab-only on documents), `documentDefinitions`/page ranges, reprocessing summary/error/status, and a workflow status equivalent to the Intake `status: 'processed'` gate.
- Rewrite the aggregation in `apps/web/services/priorAuth/query.ts` (`buildPriorAuthBaseMatchQuery`, `buildOrderScopedPriorAuthMatch`, `buildPriorAuthListPipeline`) — every `$match` path currently reads `patient.patientId`, `status`, `additionalInfo.ocrData.*`, `additionalInfo.priorAuthData.*` off the Intake. On documents these fields do not exist, so either duplicate/sync them onto documents or add a `$lookup` back to intakes.
- Rewrite `getPriorAuthDetail` / `applyPriorAuthReview` in `apps/web/services/priorAuth/detail.ts` to load and mutate a `Document` instead of an Intake, plus a new dual-write story (today it calls `mirrorIntakeToPostgres`).
- Re-key the reprocess flow: `POST /api/patient/[patientId]/prior-auth/[intakeId]/remove`, `apps/web/services/intakes/priorAuthReprocessDecision.ts`, and the `reprocessIntakeJob` all key off `intakeId` and `intake.documents[]` / `documentDefinitions`.
- Data backfill/migration for existing prior-auth intakes.
- Touches both Patient and Order surfaces plus every shared service and its tests.

This is disproportionate to a preview-source bug and introduces real regression risk to the reviewed-data workflow.

## 6. Recommendation

Do **Option A**: join the `documents` collection on `intakeId` and match `category === "Prior Authorization"`, using its `blobPath` as the preview URL (with the tiebreaker and no-hit fallback above). It fixes the reported bug on both screens with a change concentrated in one shared transform plus one batched query per list page, needs no `Document` schema change, and does not touch the edit/review workflow. Reserve Option C for a separately scoped and prioritized initiative if the product genuinely wants documents to become the system of record for prior auth.

## 7. Open questions / remaining checks before implementation

The identification rule is now locked (join by `intakeId`, `category === "Prior Authorization"`; data confirms it is effectively one-to-one). Remaining items:

1. Tiebreaker for the one intake with two matching rows: confirm the preferred rule (page-range match against `documentDefinitions`, else most recent `createdAt`). Low stakes given it is a single record.
2. No-hit coverage: quantify how many prior-auth intakes on the list have no `documents` row with `category === "Prior Authorization"` (older/un-split records) so the fallback path is exercised in tests, not just theory.
3. `documentPage`: confirm `EmbeddedFileViewer` behaves when `pageNumber` is `1` vs undefined on the single-page split PDF, and that the whole-fax page number is only kept in the no-hit fallback.
4. Removal/reprocess interaction: after a "not a Prior Auth letter" removal the `documents` row `category` is relabeled away from "Prior Authorization"; confirm the join then yields no hit and the row falls back or is hidden as expected (no stale pointer).
5. Confirm no other consumer relies on `PriorAuthLetter.documentUrl` pointing at the whole fax.

## 8. Rough estimate summary

| Option | Scope | Effort | Risk |
| --- | --- | --- | --- |
| A (documents-collection join, locked) | shared transform + `documentPage` handling + one batched documents lookup per list page + fallback | 2-3 SP | Low |
| B (broader show-from-documents) | A plus description/received-date re-source and a reusable resolver | 3-5 SP | Low-Medium |
| C (show + edit migration) | Document schema, pipeline, detail read/write, reprocess re-key, backfill | 13+ SP | High |

Recommended path: **Option A**, implemented in the shared `transforms.ts` (+ batched lookup in the two list routes) under TDD, verifying both the Patient card and Order Details Auth Review previews.
