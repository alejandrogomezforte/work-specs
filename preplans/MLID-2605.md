# MLID-2605 — FF fails to show Financial Calculator and Prior Auth (Pre-Plan)

> This is a **pre-plan**, not the implementation plan. It is the briefing that the real plan (built later via `/plan-task MLID-2605`) and any resulting work will be based on. We are crafting this file collaboratively with the user; the Jira description is a starting point, not the sole source of truth.

## 1. Jira context

- **Jira**: [MLID-2605](https://localinfusion.atlassian.net/browse/MLID-2605)
- **Type**: Bug
- **Priority**: Highest
- **Status**: In Development
- **Summary**: FF fails to show Financial Calculator and Prior Auth.
- **Area**: Order Details inside the orders-tracker (`/orders-tracker/[category]/[orderId]`).

### Steps to reproduce (from the ticket)

1. Disable the `CLINICAL_REVIEWS` feature flag.
2. Enable `ORDER_FINANCIAL_CALCULATION` and `ORDER_DETAILS_AUTH_REVIEW`.
3. Open Order Details for any order.

### Actual result

The Financial Calculator (Financial Review) and Prior Auth tabs are not shown.

### Expected result

The feature flags shall be independent. Each feature (Prior Auth, Financial Review) shall be shown when its own flag is enabled, regardless of whether `CLINICAL_REVIEWS` is on or off.

## 2. The problem

The three review tabs — **Clinical Review**, **Prior Auth**, and **Financial Review** — all live inside a single component, `OrderClinicalReviewsPanel`. That component is the only place those tabs are rendered. In the order-detail layout, the entire panel is mounted only when `CLINICAL_REVIEWS` is enabled. As a result, the Prior Auth and Financial Review tabs are coupled to `CLINICAL_REVIEWS` even though they each have their own independent feature flag.

When `CLINICAL_REVIEWS` is disabled, the panel never mounts, so the Prior Auth and Financial Review tabs disappear with it — even when `ORDER_DETAILS_AUTH_REVIEW` and `ORDER_FINANCIAL_CALCULATION` are enabled.

### Where the coupling lives

`apps/web/app/orders-tracker/[category]/[orderId]/layout.tsx` (the dashboard branch that mounts the panel):

```tsx
{isDashboardPath &&
isOrderDetailsActionsEnabled &&
isClinicalReviewsEnabled &&   // <- couples the whole panel to CLINICAL_REVIEWS
!isLoading &&
!error ? (
  <OrderClinicalReviewsPanel />
) : null}
```

Inside the panel (`apps/web/app/orders-tracker/[category]/[orderId]/components/OrderClinicalReviewsPanel.tsx`), the per-tab gating is already correct — each tab is conditional on its own flag:

```tsx
const reviewTabItems = useMemo(
  () => [
    { label: 'Clinical Review', value: 'clinical' },
    ...(isAuthReviewEnabled ? [{ label: 'Prior Auth', value: 'auth' }] : []),
    ...(isFinancialReviewEnabled
      ? [{ label: 'Financial Review', value: 'financial' }]
      : []),
  ],
  [isAuthReviewEnabled, isFinancialReviewEnabled]
);
```

So the bug is in the outer mount condition in `layout.tsx`, not in the panel's per-tab logic.

### Secondary observation — inconsistent routed-content gates

In `layout.tsx`, the routed children for each sub-route are gated differently:

- Clinical reviews content: requires `ORDER_DETAILS_ACTIONS && CLINICAL_REVIEWS`.
- Financial review content: requires `ORDER_DETAILS_ACTIONS && ORDER_FINANCIAL_CALCULATION`.
- Auth review content: requires only `ORDER_DETAILS_AUTH_REVIEW`.

Even if a user reached `/financial-review` or `/auth-review` directly by URL, the dashboard would still have no tab to navigate there because the panel hosting the tabs is hidden behind `CLINICAL_REVIEWS`. The planning step should confirm whether `ORDER_DETAILS_ACTIONS` should also gate the panel mount, or whether each feature should be fully self-gated.

## 3. Current behaviour

- The Order Details dashboard mounts `OrderClinicalReviewsPanel` only when `CLINICAL_REVIEWS` is enabled.
- With `CLINICAL_REVIEWS` disabled but `ORDER_FINANCIAL_CALCULATION` and `ORDER_DETAILS_AUTH_REVIEW` enabled, the panel does not render, so neither the Prior Auth tab nor the Financial Review tab is visible.
- The feature flags are therefore effectively dependent on `CLINICAL_REVIEWS`, contrary to the intended design.

## 4. Expected behaviour

- The three feature flags are independent.
- The review panel (and its tab strip) is shown when **any** of the relevant features is enabled.
- The Clinical Review tab appears only when `CLINICAL_REVIEWS` is enabled.
- The Prior Auth tab appears only when `ORDER_DETAILS_AUTH_REVIEW` is enabled.
- The Financial Review tab appears only when `ORDER_FINANCIAL_CALCULATION` is enabled.
- Disabling `CLINICAL_REVIEWS` hides only the Clinical Review tab; Prior Auth and Financial Review remain visible when their own flags are on.
- The default/active tab selection must behave sensibly when `CLINICAL_REVIEWS` is off (the panel currently defaults the active tab to `'clinical'` — to confirm during planning).

## 5. How the panel actually renders (architecture)

`OrderClinicalReviewsPanel` is the single component that renders the sub-tab strip **and** the active tab's content. It is mounted on all four order-detail paths:

| Path | Who renders the panel | Current gate (before fix) |
|---|---|---|
| Dashboard (`basePath`) | `layout.tsx` mounts it directly | `ORDER_DETAILS_ACTIONS && CLINICAL_REVIEWS` |
| `/clinical-reviews` | route `page.tsx` returns the panel, shown via `children` | `ORDER_DETAILS_ACTIONS && CLINICAL_REVIEWS` |
| `/financial-review` | route `page.tsx` returns the panel, shown via `children` | `ORDER_DETAILS_ACTIONS && ORDER_FINANCIAL_CALCULATION` |
| `/auth-review` | route `page.tsx` returns the panel, shown via `children` | `ORDER_DETAILS_AUTH_REVIEW` (no master gate today) |

The panel reads the URL (`activeReviewTab`) to decide which tab is active, and builds the tab strip from the three flags. Because the **dashboard** is the landing path for any order, and its mount requires `CLINICAL_REVIEWS`, disabling `CLINICAL_REVIEWS` hides the entire panel — taking Prior Auth and Financial Review with it. That is the reported bug.

## 6. Decisions (resolved with the user)

- **`ORDER_DETAILS_ACTIONS` stays as the panel-level master switch.** The whole review panel is on/off via this flag. Inside the panel, each tab is gated by **only** its own flag. (`ORDER_DETAILS_ACTIONS` is *not* one of the three tab flags — it is the container switch, so this satisfies "each tab tied only to its own FF".)
- **Tab → flag mapping (1:1, independent):**
  - Clinical Review → `CLINICAL_REVIEWS`
  - Prior Auth → `ORDER_DETAILS_AUTH_REVIEW`
  - Financial Review → `ORDER_FINANCIAL_CALCULATION`
- **Auth review is pulled under the master switch** for consistency: the `/auth-review` routed content currently does **not** require `ORDER_DETAILS_ACTIONS`; the fix adds it so all three tabs live under the same container switch. (Behavior change for auth — intentional.)
- **Default active tab:** when `CLINICAL_REVIEWS` is off and no sub-route matches (dashboard), the active tab falls back to the first **enabled** tab instead of always `'clinical'`.

## 7. Solution — the changes (acceptance criteria)

Two files. Each tab keyed to exactly its own flag, all under the `ORDER_DETAILS_ACTIONS` master switch.

### 7.1 `layout.tsx`

- **Dashboard mount** — keep the master switch; mount when any tab flag is on:
  ```tsx
  isDashboardPath &&
  isOrderDetailsActionsEnabled &&
  (isClinicalReviewsEnabled || isAuthReviewEnabled || isFinancialReviewEnabled)
  ```
- **Routed-content gates** — each is `master && own-flag`:
  - clinical: `isClinicalReviewsPath && isOrderDetailsActionsEnabled && isClinicalReviewsEnabled` (unchanged)
  - financial: `isFinancialReviewPath && isOrderDetailsActionsEnabled && isFinancialReviewEnabled` (unchanged)
  - auth: `isAuthReviewPath && isOrderDetailsActionsEnabled && isAuthReviewEnabled` (**add** master switch)

### 7.2 `OrderClinicalReviewsPanel.tsx`

- Add `isClinicalReviewsEnabled = useFeatureFlag(FeatureFlag.CLINICAL_REVIEWS)`.
- Make the **Clinical Review** tab conditional on `isClinicalReviewsEnabled` (Prior Auth and Financial Review are already conditional on their own flags):
  ```tsx
  ...(isClinicalReviewsEnabled ? [{ label: 'Clinical Review', value: 'clinical' }] : []),
  ...(isAuthReviewEnabled ? [{ label: 'Prior Auth', value: 'auth' }] : []),
  ...(isFinancialReviewEnabled ? [{ label: 'Financial Review', value: 'financial' }] : []),
  ```
- Fix `activeReviewTab` to fall back to the first enabled tab when clinical is off, so the panel never tries to render clinical content (`ReviewHistoryList`) with `CLINICAL_REVIEWS` disabled.
- Optional cleanup: gate the `useClinicalReviews` fetch on `isClinicalReviewsEnabled` so the panel does not fetch clinical data when that flag is off.

The panel does not need to know about `ORDER_DETAILS_ACTIONS` — it never mounts unless the master gate already passed in `layout.tsx`.

## 8. Financial calculator dependency audit (ticket's core claim) — CRITERION

The ticket claims the flag coupling specifically breaks the Financial Calculator. **Criterion: confirm there is no `CLINICAL_REVIEWS` dependency anywhere in the financial calculator (UI + backend), and remove any that is found.**

Investigation already performed (2026-06-17) — findings to be re-verified during implementation:

- **Financial review UI** (`financial-review/_components/`: `FinancialReviewIndex`, `Calculator`, `CalculationCard`, `CalculationBreakdown`) — no `CLINICAL_REVIEWS` reference, no `useClinicalReviews` import.
- **Patient-side calculator module** (`apps/web/components/Modules/financial-calculator/`) — no `CLINICAL_REVIEWS` reference.
- **Backend** — `apps/web/app/api/orders/new/[orderId]/financial-calculations/route.ts`, the `maintenance` equivalent, and `pages/api/patients/[id]/financialCalculation.ts` have **no feature-flag gating at all** (no `CLINICAL_REVIEWS`, no `ORDER_DETAILS_ACTIONS`, no `ORDER_FINANCIAL_CALCULATION`).
- The only clinical link in `financial-review/page.tsx` is structural: it returns `<OrderClinicalReviewsPanel />` (the shared panel).

**Conclusion:** the entire `CLINICAL_REVIEWS`→financial-calculator coupling is the single structural fact that the Financial Review tab is hosted inside `OrderClinicalReviewsPanel`, which the dashboard mounts behind `CLINICAL_REVIEWS`. Section 7 removes that coupling. No backend change is required. Implementation must still run this audit (grep `CLINICAL_REVIEWS` across the financial-review UI, the `financial-calculator` module, and the financial-calculations API routes) and remove anything found.
