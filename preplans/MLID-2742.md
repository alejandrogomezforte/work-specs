- **Summary**: Build an in-house masked-input component in `@repo/ui` to replace the `@mona-health/react-input-mask` fork.

### Why (PE request)

During the Next.js 16 / React 19 migration, the original `react-input-mask` was replaced with the `@mona-health/react-input-mask` fork (the original used `ReactDOM.findDOMNode`, removed in React 19). PE (Anatoliy) does not want production depending on a third-party fork, and asked us to own a purpose-built component in `@repo/ui`. This is **Option B (build our own)**, not vendoring the fork's source (rejected Option A).

## 2. Current state (grounding from the codebase)

The fork touches **4 code files**, not the "9" quoted in the ticket (the 9 are downstream consumers of the two wrappers):

| File | How it uses the fork |
|------|----------------------|
| `apps/web/components/UI/MaskedInput/MaskedInput.tsx` | Imports the `InputMask` runtime component. Passes `StyledInput` as a **single child element**; the fork clones it with masked props. |
| `apps/web/components/UI/MUI/MuiPhoneField.tsx` | Its `PhoneMaskAdapter` renders `<InputMask>` with **no child**, so the fork renders its **own bare `<input>`**. Used as MUI TextField's `inputComponent`. |
| `apps/web/components/UI/StyledInput/StyledInput.tsx` | Imports only the `InputState` **type**. |
| `apps/web/types/mona-health-react-input-mask.d.ts` | Type shim restoring `Props` / `InputState` (the fork ships `declare const InputMask: any`). |

**Out of scope:** `utils/constants/maskedInput.ts` (numeral/currency masks) and `MaskedInputExtended` use a different, cleave-style masking library ‚Äî they do not import `@mona-health/react-input-mask`. We leave them alone.

### The masks actually used (only 3, digit-only)

- `+1 (999) 999-9999` ‚Äî phone (hardcoded in `MuiPhoneField`)
- `99/99/9999` ‚Äî date (passed as `mask` prop to `MaskedInput`)
- `999999` ‚Äî 6-digit code (passed as `mask` prop to `MaskedInput`)

Only the `9` digit placeholder is used. No alpha, alphanumeric, regex, or optional-section masks. Advanced fork features (`beforeMaskedStateChange`, custom `maskChar`, `maskPlaceholder`) are **not** used and are out of scope.

## 3. Key design constraint

The two wrappers consume the fork in **two different shapes**:

1. `MaskedInput` gives it a **child element** to clone (a styled input).
2. `MuiPhoneField` gives it **no child** and expects it to render its **own bare `<input>`** (MUI supplies the field chrome).

Therefore the new `@repo/ui` component is a **headless, unstyled masked `<input>`**: it owns caret logic and mask formatting, accepts `className` / `style` / `ref`, and renders a plain `<input>`. It must **not** wrap MUI TextField ‚Äî that would break the `MuiPhoneField` case, which needs a bare input inside MUI. One primitive satisfies both wrappers.

### The hard part ‚Äî caret management

Keeping the cursor correct while typing, deleting across fixed characters (`(`, `)`, `-`, `/`), pasting a full value, range-selecting, autofill, and mobile keyboards. Needs manual `setSelectionRange` and selection tracking. These edge cases mostly appear **only in a real browser**, not in jsdom unit tests. Mitigation: reference the caret algorithm from the MIT-licensed original / fork (confirm license first), keep the existing wrapper tests + 9 consumers as a regression net, and do mandatory real-browser verification.

## 4. Decisions taken

- **Component API**: clean own-API headless masked input (`value`, `onChange`, `mask`, `alwaysShowMask`, `className`, `ref`), and adapt both wrappers to it ‚Äî not reproducing the fork's clone-a-child quirk. Only 2 consumers, so churn is small and the API stays honest.
- **Name**: `MaskInput` in `@repo/ui` (avoids clashing with the existing app-level `MaskedInput` wrapper, which stays as the app wrapper).
- **Styling**: Emotion / `sx` only ‚Äî no CSS modules (a `@repo/ui` rule).

## 5. The three steps ‚Üí three subtasks

### Step 1 ‚Äî Build the component (subtask ST1)

- New `packages/ui/src/components/MaskInput/` following the folder convention: `index.ts`, `MaskInput.tsx`, `MaskInput.styles.ts`, `MaskInput.test.tsx`, `MaskInput.stories.tsx`, plus a `useMaskedCaret` hook.
- `'use client'`, digit-only (`9`) masks, hardcoded to the 3 patterns' behavior.
- Export from the `@repo/ui` barrel `index.ts`.
- **Gating action item:** confirm the caret algorithm's license (original `react-input-mask` is MIT; verify `@mona-health` fork is also MIT) before reusing it.
- Covered by unit tests (type, delete, delete-across-fixed-char, paste, range-select) and a Storybook story. jsdom's `setSelectionRange` support is limited, so the Storybook story is the venue for real-browser caret verification of the component in isolation.

### Step 2 ‚Äî Swap the wrappers (subtask ST2)

- Point `MaskedInput` and `MuiPhoneField` internals at the new `MaskInput`; update `StyledInput`'s type import.
- Keep all existing wrapper tests and the 9 downstream consumers green ‚Äî no consumer changes.
- Build + verify the component in isolation (Step 1) **before** this swap, so rollback stays easy.

### Step 3 ‚Äî Sunset the fork + acceptance (subtask ST3)

- Remove `@mona-health/react-input-mask` from `apps/web/package.json` + lockfile (an `npm uninstall` ‚Äî run with the user's OK).
- Delete the `types/mona-health-react-input-mask.d.ts` shim.
- Real-browser verification of all 3 masks across the real forms: typing, deleting across fixed characters, paste, range selection, autofill, mobile keyboard. No React 19 console errors (no `findDOMNode`, no caret jump-to-end). All existing tests green.

## 6. Acceptance criteria (from the ticket)

- New masked-input component lives in `@repo/ui` with a Storybook story and unit tests.
- `MaskedInput` and `MuiPhoneField` use the in-house component; all 9 consumers work unchanged.
- `@mona-health/react-input-mask` is removed from dependencies.
- The 3 masks behave correctly in a real browser (type, delete-across-fixed, paste, range-select, autofill, mobile).
- No React 19 console errors. All existing tests remain green.
