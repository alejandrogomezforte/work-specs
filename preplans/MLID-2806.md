## 1. Summary

Infusion Guides record authorizations in WeInfuse, but LISA cannot see them. Today the only authorization information inside LISA is the Auth Letter Review section, and the "Authorization Submission Date" on a New Order is typed in by hand — an extra manual step that can be wrong or forgotten.

The epic as written by the Product Owner describes the work as three user interface deliverables. That framing assumes the data is already inside LISA, which it is not. The transfer of the authorizations and their notes from WeInfuse into LISA is real engineering work with its own storage, its own entry points and its own failure modes. It is therefore separated into a deliverable of its own — **D0** — and the three original deliverables are reduced to display and behaviour only.

Two facts change the shape of the epic relative to how it was written:

1. **The transfer mechanism already exists in the product.** Looker reports run against WeInfuse and call LISA webhooks, which insert the rows into LISA tables. D0 replicates that pattern for two new feeds: authorizations, and authorization notes.
2. **The order linking already exists.** On the patient record's Orders tab, a user can already link a LISA order to a WeInfuse order series, and the chosen series identifier is stored on the LISA order itself. The epic's phrase "this will come from the Order Linking" points at this existing capability. Nothing in D1, D2 or D3 has to build it.

Together these mean the authorization-to-order relationship is a **read-time join** on a value that is already stored. No new write path, no synchronisation job, and no code that reacts to the three "triggers" the ticket lists — a link or an unlink is simply reflected the next time the data is read.

A further piece — showing authorization changes in Order History — is named in the ticket only as context and is tracked in a different epic.

_To complement:_ the ticket does not define what happens when an authorization arrives for a patient LISA does not know, nor which user roles may see the new tab, nor whether the patient record header change is part of this epic's acceptance or a separate design cleanup.

**Reference implementation for D0-T2**

The `looker_orders` webhook is the closest existing example and should be read before designing the new ones. It is the feed that brings WeInfuse orders into LISA, and it is documented in `analysis/weinfuse-orders-overview.md`. Its shape:

- An endpoint per feed under the webhooks API folder, authenticated by a shared token supplied in a request header and compared against an environment secret.
- A thin handler that does nothing but delegate, plus optional payload archiving behind a feature flag for troubleshooting.
- A separate transform-and-upsert utility holding all the real logic. This is where a replicated webhook will spend its effort.
- A persistence entry point that hands the prepared rows to the repository in the data access package.

Five behaviours in that utility are worth carrying over, because each exists to fix a real incident or data quirk:

1. **Delta, not snapshot.** An empty payload is a no-op. Treating absence as "no longer active" once mass-deactivated thousands of order rows from a single empty delivery. Any active/inactive state should be derived from a field in the payload, never from presence in it.
2. **Merge, do not replace.** A report delivery is a partial payload — fields that arrive empty are deliberately skipped so stored values survive. A blind overwrite nulls columns on every run.
3. **Deduplicate before writing.** Report joins fan out: one logical record can arrive as several rows differing only in the joined columns. A single upsert statement cannot touch the same row twice, so duplicates must be collapsed and logged.
4. **Resolve foreign keys at write time.** Incoming rows carry WeInfuse identifiers, not LISA ones. The repository resolves them to the parent table's identifier and stores null when the parent is unknown, keeping the raw value so resolution can be re-run later.
5. **Fail loudly.** A persistence failure is re-thrown so the endpoint answers non-2xx. A swallowed failure reads as a healthy sync while nothing is being written.

The authorizations feed is the closer analogue of the two, since it carries the order series identifier that everything downstream joins on. The notes feed is a child feed and mainly needs its parent resolved, which is behaviour 4.

## 3. Data overview

Two new tables receive WeInfuse data. Everything else is already in place and is read.

**PostgreSQL only.** Decided with the Principal Engineer: no new MongoDB structures are created for this epic. The PostgreSQL migration is close to finished and MongoDB is expected to be dropped around the time this epic is delivered, so anything new born in MongoDB would be migration debt on the day it ships.

| Collection / table | Deliverable | Action | Why |
|---|---|---|---|
| WeInfuse authorizations (new, PostgreSQL) | D0-T1 | create | Receives the authorizations feed. Carries the order series identifier used for the join. |
| WeInfuse authorization notes (new, PostgreSQL) | D0-T1 | create | Receives the notes feed. Each note belongs to one authorization. |
| New Orders | D1, D2, D3 | read / edit | Already stores the linked WeInfuse order series identifier. Read for the join; written only if D3 stores the derived date rather than computing it on read. |
| Patients | D1 | read | Filter authorizations by patient, and supply the header fields. |

**Row identity.** Each new table needs a natural key the feed can upsert on, decided in D0-T1. Two warnings from the existing orders feed. First, the older tables carry a transitional identifier column inherited from the MongoDB migration; the data access package's own guidance says never to add one to a new table, so these two tables must not have it. Second, the WeInfuse identifier on the existing orders table is deliberately *not* unique, because the same WeInfuse record can arrive as more than one row — so a WeInfuse identifier cannot be assumed unique without checking the actual feed first.

_To complement:_ confirm how the New Orders and Patients data is read at the time this epic is built, since the join crosses from the new PostgreSQL tables to records that may still be served from MongoDB while the migration finishes. Decide whether a repeated feed run replaces existing rows or appends — the existing orders feed upserts and merges rather than replacing, and that is the safer default. Decide whether the D3 date is stored on the order or computed at read time.

## 5. Deploy overview

Not a plain deploy. Nearly all of the weight is in D0.

D0 needs the two new tables created through the project's migration system, the two new entry points deployed and reachable, the two Looker reports configured on the WeInfuse side to call them on the required cadence, and the one-time full historical load. Part of that work is outside this repository.

Following the existing Looker webhooks, each new endpoint authenticates against a shared token held in an environment variable. Those are new environment variables and must be wired into Azure before the reports are pointed at the endpoints — otherwise every delivery is rejected. Whether the two feeds share one token or take one each is a decision for the plan.

D3 needs a decision, applied at release, about orders that already carry a hand-entered submission date.

D1 and D2 are ordinary user interface releases, expected to be gated by a feature flag.

_To complement:_ confirm who configures the Looker reports and on what cadence. Decide the feature flag structure — one flag for both surfaces, or one each, and whether D3 needs its own so the derived date can be switched off without hiding the tabs. Confirm which secrets or configuration the new entry points need and who wires them into Azure. State the required deploy order between the migration, the entry points, the historical load and the user interface release.

## 6. Open questions

- Will the New Orders and Patients data be served from PostgreSQL by the time D1 and D2 are built, or must the join cross two databases in the meantime?
- Is the derived Authorization Submission Date computed at read time or stored on the order? Read time is simpler and self-correcting; stored is queryable and filterable.
- What happens to submission dates users have already entered by hand that disagree with the derived value? — options: overwrite / keep and mark / report only
- What happens when an incoming authorization references a patient or an order series LISA does not have? — options: store unlinked and let a later link resolve it / reject and report / quarantine. Precedent: the existing orders feed stores the row with a null link and keeps the raw WeInfuse value, so resolution can be re-run later.
- Does the required refresh cadence describe how often the Looker reports run, or how often the user interface re-reads what is stored?
- Which roles may see the WeInfuse Auths tab, and is the rule the same on the patient record and on the order?
- Is the patient record header change in scope for this epic's acceptance, or does it belong to a separate design ticket?
- Is the one-time historical load a script run once by an engineer, or a re-runnable job exposed in the admin jobs interface?
- Does anything else in LISA read the Authorization Submission Date today — reporting, filters, exports — that changes behaviour once the field becomes system driven?
- Does the Product Owner need the epic restructured in Jira to show D0, or does D0 stay an engineering-side breakdown under the existing three deliverables?
