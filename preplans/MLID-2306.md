# MLID-2306 — Order Details: Auth Review (Preplan)

Story under epic **MLID-2206** (Order Details v2 — Show Each Step Under Order Tracker).
Goal of the story: add an **Auth Review** tab inside the order detail view that shows a
**filtered** list of the patient's prior authorization documents, scoped to the relevant
documents for that specific order.

---

## 1. Current context

This section documents what already exists today for prior authorizations at the
**patient level**, because the order-level Auth Review tab will reuse the same data,
collection, and presentation, with an extra order-scoped filter on top.

### 1.1 Route and entry point

- Route: `/patient/[patientId]/prior-auth`
- Page file: `apps/web/app/patient/[patientId]/prior-auth/page.tsx`

The page is intentionally thin. It only composes three things:

```tsx
<FeatureFlagWrapper flag={FeatureFlag.PRIOR_AUTH}>
  <PatientTabsModule>
    <PatientPriorAuth />
  </PatientTabsModule>
</FeatureFlagWrapper>
```

- `FeatureFlagWrapper` gates the whole feature behind the `PRIOR_AUTH` feature flag.
- `PatientTabsModule` renders the patient-level tab navigation (Details, Calls History,
  Financial Calculator, Clinical Reviews, **Prior Auth**, Documents, Insurance,
  Appointments, Orders).
- `PatientPriorAuth` is the actual feature component (table + filters + preview).

There is also a detail route for a single record:

- `/patient/[patientId]/prior-auth/[intakeId]` → `page.tsx` in the same folder, which
  renders the single prior-auth review view.

### 1.2 React components involved

Module folder: `apps/web/components/Modules/patient/components/patientPriorAuth/`
- `PatientPriorAuth.tsx` — the list/table component (this is the one to mirror for the order tab)
- `PriorAuthReview.tsx` — the single-record review/detail view
- `styles.module.css`, `PriorAuthReview.module.css` — CSS Modules styling
- `__tests__/PatientPriorAuth.test.tsx`, `__tests__/PriorAuthReview.test.tsx`

`PatientPriorAuth` is a client component. It:
- reads the active patient from `usePatientProvider()` context,
- keeps all filter/pagination state in URL search params (so filters survive refresh and
  are shareable),
- fetches data from the API routes described below,
- renders the table, the filter controls, pagination, and the preview dialog.

### 1.3 API routes and data source

The component talks to three endpoints, all under
`apps/web/app/api/patient/[patientId]/prior-auth/`:

| Endpoint | File | Purpose |
|----------|------|---------|
| `GET /api/patient/[patientId]/prior-auth` | `route.ts` | Paginated, filtered, sorted list of prior-auth letters |
| `GET /api/patient/[patientId]/prior-auth/filter-options` | `filter-options/route.ts` | Distinct payor + drug values for the dropdowns (kept constant regardless of applied filters) |
| `GET /api/patient/[patientId]/prior-auth/[intakeId]` | `[intakeId]/route.ts` | Single record (used to resolve the preview's document description) |
| `DELETE /api/patient/[patientId]/prior-auth/[intakeId]/remove` | `[intakeId]/remove/route.ts` | Remove a prior-auth record |

### 1.4 Collection / data model being displayed

There is **no dedicated "prior auth" collection**. The records come from the **`Intake`**
model (`apps/web/models/Intake.ts`) — the document-intake collection. Prior auth is a
document *type* that goes through a specific OCR pipeline, and the parsed result is stored
on the intake document.

Link from patient to intakes: the intake's `patient.patientId` holds the patient's
**WeInfuse IV id**, so the query matches against `patient.we_infuse_id_IV`.

The base match query lives in `apps/web/services/priorAuth/query.ts`
(`buildPriorAuthBaseMatchQuery`) and selects only the relevant intakes:

```js
{
  'patient.patientId': patientWeInfuseId,            // patient.we_infuse_id_IV
  status: 'processed',
  'additionalInfo.ocrData.letterMetadata.documentType.value': { $exists: true },
  'additionalInfo.priorAuthData.reprocessingStatus': { $ne: 'pending' },
}
```

Two important sub-objects on each intake:

- `additionalInfo.ocrData` — the **raw OCR** output (drug, payor, dates, document type, etc.).
- `additionalInfo.priorAuthData` — the **user-verified / reviewed** values that override OCR.

The transform `transformIntakeToPriorAuthLetter` in
`apps/web/services/priorAuth/transforms.ts` flattens an intake into the `PriorAuthLetter`
shape used by the UI. **Verified values (`priorAuthData`) take precedence over raw OCR
values** for drug, payor, and auth dates. Key field mappings:

| UI field (`PriorAuthLetter`) | Source (verified first, then OCR) |
|------------------------------|-----------------------------------|
| `drug` | `priorAuthData.drugName` → `ocrData.service.brandDrugName` → `ocrData.service.genericDrugName` |
| `payor` | `priorAuthData.payorName` → `ocrData.payor.name` |
| `receivedAt` | `intake.createdAt` |
| `letterMetadata.documentType` | verified letter type → `ocrData.letterMetadata.documentType` |
| `authValidFrom` | `priorAuthData.startDate` → `ocrData.authorization.effectiveDate` |
| `authValidTo` | `priorAuthData.endDate` → `ocrData.authorization.expirationDate` |
| `reviewStatus` | `priorAuthData.reviewStatus === 'reviewed' ? 'reviewed' : 'needs_review'` |
| `documentUrl` / `documentPage` | first document blob + prior-authorization page range |

Types are defined in `apps/web/types/priorAuth.ts`.

### 1.5 Columns displayed

The table (in `PatientPriorAuth.tsx`) renders these columns:

| Column | Content |
|--------|---------|

| **Drug** | `letter.drug` |

Row click navigates to the single-record detail route
(`/patient/[patientId]/prior-auth/[intakeId]`).

### 1.6 Filters displayed

Filter controls above the table (all reflected in the URL query string):

| Filter | Control | Backed by |
|--------|---------|-----------|
| **Received Date Range** | `DateRangeSelect` (start/end) | `createdAt` `$gte` / `$lte` |
| **Drug** | `Select` dropdown | matches `ocrData.service.brandDrugName` OR `ocrData.service.genericDrugName` |
| **Letter Type** | `Select` dropdown (`PRIOR_AUTH_LETTER_TYPES`) | matches letterType / `documentType` across verified + OCR paths |

default 25, max 100).

Drug and Payor dropdown options come from the separate `/filter-options` endpoint so the
available choices stay constant even after a filter is applied.

### 1.7 Detail view — 3-step review behavior (the part we must replicate)

This subsection documents exactly how the existing single-record detail view behaves today,
because the order Auth Review detail must be a faithful port of it. Source file:
`components/Modules/patient/components/patientPriorAuth/PriorAuthReview.tsx`.

**Route and params.** Detail route is `/patient/[patientId]/prior-auth/[intakeId]`. The
component reads `patientId` and `intakeId` from `useParams()`. (The "docId" in the row link
is the **intake `_id`**.)

**Step model — 3 steps rendered as top subtabs:**
1. **Verify Letter Type** — a `Select` for the letter type + an "AI Letter Summary" card.
2. **Verify Data** — the `VerifyDataForm` (fields differ by letter type).
3. **WeInfuse Summary Note** — the `WeinfuseSummaryNote` (generated, copyable note).

**Navigation is pure React state — the URL never changes.** `currentStep` is
`useState<Step>` (`Step = 1 | 2 | 3`). `handleNextStep` / `handlePrevStep` / `handleStepClick`
only call `setCurrentStep`. There is no router push for step changes, so the route stays on
`.../[intakeId]` across all three steps. (Confirms the reviewer's observation.)

**Bottom navigation buttons (labels are contextual, not "Next/Previous"):**
- Previous button label: `currentStep === 2 ? 'Back to Letter Type' : 'Back to Edit Fields'`.
- Next button label: `currentStep === 1 ? 'Verify Data' : 'Generate Note'`.
- The previous/next buttons render only when not on the boundaries (prev hidden on step 1,
  next hidden on step 3).

**Form-validation gating.** Only the **step 2 → 3** transition is gated: the next button is
`isDisabled={currentStep === 2 && !step2IsValid}`. `step2IsValid` comes from `VerifyDataForm`
via its `onChange(formData, isValid)` callback. Step 1 → 2 is **not** gated.

**Single-step letter types — IMPORTANT branch.** `SINGLE_STEP_LETTER_TYPES = ['Receipt',
'Reminder', 'Other']`. When the selected letter type is one of these, steps 2 and 3 are
**hidden** (only the "1. Verify Letter Type" subtab shows), the form payload is cleared, and
the finish button appears directly on step 1. Any non-single-step type shows all 3 steps.

**Initial step on open.** After fetching, if the document is already `reviewed`, is not a
single-step type, and has filled verify data, the view opens directly on **step 3** (the
summary). Otherwise it opens on **step 1**.

**Finish / save — two modes, both redirect back to the list:**
- When `reviewStatus === 'needs_review'`: button reads **"Finish & Mark Reviewed"** and calls
  `handleFinishReview`, which PATCHes with `reviewStatus: 'reviewed'`.
- When already `reviewed`: button reads **"Finish & Save Changes"** and calls
  `handleSaveChanges`, which PATCHes the verify data + letter type **without** changing
  `reviewStatus`.
- On success both show a notification and `router.push` back to `/patient/[id]/prior-auth`.

**What the PATCH does in the background** (`.../[intakeId]/route.ts`, `PATCH`): it merges into
`additionalInfo.priorAuthData` on the intake — `letterType` (as an `OCRField`), each verify-data
field (stored as `OCRField` objects with OCR page numbers backfilled), and `reviewStatus`. It
validates patient ownership (`intake.patient.patientId === patient.we_infuse_id_IV`) and saves
via Mongoose `intake.save()`. **This is where `priorAuthData.liDrugId` gets written** — the same
field the order-scoping drug match relies on (section 2.7 / 2.10).

**Document preview in the detail view is INLINE, not a modal.** The right panel renders
`InlineDocumentPreview` wrapping `EmbeddedFileViewer` (split layout: review steps left, fax
right). This differs from the **list** view, whose preview opens `EmbeddedFileViewer` in a
dialog. Section 2.5's "reuse the EmbeddedFileViewer modal" decision is about the **list**
preview; the **detail** view keeps the inline right-panel fax render.

**Header extras to port:** patient name, a **read-only** Review Status indicator (colored dot +
"Needs Review"/"Reviewed" — see discrepancy note below), a letter-type chip (shown from step 2
on), and a **"View Fax in Spruce ↗"** link when `spruceUrl` is present.

**Admin-only removal flow.** On step 1, admins (`hasAdminPermissions`) see a "This is not a
Prior Authorization letter" action that opens a removal panel (pick a corrected document type +
description) and POSTs to `.../[intakeId]/remove`. Non-admins do not see it.

**Discrepancy with section 2.6 to reconcile when porting:** section 2.6 describes a "Review
Status selector (Needs Review / Reviewed)". The **current implementation has no selector** — the
header status is display-only and the status flips to `reviewed` solely via the
"Finish & Mark Reviewed" button. Decide in section 3 whether the order view keeps the existing
display-only behavior or actually introduces a selector.

## 2. Requirements specs and UI design specs

### 2.1 Summary of the ask

Add an **Auth Review** tab inside the **order detail** view (Order Details v2). The tab
shows the **same prior-auth experience that already exists** at `/patient/[id]/prior-auth`
(the list table + the 3-step detail review), but with two changes:

1. The list is **filtered down to the prior-auth documents relevant to that one order**
   (instead of every prior-auth document for the patient).
2. The presentation is rebuilt using the **new MUI X DataGrid** (`@repo/ui` `DataGrid`)
   that recent Order Details v2 screens use, matching the designer's mock.

Everything else — the columns, the meaning of each filter, the 3-step detail review —
is a **port** of what we already have in `/patient/[id]/prior-auth`.

### 2.2 New route / tab placement

- Order detail dashboard today lives at `/orders-tracker/new/[id]/`.
- The Auth Review tab must be a **new route** under it:
  - **Index (list view):** `/orders-tracker/new/[id]/auth-review`
  - **Detail (single letter, 3-step review):** `/orders-tracker/new/[id]/auth-review/[letterId]`
- In the order detail view it appears as a **sub-tab** alongside the existing
  **Clinical Reviews** and **Financial Review** tabs (the row of tabs labelled
  "Clinical Reviews · Auth Review · Financial Review" in the mock).

This mirrors the playground mock routing:
- `reviewTab === 'auth'` and no sub-id → `<AuthReviewIndex>` (list). The header row shows the
  section tabs (Clinical Reviews · Auth Review · Financial Review).
- `reviewTab === 'auth'` with a sub-id → `<AuthReviewDetail>` (one letter). The header row is
  **taken over** by `← Back` + "Auth Review" title (left) and the Review Status selector +
  "View Fax in Spruce" link (right) — see section 2.6. The Back button navigates from the
  detail route back to the index route (detail mode → table mode).

Mock reference files (in the sister repo `~/Dev/li-ui-playground`):
- `src/pages/OrderDetail.tsx` — order detail shell + tab routing
- `src/pages/AuthReview/AuthReviewIndex.tsx` — the list/table view
- `src/pages/AuthReview/AuthReviewDetail.tsx` — the single-letter 3-step view

> li-call-processor. Use it only to read the **specific Auth Review sub-tab UI intent**,
> not as a structural template.

### 2.6 Detail view — 3-step review (port existing)

Clicking a row (or "Open" in the preview) enters the single-letter detail view. It is a
**3-step review wizard** that **already exists** in `/patient/[id]/prior-auth`
(`PriorAuthReview.tsx`):

1. **Verify Letter Type** — choose/confirm letter type; shows an AI letter summary; the fax
   document renders on the right. Primary action: "Verify Data".
2. **Verify Data** — verify extracted fields: Auth Number, Payor Name, Start/End Date,
   Drug Name, Procedure Code, Benefit Type, Number of Visits, Frequency, Dose, Treating
   Address, etc. Primary action: "Generate Note".
3. **WeInfuse Summary Note** — generated summary note (copyable) to paste into WeInfuse;
   "Back to Edit Fields" and "Finish & Mark Reviewed".

**Detail header row (updated 2026-06-03 — designer added a back affordance).** Above the
step subtabs, the detail view uses a **takeover** of the section-tabs row: the
"Clinical Reviews · Auth Review · Financial Review" tabs are **replaced** while a letter is
open, by a single header row laid out in two halves:

- **Left half:** a **`← Back`** button (text style, left-arrow icon) + a thin vertical divider
  + the bold **"Auth Review"** section title (h5). The Back button returns to the **list**
  view (the auth-review index route) — it is the switch from detail mode back to table mode.
- **Right half:** the **Review Status** selector (`Needs Review` / `Reviewed`) on the left and
  the **"View Fax in Spruce ↗"** link on the right.

When **no** letter is open (list/index view), this same row instead shows the normal section
tabs (Clinical Reviews · Auth Review · Financial Review). So entering a letter swaps tabs →
(Back + title); leaving it swaps back.

Mock reference: `~/Dev/li-ui-playground/src/pages/OrderDetail.tsx` (the `isAuthDetail`
takeover branch) and `src/pages/AuthReview/AuthDetailHeader.tsx` (Review Status selector +
Spruce link).

**Note on the Review Status selector vs. the current patient view.** Section 1.7 flagged that
the *existing patient* detail view renders review status as a **display-only** dot, with no
selector — status flips to `reviewed` only via the finish button. The **new order design
resolves that discrepancy in favor of a real selector**: the order Auth Review detail header
has an actual `Needs Review` / `Reviewed` dropdown (as in the screenshot and
`AuthDetailHeader.tsx`). So the order view adds a selector the patient view does not have.

The task is to **port** this 3-step review into the order route, not to redesign it — but with
the updated header (Back + "Auth Review" title takeover, and the real Review Status selector)
described above.

### 2.7 Order-scoped filter requirement (the new business logic)

This is the only genuinely new logic versus the patient view. A patient prior-auth document
is included in the order's Auth Review list only when **both** conditions hold:

1. **Date condition** — the prior-auth document's **decision date** (from OCR) is
   **on or after the LISA order creation date**.
   - If the decision date is **NULL**, fall back to the **OCR fax document received date**.

2. **Drug condition** — the **drug from the prior-auth document matches the order's drug**,
   **OR** the prior-auth drug is **unknown / null**.
   - The PO speaks in terms of drug *name*, but we should match by **drug id** where possible
     (the prior-auth verified data carries `liDrugId`; the order carries its own drug id) so
     the match is reliable rather than string-based.

### 2.8 Feature flag

The order Auth Review tab gets its **own feature flag: `ORDER_DETAILS_AUTH_REVIEW`**. It is
separate from the patient-level `PRIOR_AUTH` flag and from the Order Details v2 flag. The name
follows the existing order-detail sub-feature family in the `FeatureFlag` enum
(`ORDER_DETAILS_DOCUMENTS`, `ORDER_DETAILS_ACTIONS`). Add it to the `FeatureFlag` enum +
`DEFAULT_FEATURE_FLAGS` (default `false`), and gate both the tab/route and the API routes
behind it (UI + server, per RBAC/feature-gating convention).

### 2.9 Resolved decisions (recap)

- **Received Date Range filter:** dropped from the order tab UI (the order-scoping date
  criterion makes it redundant). Follow the mock.
- **Document preview:** reuse the existing `EmbeddedFileViewer` modal (the mock's side-drawer
  is ignored — the side-drawer was a prototype simplification; we use the existing PDF viewer instead).
- **Feature flag:** new `ORDER_DETAILS_AUTH_REVIEW` flag (named to match the existing
  `ORDER_DETAILS_*` family in the `FeatureFlag` enum).

### 2.10 To resolve in the next stage (TBD — source fields for the filter)

These are deferred to section 3 / the next investigation stage and are marked **TBD**:

- **RESOLVED — Decision date source field:** `additionalInfo.ocrData.authorization.dateOfDecision`.
  It is an OCR field object (`{ value: "YYYY-MM-DD", page: [...] }`); the OCR schema comment
  describes it as "Date the authorization decision was made (YYYY-MM-DD)". This is distinct
  from the "Received" column, which uses `intake.createdAt`.

  Stage-data coverage check (collection `intakes`, base match
  `status: 'processed'` + `additionalInfo.ocrData.letterMetadata.documentType.value` exists):
  - 187 processed prior-auth intakes total.
  - Only **74 (~40%)** have a non-empty `dateOfDecision.value`.
  - So **~60% have no decision date** — the fax-received-date fallback is the common path,
    not an edge case.

  Implementation notes for section 3:
  1. `dateOfDecision.value` is a **string** in `"YYYY-MM-DD"` format, not a `Date`. The order
     creation date it is compared against is a real `Date`, so the filter must convert with
     `$dateFromString` and guard malformed/empty values via `onError` / `onNull` — a raw
     string-to-Date comparison will not work.
  2. Sibling date fields on `authorization` by frequency in the sample: `effectiveDate` (78),
     `dateOfDecision` (75), `expirationDate` (50). `effectiveDate` is marginally more common,
     but per the spec wording `dateOfDecision` is the correct literal match for "decision date".

- **RESOLVED — Fax received date (fallback) source field:** `intake.createdAt`.
  When `dateOfDecision` is null (the common case — ~60% of documents), the date condition
  falls back to the intake's `createdAt`, i.e. the moment LISA ingested the Spruce fax and
  created the intake record. This is the same field the patient prior-auth table already
  displays as the "Received" column (`transforms.ts` → `receivedAt`).

  Why this field:
  - It is the only "received" timestamp that is **always present** on every intake, so the
    fallback branch can never itself be null.
  - It is stored as a real `Date` (not an OCR `"YYYY-MM-DD"` string), so — unlike
    `dateOfDecision` — it needs **no** `$dateFromString` conversion and compares directly
    against the order creation `Date`.
  - The OCR-extracted alternatives (`letterMetadata.letterDate`, `letterMetadata.documentDate`,
    `receipt.receiptDate`) were rejected: they are frequently null and/or only populated for
    specific letter types, so they cannot be relied on as the universal fallback.

  Net date-condition logic for section 3:
  `effectiveDecisionDate = dateOfDecision.value (parsed via $dateFromString) ?? intake.createdAt`,
  then include the document when `effectiveDecisionDate >= orderCreationDate`.
- **RESOLVED (provisional) — Order creation date source field:** `neworders.createdAt`.
  The date condition compares the document's effective date against the order's own
  Mongo `createdAt` timestamp (when the `neworders` row was inserted).

  Provisional: to be confirmed with the PO. If the PO wants a different business date
  (for example a WeInfuse/LISA-sourced order date that can differ from when our row was
  written), this is the single field to swap — the comparison logic stays the same.
- **RESOLVED — Order drug id field:** `neworders.drug` (a stringified `lidrugs._id`).

  Both sides of the drug-match condition resolve to the same identifier space — a
  `lidrugs` document `_id`:
  - **Order side:** `neworders.drug` holds the drug `_id` as a **string**. The orders-tracker
    list pipeline (`services/mongodb/ordersTracker.ts`) converts it with
    `$convert: { input: '$drug', to: 'objectId', onError: null }` and `$lookup`s against
    `COLLECTION.LIDrugs` → collection **`lidrugs`** to resolve the drug name shown in the table.
  - **Prior-auth side:** `additionalInfo.priorAuthData.liDrugId.value` also holds a
    `lidrugs._id` as a string (captured during manual verification; surfaced as
    `PriorAuthLetter.liDrugId`).

  Verified against stage data: sampled `priorAuthData.liDrugId` values
  (e.g. Actemra, Gammagard Liquid, Panzyga, Leqembi) all resolve to `lidrugs` documents
  whose `name` matches the stored `priorAuthData.drugName`.

  Match logic for section 3:
  - Both ids are stored as hex strings, so the match can be a direct **string equality** of
    `neworders.drug` and `priorAuthData.liDrugId.value` (no ObjectId conversion required for
    the comparison itself).
  - Include the document when `priorAuthLiDrugId === orderDrugId` **OR** the prior-auth
    `liDrugId` is unknown/null (un-reviewed documents have no `liDrugId`, so they match on the
    null branch by design).
  - Dependency: **MLID-2363** (drug-name mapping during Prior Auth OCR) is what will eventually
    populate a drug id from OCR rather than only from manual verification. Until then, id-based
    matching only fires on reviewed documents; everything else takes the null/unknown branch.

## 3. Plan for implementation

### 3.1 Guiding principles

- **Port + adapt, don't fork.** The Auth Review feature is the existing patient prior-auth
  feature with (a) one extra order-scoping filter on the list and (b) new chrome (MUI X
  DataGrid list + takeover header). The business logic — fetching intakes, transforming them,
  the 3-step review, the PATCH save — is identical. Shared logic moves into the service layer;
  order-specific routes/UI are thin wrappers on top.
- **Mirror the existing review sub-tab pattern exactly.** Clinical Reviews and Financial Review
  already establish the integration shape inside the order detail (real routes + a reused
  panel). Auth Review follows the same shape — no new pattern invented.
- **TDD throughout** (red-green-refactor per the repo standard).
- **Gate everything** behind `ORDER_DETAILS_AUTH_REVIEW` at both UI and server.

### 3.2 How it plugs into Order Details v2 (routing & sub-tab integration)

Order detail shell discovered in the codebase:

- Route: `app/orders-tracker/[category]/[orderId]/` (`category` ∈ `new | maintenance | lead`;
  "new order detail" = `category === 'new'`).
- `layout.tsx` (`OrderDetailsLayout`) is the shell: top tabs (Order Dashboard / Order History /
  Documents), the order header card, then it conditionally renders `children` per sub-route
  (each gated by a feature flag).

### 3.4 Data layer — the order-scoped filter (the only genuinely new logic)

Implemented **server-side** in the list API route, reusing the existing prior-auth query
building blocks.

- Load order context from `neworders` by `orderId`: `createdAt` (order creation date — TBD
  provisional per 2.10) and `drug` (stringified `lidrugs._id`). Resolve the patient
  `we_infuse_id_IV` from the order's patient link (confirm exact link field during
  implementation; `OrderDetailsData.patientId` is the UI-layer handle).
- Extend `services/priorAuth/query.ts` with `buildOrderScopedPriorAuthMatch({ orderCreatedAt,
  orderDrugId })` returning a `$match` with `$expr: { $and: [dateCondition, drugCondition] }`,
  appended **after** `buildPriorAuthBaseMatchQuery`:
  - **Date condition:**
    `(dateOfDecision.value → $dateFromString with onError/onNull) ?? intake.createdAt`
    must be `$gte` `orderCreatedAt`. (Decision date is an OCR `"YYYY-MM-DD"` **string**; fallback
    `intake.createdAt` is already a `Date` — see 2.10.)
  - **Drug condition:** `priorAuthData.liDrugId.value == orderDrugId` (string equality)
    **OR** `liDrugId` null/missing.
- `orderCreatedAt`/`orderDrugId` are scalars injected as literals → **no `$lookup` to
  `neworders`** needed.
- Refactor the existing patient list pipeline (in
  `app/api/patient/[patientId]/prior-auth/route.ts`) into a shared
  `buildPriorAuthListPipeline({ patientWeInfuseId, filters, sort, pagination, orderScope? })`
  so the patient route and the new order route share filtering/sort/pagination, with the order
  route passing `orderScope`.
