## 🔴 Critical: parent-report reconciliation has a real race window that lets `failed` children leak under a `sent` parent

`reconcileParentReport` in `apps/web/services/weeklyProviderReport/applySendWebhook.ts:59-62` only flips the parent when its current status is `sent`:

```ts
await WeeklyProviderReport.findOneAndUpdate(
  { _id: reportId, status: 'sent' },
  { $set: { status: 'failed' } }
);
```

But the producer in `apps/web/services/jobs/definitions/sendWeeklyProviderReport.ts` doesn't set the parent to `sent` until *after* the entire dispatch loop completes (line 317-319). During the loop, `parent.status` is `'pending'` — set by `apps/web/app/api/weekly-provider-reports/[id]/send/route.ts:88` when the user kicks off the send.

### Failure ordering

1. Job inserts send records 1…N as `pending`, calling Spruce on each.
2. A webhook for send N arrives (last of the batch, failure case) *before* the job's final `findByIdAndUpdate` runs.
3. Reconciler sees `pendingRemaining === 0`, sees `anyFailed`, but the filter `status: 'sent'` doesn't match — parent is still `'pending'`. No-op.
4. Job then sets parent to `'sent'` because `succeededCount > 0`. Note: `succeededCount` only counts **dispatch** success, not delivery success — line 290 increments on a 2xx from Spruce, well before Spruce's delivery webhook returns.
5. Final state: parent `sent`, child `failed`. Nothing ever fixes it.

The window is narrow but not theoretical — Spruce delivery webhooks for fast channels (email) routinely fire within hundreds of ms. The test on lines 393-405 of `applySendWebhook.test.ts` (“leaves the parent report alone when pending sends remain”) shows the team thought about the inverse case but missed this one.

### Suggested fix

Either:
- Widen the filter to `status: { $in: ['sent', 'pending'] }`, or
- Move the parent terminal-state decision into the reconciler instead of the job loop.

The simpler change is broadening the `$in`.

## 🟡 Worth flagging (borderline critical, depends on intent)

### `succeededCount` reflects API accept, not delivery

Line 290 in `sendWeeklyProviderReport.ts` increments `succeededCount` based on `dispatchOneSend` returning `true`, which only means Spruce accepted the request. Combined with the issue above, this means the job sets parent to `sent` regardless of whether actual delivery succeeded.

The webhook reconciliation is the only thing that downgrades, and per the issue above it can be missed.

Same root cause — flagging because the implication is: a 100%-failed-delivery batch can show as `sent` if all webhooks land before line 317-319 runs.
