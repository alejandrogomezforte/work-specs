# [MLID-2306] feat(auth-review): Order Details — Auth Review sub-tab

## Summary

- Adds an **Auth Review** sub-tab inside the order detail view at
  `/orders-tracker/new/[id]/auth-review` (list) and
  `/orders-tracker/new/[id]/auth-review/[letterId]` (single-letter review).
- Ports the existing patient prior-auth experience — same `Intake` collection,
  same 3-step review wizard, same patient-based ownership rules — but filtered
  **server-side to documents relevant to that specific order** (by decision date
  and drug), rebuilt on the `@repo/ui` `DataGrid` with multi-select
  `Autocomplete` filters and a new takeover header (Back + a real Review Status
  selector).
- Everything is gated behind a new `ORDER_DETAILS_AUTH_REVIEW` feature flag at
  both the UI and the server. The flag defaults to `false`.
- The existing patient prior-auth review screen is refactored to consume the
  same shared wizard hook and shared service functions, so both surfaces stay in
  lock-step with zero behavior change on the patient side.

## What changed and why

### Server: order-scoped prior-auth pipeline

- `services/priorAuth/query.ts` gains two shared, exported builders:
  - `buildPriorAuthListPipeline(...)` — the full list aggregation (match → sort →
    facet for count + paginated data), now used by **both** the patient route and
    the new order route. The patient list route is refactored to delegate to it
    with no behavior change.
  - `buildOrderScopedPriorAuthMatch(...)` — an extra `$match` appended only when
    an `orderScope` is supplied, narrowing intakes to the order's date and drug
    context. It is injected as literal scalars (order `createdAt`, drug id, drug
    names) — no `$lookup` back to `neworders`.
- Date condition: effective decision date =
  `$dateFromString(ocrData.authorization.dateOfDecision.value)` (a `YYYY-MM-DD`
  string, present on a minority of intakes) coalesced to `intake.createdAt`;
  included when `>= order.createdAt`.
- Drug condition (symmetric fuzzy rule): show when the verified
  `priorAuthData.liDrugId` hard-matches the order drug id, OR — when no
  `liDrugId` is set — when either of the order's `name` / `genericName` is a
  case-insensitive substring of either the document's OCR `brandDrugName` /
  `genericDrugName`, OR both OCR names are empty (passthrough). A `$type` string
  guard wraps each name field so a malformed (e.g. array) value cannot throw at
  query time.

### Server: shared detail service + order routes

- `services/priorAuth/detail.ts` (new) extracts `getPriorAuthDetail(...)` and
  `applyPriorAuthReview(...)` out of the patient detail route into shared
  functions returning discriminated results (`not_found` / `ownership_error` /
  `no_fields`). The patient `[intakeId]/route.ts` is refactored to delegate; the
  order routes reuse the same logic.
- New order-scoped routes under `app/api/orders/new/[orderId]/auth-review/`:
  `route.ts` (list), `filter-options/route.ts` (order-scoped distinct
  drugs/payors), and `[letterId]/route.ts` (detail GET + PATCH). Each enforces
  session (401) and the `ORDER_DETAILS_AUTH_REVIEW` flag (403), resolves the
  order → patient, and verifies ownership by
  `intake.patient.patientId === patient.we_infuse_id_IV` (403 on mismatch, never
  404, to avoid leaking existence).
- `models/NewOrder.ts` + `services/mongodb/newOrders.ts` — a lean Mongoose model
  (`strict: false`, only `patient` / `drug` / `createdAt` typed) and a
  `getNewOrderById` helper to load just the fields the routes need.
- `services/mongodb/liDrugLookup.ts` — `getLiDrugNamesById` resolves the order's
  drug id to `{ name, genericName }` (Mongoose) for the fuzzy drug rule.

### Client: list, detail, and shared wizard

- `auth-review/_components/AuthReviewIndex.tsx` — the list: `@repo/ui` `DataGrid`
  with Received / Drug / Payor / Letter Type / Auth Valid Period / Review Status
  columns, four multi-select `Autocomplete` filters, active-filter chips with
  "Clear all filters", and an embedded preview side panel.
- `auth-review/_components/AuthReviewDetail.tsx` — the detail takeover: Back +
  "Auth Review" header, a Review Status selector (a reviewed letter can be
  reverted to "Needs Review", guarded by a confirmation; a letter that still
  needs review is shown as a static label and only becomes reviewed by completing
  the wizard), the 3-step wizard, an inline document viewer, and an admin-only
  "remove from Prior Auth" panel.
- `usePriorAuthReviewWizard.ts` (new shared hook) — the step orchestration
  (single-step vs multi-step letter types, step 2 validity gate, start-on-step-3
  for already-reviewed letters, finish vs save payloads). `PriorAuthReview.tsx`
  (patient view) is refactored to consume it; the patient experience is
  unchanged.
- `OrderClinicalReviewsPanel.tsx` / `layout.tsx` wire the new tab. With the flag
  on, the order detail shows three tabs in order: **Clinical Review**,
  **Auth Review**, **Financial Review**.
