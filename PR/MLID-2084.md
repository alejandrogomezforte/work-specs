# [MLID-2084] feat(notifications): order-create backfill + badge perf

## Summary

- Closes the QA-fail on MLID-2084 where uploading documents to a patient **before** creating an order left the new order's badge counter at 0. The upload-time `notifyDocumentCreated` trigger only fires when an order already exists — documents predating the order produced no notifications.
- Adds an order-creation-time trigger `notifyOrderOfRecentDocuments` that backfills notifications for documents the patient already had in the last 30 days. Same notification shape and same `documentAdded` SignalR event as the existing trigger, so the badge UI requires no changes.
- Cleans up several perf problems surfaced during QA: per-document SignalR fan-out, per-event refetch of every visible order, and the >100-orderIds cap on the counts endpoint.

## Root Cause

`apps/web/services/notifications/documentNotification.ts` only writes notifications when a document is created and an order already exists. The reverse flow (documents uploaded first, order created afterwards) had no path to write notifications for the pre-existing documents — so badge counts on the newly-created order were always 0 until a *future* document upload happened.

## What Changed and Why

### 1. Backfill trigger on order create

**Before:** `notifyDocumentCreated` was the only path to write a `(order, document)` notification. It fires at document-upload time and walks every active order for the patient. Documents uploaded before any order exists produced zero notifications.

**After:** `notifyOrderOfRecentDocuments({ orderId, assignedTo, patientId })` runs at order-creation time. Behaviour:

1. Gated by the existing `ORDER_DOCUMENT_NOTIFICATIONS` feature flag.
2. Skips silently if `assignedTo` or `patientId` is falsy (matches existing upload-time behaviour for unassigned orders).
3. Calls `getDocumentsByPatientId(patientId, { since: now − 30d })`.
4. Writes one `user`-type notification per document via the shared primitive.
5. Emits a single coalesced `documentAdded` SignalR broadcast for the order (last document's metadata as a representative payload).

The 30-day window is a typed inline constant (`BACKFILL_LOOKBACK_DAYS = 30`). Not exported, not an env var — promotable later if Ops wants per-environment tuning.

The wiring lives in `apps/web/app/api/orders/new/route.ts` after `autoAssignOrders` and before `postOrderCreatedSlackNotification` (matches the existing fire-and-forget pattern next to `writeAuditLog`).

### 2. Broadcast coalescing

**Problem found in QA:** the first cut of the backfill helper looped one broadcast per document. For a patient with 20 in-window documents, that was 20 `documentAdded` events → 20 badge-refresh round-trips per connected user.

**Fix:** writes are still one-per-document (each notification row is a real, distinct entity), but the broadcast is coalesced to one event per order. Client listeners only treat `documentAdded` as a refresh trigger — they don't read the payload — so a single event is sufficient.

Implemented by splitting `dispatchOrderDocumentNotification` into two private primitives (`writeOrderDocumentNotification` and `broadcastDocumentAddedEvent`). `notifyDocumentCreated` continues to compose them as before (one write + one broadcast per assigned order — observable behaviour unchanged).

### 3. Badge hook: targeted refetch and additive skip

**Problem found in QA:** every `documentAdded` event triggered `useOrderNotificationCounts` to POST `/api/notifications/counts` with **every** `_id` in `newOrders.data`. With pageSize 100 plus one locally-appended order, that's 101 ids → 400 against the route's cap. And the same payload fired multiple times per badge update.

**Fix:**
- **On `documentAdded`**: read the `orderId` from the event payload; if it's in the visible set, refetch only that one id and merge the response into state. The hook no longer wastes bandwidth refetching unchanged counts.
- **On `orderIds` change**: keep the bulk refetch for non-additive changes (filter, pagination), skip it when the change is purely additive (e.g. the local `addLocalOrders` after creating an order). The SignalR `documentAdded` handler refreshes the new order's count; the bulk refetch is redundant.
- **On `notification:read`**: unchanged. Its payload doesn't carry `orderId`, so a full refetch is still correct.

Net effect: creating one order = one POST with one id = 200 OK. Down from 20 POSTs each with 101 ids before the fix.

### 4. Counts endpoint cap

Raised `MAX_ORDER_IDS` from 100 → 500 in `apps/web/app/api/notifications/counts/route.ts`. The cap is no longer load-bearing — it's headroom for the rare case where a client's local list has grown past one page on a real filter/page change. Bumped after the hook hygiene was fixed, not as a workaround for the fan-out.

### 5. Coverage backfill for `document.ts`

The feature lands `services/mongodb/document.ts` at the project's 80% coverage threshold. Added tests for the previously-uncovered functions (`updateDocumentEmbeddingStatus` branches, `updateDocumentCategoryAndDescriptionForIntakePdf` paths, and the four catch blocks on the read functions). File-level statement coverage moved from 66.92% → 91.73%.
