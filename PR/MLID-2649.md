# [MLID-2649] chore(mcp): migrate Atlassian MCP server from SSE to Streamable HTTP transport

## Summary

- Atlassian is retiring the HTTP+SSE transport endpoint (`https://mcp.atlassian.com/v1/sse`) after **30 June 2026**.
- The committed, team-shared `.mcp.json` still pointed the `atlassian` MCP server at that SSE endpoint, so after the cutoff the Atlassian MCP integration would stop connecting ÔÇö breaking all Jira/Confluence MCP tooling in Claude Code for the whole team.
- This PR swaps the `atlassian` entry to the Streamable HTTP transport (`type: "http"`, `url: ÔÇŽ/v1/mcp`). Because the file is committed, this single change migrates everyone.

## Root Cause

Not a bug ÔÇö a proactive deprecation migration. The Atlassian Remote MCP server has been emitting a deprecation notice on every response stating that the `/v1/sse` HTTP+SSE transport will be unsupported after 30 June 2026, and that clients should move to the Streamable HTTP endpoint at `/v1/mcp`. Our `.mcp.json` had not yet been updated.

Reference: https://community.atlassian.com/forums/Atlassian-Remote-MCP-Server/HTTP-SSE-Deprecation-Notice/ba-p/3205484

## What Changed and Why

The `atlassian` server entry in the repo-root `.mcp.json` was migrated from the deprecated SSE transport to the Streamable HTTP transport.

Before:

```json
"atlassian": {
  "type": "sse",
  "url": "https://mcp.atlassian.com/v1/sse"
}
```

After:

```json
"atlassian": {
  "type": "http",
  "url": "https://mcp.atlassian.com/v1/mcp"
}
```

Design notes:

- `type: "http"` is Claude Code's identifier for the Streamable HTTP transport ÔÇö the transport behind the `/v1/mcp` endpoint that Atlassian's deprecation notice points to.
- Existing Atlassian OAuth authorization carries over; no re-authentication is required.
- The change takes effect on the next Claude Code restart, since MCP servers connect at startup.
- The other server entries (`li-mcp`, `mongodb`, `playwright`) are unaffected by the Atlassian cloud deprecation and were intentionally left unchanged. In particular, `li-mcp` also uses an SSE transport but points at a local server (`http://localhost:3002/phi/sse`), which is out of scope for this Atlassian-specific migration.

No code, dependencies, or environment variables are involved ÔÇö this is a single configuration swap.
