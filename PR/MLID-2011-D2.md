# [MLID-2011] feat: Document Notifications — SignalR + API Routes + Triggers (D1+D2)

## Summary

- **SignalR broadcast integration** — server-side `broadcastToAll()` function for pushing real-time events to all connected clients via Azure SignalR Service
- **Feature flag** `ORDER_DOCUMENT_NOTIFICATIONS` — gates all notification behavior at both API and service layers
- **Notification data model** — `Notification` + `NotificationRead` Mongoose models with full CRUD service (PE delivery, PR #1049)
- **Notification API routes** — `GET /api/notifications` (paginated list + unread count), `PATCH /api/notifications/:id/read` (single mark read), `PATCH /api/notifications/bulk-read` (bulk mark read)
- **Document creation triggers** — fire-and-forget `notifyDocumentCreated()` helper wired at all document creation entry points (manual upload + JotForm/WeInfuse job), creating `user`-type notifications targeting the order's assigned user and broadcasting `documentAdded` SignalR events

## What Changed and Why

### 1. SignalR Broadcast (MLID-2080)

Added `broadcastToAll()` to the existing SignalR service. This is the server-to-client push mechanism — the server POSTs to Azure SignalR REST API, which fans out to all connected browsers. Each browser decides if the event is relevant to its current view.

The negotiate endpoint and client context were built by the PE in a prior PR merged directly to develop.

### 2. Notification Data Model (MLID-2081, PE delivery)

Two collections: `notifications` (one doc per event, shared across users) and `notificationreads` (per-user read receipts for `user`-type notifications). Two notification types: `task` (any team member can acknowledge, inline read tracking) and `user` (per-user independent read state).

Document notifications use `user` type with `targetUserId` set to the order's assigned user.

### 3. Notification API Routes (MLID-2082)

Three REST endpoints following the project's auth → feature flag → validate → execute → respond pipeline:

- **GET** returns `{ data: INotification[], unreadCount: number }` — no `totalCount`/`totalPages` since the service doesn't provide it. Accepts `limit` (max 100, default 50) and `offset` (default 0).
- **PATCH single** calls `markUserNotificationAsRead()` (idempotent) then broadcasts `notification:read`.
- **PATCH bulk** validates `notificationIds` (non-empty string[], max 100) then calls `bulkMarkUserNotificationsAsRead()` and broadcasts.

### 4. Document Creation Triggers (MLID-2083)

**Investigation finding**: ALL document creation goes through a single function `createDocumentRecord()`. Two active entry points call it: the split-pdf upload route and the JotForm/WeInfuse background job.

**Design decision**: Created a dedicated `notifyDocumentCreated()` helper called at each call site (not inside `createDocumentRecord()`) because `source` metadata (`upload` vs `e-order`) is only knowable at the call site. The helper:

1. Checks `ORDER_DOCUMENT_NOTIFICATIONS` feature flag
2. Looks up the patient to get `we_infuse_id_IV`
3. Queries active `neworders` by WeInfuse patient ID to find `assignedTo`
4. For each assigned order: creates a notification + broadcasts SignalR event
5. Skips unassigned orders (no user to notify)
6. All errors caught and logged — never propagates to callers

Uses `void` (fire-and-forget) at call sites, matching the existing non-blocking pattern for OCR job scheduling.

**Spruce fax note**: Fax documents only reach the `documents` collection when a user manually processes them via the split-pdf route. Notifications fire at manual processing time, not fax receipt.
