# [MLID-2015] feat(pharmacy-eligibility): exclude treatments from pharmacy eligibility

## Summary
- Treatments (`is_treatment == true`) should never be pharmacy eligible
- Added a new `$switch` branch to the pharmacy eligibility computation that short-circuits to `"No"` when the drug is a treatment
- The new gate is evaluated after the insurance-exists check and before the drug/location/insurance eligibility checks

## What Changed and Why

### Pipeline change (`ordersTracker.ts`)

Inserted a new branch at index 1 in the `pharmacyEligible` `$switch.branches` array:

```javascript
{
  case: { $eq: [{ $ifNull: ['$drug.is_treatment', false] }, true] },
  then: 'No',
}
```

**Branch order after change:**
1. No primary insurance → `"Waiting on Insurance Input"`
2. Drug is a treatment → `"No"` **(NEW)**
3. All conditions met → `"Yes"`
4. Default → `"No"`

Uses `$ifNull` with `false` default to handle cases where `drug` is an unresolved string (same pattern as `$drug.pharmacyEligible`). No new lookups needed — `drug` is already expanded by `expandIdsAggregate`.

### Tests (`ordersTracker.test.ts`)

Two new test cases:
1. **Branch shape**: Verifies `branches[1]` contains the `is_treatment` check with correct `$eq`/`$ifNull` structure and `then: 'No'`
2. **Branch ordering**: Verifies the sequence is `Waiting on Insurance Input` → `No` → `Yes`
