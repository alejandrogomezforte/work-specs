# [MLID-1819] feat(order-eligibility): 340B background jobs — hospital system triggers

## Summary

- Add Pulse background job infrastructure for 340B eligibility recalculation triggered by hospital system changes
- Implement NPI diff strategy: only orders tied to added/removed NPIs are recalculated (unchanged NPIs are skipped)
- Implement hospital rename trigger: recalculate all orders tied to a renamed system's NPIs
- Modify `bulkUpdateProviders()` to return `previousNpis` for diff computation
- Wire triggers into bulk providers API route and hospital PUT route (fire-and-forget)
- Use `FeatureFlagService.getFeatureFlag()` (direct MongoDB query) instead of `isFeatureEnabled()` (HTTP self-call) to avoid silent auth failures in server-side code

## What Changed and Why

### 1. NPI Diff Strategy (`triggers/hospitalNpiUpdate.ts`)

When a hospital system's NPI list is bulk-updated, only the **difference** matters. The trigger computes `added = newNpis \ oldNpis` and `removed = oldNpis \ newNpis`. Only orders tied to these affected NPIs are recalculated. Re-uploading the same list results in zero recalculations.

### 2. Hospital Rename Trigger (`triggers/hospitalRename.ts`)

The stored 340B eligibility value includes the hospital system name (e.g., `"Baptist Health"`). When a system is renamed, all orders tied to any of its NPIs must be recalculated to reflect the new name. The trigger detects name changes by comparing the old name (fetched before update) with the new name.

### 3. `bulkUpdateProviders()` Returns Previous NPIs

To compute the NPI diff, the bulk update function now fetches the current provider list via `findOne()` before performing the `findOneAndUpdate()`. The previous NPIs are returned alongside the updated document so the API route can pass both old and new lists to the trigger.

### 4. Job Handlers with Per-Order Error Handling

Both job handlers share a `recalculate340BForOrders()` helper that:
- Queries `neworders` by `provider IN [npis]`
- Calls `compute340BForOrder()` per order
- Persists `hospital340BEligible` via `updateOne()`
- Catches per-order errors and continues (batch resilience)
- Calls `done()` with error only on job-level failures

### 5. `FeatureFlagService.getFeatureFlag()` over `isFeatureEnabled()`

During local testing, jobs were not being scheduled despite the feature flag being enabled in MongoDB. Root cause: `isFeatureEnabled()` makes an HTTP fetch to `${NEXTAUTH_URL}/api/feature-flags/ORDER_ELIGIBILITY`, but that endpoint requires an authenticated session. The internal fetch from server-side code has no cookies, gets 401, and silently falls back to the default value (`false`). Fixed by using `FeatureFlagService.getFeatureFlag()` which queries MongoDB directly — the correct approach for server-side service code.

### 6. Fire-and-Forget Pattern

Both API routes call triggers inside a `try/catch` block after the main response data is ready. Trigger errors are logged but never fail the API response. This ensures hospital system CRUD operations are not blocked by eligibility calculation issues.
