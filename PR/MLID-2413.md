# [MLID-2413] fix(notifications): cleanup notifications + write audit on document unlink

## Summary

- Bug fix for the order-notification badge counter mismatch when a `category: "Order"` document is unlinked from a new order. Reported by Olha.
- Adds `handleOrderUnlink(documentId, removedOrderIds, options)` — symmetric pair with the renamed `handleOrderLink` — that runs four steps per removed orderId: delete matching `notifications` rows, delete the associated `notificationreads` rows, write a remove-side `auditLogs` entry on the order, and broadcast `documentUnlinked` via SignalR.
- Order-history "Change" column renderer now produces the PO-specified text for `category: "Order"` rows on both link and unlink, while keeping the legacy `New Document received (Document type = X)` text for every other category.
- Includes a rename of `firePropagationForNewlyLinkedOrders` → `handleOrderLink` (the existing single-caller add-side helper) for symmetric naming with the new unlink helper.

## Root Cause

`PUT /api/documents/[documentId]/link-orders` only fires propagation for the **added** delta of order IDs. When the linked-orders set shrinks (an unlink):

- Notification records are left orphaned in `notifications` — still counted by the badge query, still rendered as "New" on the order's Documents tab.
- No `notificationreads` cleanup runs.
- No `auditLogs` entry is written for the remove side — Order History shows no record of the unlink.

The fix computes the removed delta in the same route handler and dispatches to the new orchestrator alongside the existing add-delta path.

## What Changed and Why

### `handleOrderUnlink` orchestrator (4-step per-orderId cleanup)

For each removed `(orderId, documentId)` pair, the helper runs in order:

1. **Delete `notifications`** — query with the dual-`$elemMatch` filter (`$and` of two `$elemMatch` clauses on order entity + document entity). Dot-notation is intentionally avoided because it cross-matches different array elements and produces false positives — the trap is documented inline in `notification.test.ts:319-342`. The helper first runs `find(...).select('_id').lean()` to capture the IDs (Mongoose `deleteMany` does not return them), then runs `deleteMany` with the same filter.
2. **Delete `notificationreads`** — driven off the IDs collected in step 1. Per PO direction (2026-05-26) the unlink cleanup is now complete — the prior audit-retention exemption was reversed.
3. **Write a remove-side `auditLogs` entry** — `action: 'update'`, `source: 'user'`, `actorId: <session user>`, `changes: [{ field: 'documents', from: { _id, category, source }, to: null }]`. Structural inverse of the existing `writeOrderDocumentAuditEntry`.
4. **Broadcast `documentUnlinked { orderId, documentId }`** — new SignalR event, did not exist anywhere else in the codebase. Both client hooks (Documents tab + orders-tracker table) subscribe to it.

Per-step failure semantics:

- Notifications-delete failure → skip steps 2-4 for THIS order (consistency — no orphaned reads, no audit claiming an unlink that didn't happen at the DB level); continue to the next order in the loop.
- Reads-delete failure → log and continue to audit + broadcast (the unlink already happened at the DB layer).
- Audit step never throws (the helper is no-throw by contract).
- Broadcast failure → log; DB cleanup is already applied.

Each step has its own inner try/catch; the orchestrator itself never throws — mirroring the established `handleOrderReassignment` pattern.

### Order-category-scoped renderer

```typescript
documents: {
  label: 'New Document',
  category: 'Documents',
  renderChange: (change: AuditLogChange) => {
    const from = change.from as { category?: string } | null;
    const to = change.to as { category?: string } | null;
    const category = from?.category ?? to?.category ?? 'Unknown';

    // PO-specified text — Order category only.
    if (category === 'Order') {
      if (change.from === null) {
        return 'A Document has been added (Document type = Order)';
      }
      if (change.to === null) {
        return 'A Document has been removed (Document type = Order)';
      }
    }

    // Legacy format preserved for every other category.
    return `New Document received (Document type = ${category})`;
  },
},
```

Per PO direction, only `category: "Order"` documents get the new "added/removed" text. All other categories (Referral Copy, Other, Lab Results, etc.) keep the legacy `New Document received (Document type = X)` text unchanged. This was verified visually during local UI testing — Order Category rows render the new text while Referral Copy / Other rows on the same order continue to use the legacy text.

### Rename: `firePropagationForNewlyLinkedOrders` → `handleOrderLink`

A single-caller helper (the link-orders route) — renamed for symmetric naming with `handleOrderUnlink`. The single production import site is updated, both test files updated, the log message refreshed to match the new style. The function stays in `documentNotification.ts` because it depends on internal helpers there (`createSignalRResolver`, `propagateDocumentNotificationToOrder`). Zero behavior change.

### Test file consolidation

A duplicate test file was created at `apps/web/app/api/documents/[documentId]/link-orders/route.test.ts` during the TDD cycle and later deleted; its content was merged into the existing `__tests__/route.test.ts`. One test file per route now, matching the codebase's dominant co-located-tests convention.

### Out of scope (separately tracked)

A pre-existing reassignment-cascade gap surfaced during this session's UI testing: when an order is created with `assignedTo: ""` and later assigned to a user via the standard reassignment flow, the non-Order documents that auto-attached prior to assignment never get backfilled notifications. `handleOrderReassignment` early-returns when `previousAssignedTo` is empty (it is a retarget, not a create). This is a MLID-2391 gap, not in scope here; it will be picked up on a separate hotfix branch.
