# [MLID-2391] fix(orders): reassignment cascades notifications to new assignee (neworders)

## Summary

- Reassigning a new order now retargets every existing document notification from the previous assignee to the new one (one `Notification.updateMany` per order, fire-and-forget). Closes the gap where new assignees saw no badges or chips for documents on orders reassigned to them, and previous assignees kept seeing stale ones.
- New assignees see every document as **unread** — including ones a prior assignee had marked read — because the unread predicate now joins on the notification's **current** `targetUserId`. Prior `notificationreads` rows are retained for audit but no longer participate in the join (their `userId` no longer matches).
- Three workarounds added under MLID-2011 / MLID-2085 to mask the staleness are removed in the same PR: `userReadScope: 'any'` on the entity-count routes, the `isCallerOrderAssignee` server fallback on mark-as-read, and the relaxed `orderAssignedTo === userId` clause in the client chip filter. With `targetUserId` always live, each becomes either dead code or incorrect (over-show / over-authorize).
- A new SignalR event `order:reassigned { orderId }` triggers live refetch on both the orders-tracker table badge and the Documents-tab badge/chips.

## Root cause

`notification.targetUserId` was a write-time snapshot of `order.assignedTo` and was never updated on reassignment. The new assignee then failed every strict `targetUserId === userId` check. MLID-2085 patched the symptom on the read side by falling back to the order's current `assignedTo`, but the notification documents kept carrying stale `targetUserId` values, with all the cost of the fallbacks. This PR fixes the write side at the source and removes the read-side workarounds.

## Why retarget (not delete + recreate)

A delete-and-recreate model was prototyped and rejected because the recreate path reuses `notifyOrderOfRecentDocuments`, which only looks back 30 days — reassigning an order older than that window would silently lose notifications. Recreate also fans out N false-positive audit entries per reassignment. Retarget is one Mongo write per order, preserves notification `_id` (so `notificationreads` rows stay valid), and gives the PO requirement for free via the predicate join.

## Migration / visible side effect after deploy

For any order that was reassigned before this PR ships, recorded `notificationreads` rows carry the previous assignee's `userId`. Under the new current-target predicate they stop counting toward "read." **Badge counts on historically-reassigned orders will recompute and may increase** as soon as this ships. No data migration is involved — it's a query-side reinterpretation. Confirmed empirically against staging: a real order with 16 notifications and prior reads from two former assignees showed **11 unread on app-stage (develop) vs 13 on this branch**, both against the same database — the delta is exactly the 2 prior-assignee receipts the old `userReadScope: 'any'` was counting.
