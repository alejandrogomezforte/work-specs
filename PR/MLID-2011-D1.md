# [MLID-2011] feat(notifications): D1 — SignalR broadcast primitive + notification data model

## Summary

This PR delivers **Deliverable 1** of the MLID-2011 epic (Orders: Document Notifications). It bundles two sub-tasks that together complete the backend foundation needed for document notifications:

- **D1-T1 (MLID-2080)** — Feature flag `ORDER_DOCUMENT_NOTIFICATIONS` + server-side `broadcastToAll()` function that calls the Azure SignalR Service REST API. Also lifts three SignalR constants (`HUB_NAME`, `TOKEN_EXPIRY`, `REST_API_VERSION`) into environment variables with fallback defaults.
- **D1-T2 (MLID-2081)** — `Notification` + `NotificationRead` Mongoose models and the full notification service layer (CRUD, counts, read tracking, targeting with OR logic, expiry filtering). Delivered by Anatoliy under PR #1049 (referenced as `MLID-2011` in the PR title, not the sub-task key).

After this PR merges, D2 (document triggers + notification API routes) can start. No feature is visible to end users yet — the feature flag is `false` by default and there are no UI entry points.

## What Changed and Why

### 1. `broadcastToAll` — the first real HTTP call to Azure SignalR

The PE's earlier work (PR #1046) built the SignalR **auth layer** — token generation, client-side negotiate endpoint, React context — but never actually called the Azure SignalR Service REST API. Without a broadcast primitive, nothing can push events from the server to connected clients.

`broadcastToAll(eventName, payload, hubName?)` is the missing piece:

```ts
POST {endpoint}/api/hubs/{hub}/:send?api-version=2022-06-01
Authorization: Bearer {serverToken}
Content-Type: application/json

{ "target": "documentAdded", "arguments": [{ "documentId": "..." }] }
```

Key design decisions:

- **Throws on failure** instead of silently swallowing errors. The D2 layer (document triggers) will decide what to do if the broadcast fails — wrap in try/catch for fire-and-forget semantics if desired, or propagate upward for retry. Burying errors in the primitive would make that choice impossible.
- **Generic API** — `broadcastToAll` is not tied to document notifications. Any future feature can call `signalrService.broadcastToAll('whateverEvent', data)`.
- **Broadcast to all, no rooms** — The epic plan proposed SignalR rooms as "topic channels per feature," but after discussing with the PE we decided to defer rooms until there's a second feature using SignalR. Current implementation broadcasts to all connected clients; frontend filters by event name. Tracked as Open Question #1 resolution in the plan.
- **`REST_API_VERSION = '2022-06-01'`** — Azure SignalR Service requires an `api-version` query parameter on REST calls. `2022-06-01` is the current stable. This is the first code in the codebase that makes an HTTP call to the service, so this parameter appears here for the first time.

### 2. SignalR constants moved to environment variables

`HUB_NAME`, `TOKEN_EXPIRY`, and `REST_API_VERSION` now read from env vars with current hardcoded values as fallback defaults:

```ts
const HUB_NAME = process.env.SIGNALR_HUB_NAME || 'li-hub';
const TOKEN_EXPIRY = (process.env.SIGNALR_TOKEN_EXPIRY ||
  '1h') as jwt.SignOptions['expiresIn'];
const REST_API_VERSION = process.env.SIGNALR_REST_API_VERSION || '2022-06-01';
```

- **Why:** Different environments (dev/stage/prod) may eventually want different hub names for isolation, different token lifetimes for UX tuning, or different API versions during a rollout. Hardcoding couples the app to a specific SignalR resource.
- **Why the `jwt.SignOptions['expiresIn']` cast:** `jsonwebtoken` requires the `StringValue` template literal type (e.g., `'1h'`, `'30m'`), not generic `string`. `process.env` returns `string | undefined`, so we cast once at the constant declaration.
- **Local dev** already has these three env vars in `apps/web/.env.local`.
- **Staging/prod infra** — the fallback defaults mean nothing breaks if Anatoliy hasn't added the env vars yet. Tracked as Open Question #7 in the plan — need guidance on how env vars are pushed into Azure infra (container env, Key Vault references, App Service config, terraform variables).

### 3. Feature flag `ORDER_DOCUMENT_NOTIFICATIONS`

Added to the enum and `DEFAULT_FEATURE_FLAGS` with default `false`. No code reads it yet — it exists so D2/D3 can gate the feature as they wire it in. Standard MongoDB feature flag pattern (no DB doc created until admin toggles).

### 4. Notification data model (PR #1049 — Anatoliy's work)

Two Mongoose collections:

- **`notifications`** — one document per event. Shared across users. Supports three targeting strategies (`targetUserId`, `targetLocationId`, `targetRole`) with OR-logic matching. The `type` field (`task` | `user`) determines read-tracking semantics:
  - `task` type — actionable items any team member can ack. Read state tracked inline (`isRead` / `readBy` / `readAt`). First ack clears it for everyone in the target group.
  - `user` type — informational, each user tracks their own read state via the separate `notificationreads` collection. **This is the type used for document notifications** (only the user assigned to the order can ack — decided in the plan update on 2026-04-07).

- **`notificationreads`** — per-user read receipts for `user`-type notifications. Unique on `(notificationId, userId)`.

Service layer provides everything the UI will need: list, counts (`$facet`-based aggregation covering both read-tracking types in a single query), mark-as-read (single + bulk), entity-scoped count queries. PR review iterations added: type guard on `markTaskAsRead` preventing corruption of user-type notifications, `MongoBulkWriteError` handling on bulk inserts, expiry filtering in all user-facing queries, `_id` type fix (let Mongoose Document handle `_id` typing instead of overriding to `string`), target field validation via `pre('validate')` hook, `$elemMatch` in `getCountForEntity` to avoid cross-element false positives on the `entities[]` array.

### 5. Why some D1 sub-task descriptions look re-shuffled

The PE and I aren't perfectly aligned on ticket reference conventions — his PRs reference the epic key (`MLID-2011`), mine reference the sub-task key (`MLID-2080`). The original plan had D1-T2 described as "Frontend: SignalR React context + provider" with branch name pointing at the notification-data-model branch — confusingly crossing the wires. Reality:

- **PR #1046 (MLID-2079 — already merged to develop)** — SignalR service foundation + negotiate endpoint + React context + provider wiring. This was the D0 infra code for the app side.
- **PR #1049 (MLID-2011 — merged to epic)** — the notification data model and service. This is what D1-T2 should have been described as.
- **PR #2080 (this PR's D1-T1 chunk, merged to epic via `feature/MLID-2080-signalr-broadcast-feature-flag`)** — feature flag + broadcast primitive.

The plan tracking table has been updated to reflect this.
