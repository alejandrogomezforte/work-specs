## Summary
Maintenance orders reassigned **before the `v2026.07.14` release** left their `type: "user"` document notifications pointing at the **previous owner**, so the current assignee never received them. The retarget-on-reassignment cascade for maintenance orders (MLID-2454) only shipped in `v2026.07.14`, so any reassignment done before that deploy was not migrated.

A production sweep found **230** stranded notifications, across **16** previous owners and **51** current owners. (The originally reported case â€” order `01wE` â€” was one of the 230.)

This PR adds a **one-time data migration** that retargets each stranded notification's `targetUserId` to its order's current `assignedTo`. It does **not** change any application code â€” the current code already handles reassignment correctly; this only backfills the pre-deploy gap.

## Why a migration (and why two files)
`notifications` is dual-written to Postgres. Migrations run as plain `node`, so they cannot reuse the app's `reassignUserNotificationsForOrder` (TypeScript). Following the established `041` / `0070` pattern, the fix is applied to both stores by two coordinated migrations that run in the same deploy job (Postgres first, then Mongo):

- **Mongo** â€” `apps/web/db-update/migration/043.reassign-stranded-notifications.mjs`
- **Postgres** â€” `packages/db/migrations/0083_mlid_2799_reassign_stranded_notifications.sql` (sets `target_user_mongo_id` and the resolved `target_user_id` FK via `users.mongo_id`)

Both apply the **same frozen 230-row mapping** (`{ notificationId, oldTargetUserId, newTargetUserId }`), computed once from a production detection aggregation.

## Safety
- **Idempotent** â€” each row applies only while the notification is still stranded on its recorded old owner; a re-run, or a row already moved by the deployed MLID-2454 code, is a no-op.
- **Backup + rollback** â€” the Mongo side backs up previous `targetUserId` to `migration_043_notification_target_user_backup` and has a working `down`.
- **`DRY_RUN=true`** logs the plan and writes nothing.
- **Pure data migration** â€” `db:generate` is a no-op (snapshot `0083` equals `0082`), and column names/FK were verified against the live Postgres schema.

## Test / apply plan
1. `npm run test` â€” migration unit tests (10/10 pass).
2. **Stage (Postgres):** `DATABASE_URL=<stage> npm run db:migrate` from `packages/db` â†’ validates the SQL runs cleanly (0 rows changed on stage, expected, since the ids are production-only), then confirm a clean `db:generate`.
3. **Prod dry run (Mongo):** trigger the `li-prod-db-migration` job with `DRY_RUN=true` â†’ previews the real retarget count, writes nothing. (Dry run skips the drizzle step.)
4. **Prod live:** the normal migration job applies both (drizzle first, then Mongo).
5. **Verify:** re-run the detection aggregation â†’ expect **0** stranded notifications.
