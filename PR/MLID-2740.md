# [MLID-2740] build(web): migrate LISA web bundler from webpack to Turbopack

## Summary

- Next.js 16 makes **Turbopack the default bundler**; `apps/web` was opting out with an explicit `--webpack` flag (a transitional escape hatch kept during the Next 16 migration, MLID-364). This PR moves `apps/web` onto Turbopack and removes that flag.
- Deletes the ~80-line custom `webpack()` block in `next.config.mjs` and re-expresses (or retires) each workaround using Turbopack-native configuration.
- Fixes two pre-existing issues that webpack silently tolerated but Turbopack rejects: an unsupported `esmExternals` option and two malformed CSS-Modules selectors.
- Adds MUI's `AppRouterCacheProvider` to fix a Turbopack-only emotion hydration mismatch.
- No product behavior change intended — this is build tooling only.

## What Changed and Why

### 1. Scripts — drop `--webpack`
`next dev --webpack -p 8080` → `next dev -p 8080`, and `next build --webpack` → `next build`. With the flag gone, Next 16 uses Turbopack by default for both dev and build.

### 2. `next.config.mjs` — webpack() block → Turbopack config
Turbopack does not execute the `webpack()` function and does not support webpack plugins, so the block is removed and replaced with:

- **`serverExternalPackages: ['ssh2-sftp-client', 'ssh2', 'applicationinsights']`** — leaves these unbundled and `require()`d from `node_modules` at runtime. `ssh2` loads a native `sshcrypto.node` binary that is reachable in the server build via `instrumentation.ts → initialize-jobs → job definitions → *SftpService → baseSftpService.ts`; `applicationinsights` has optional diagnostic-channel sub-imports that only resolve via Node's runtime `require()`. This replaces the old `IgnorePlugin` entries and server externals.
- **`turbopack.resolveAlias` mapping `applicationinsights` to the empty stub in the `browser` condition** — `utils/logger.ts` does a guarded dynamic `import('applicationinsights')` and is imported by ~96 client components; the runtime guard does not remove the specifier from the client module graph, so without this alias `applicationinsights` would leak into the client bundle. Replaces the webpack `resolve.alias.applicationinsights = false`.
- **Removed `experimental.esmExternals: 'loose'`** — Turbopack hard-errors on it (`esmExternals = "loose" is not supported`). Turbopack handles ESM/CJS interop natively; `transpilePackages: ['next-auth', '@auth/core']` continues to bundle the ESM-only next-auth.
- **Dropped entirely (no Turbopack equivalent needed):** the `li-public-api` `IgnorePlugin` (there are no real imports of the sibling NestJS app) and the `node:` `NormalModuleReplacementPlugin` (Turbopack supports the `node:` scheme and externalizes Node built-ins natively).

### 3. CSS — malformed `:global()` selectors
Turbopack's CSS parser (stricter than webpack's css-loader) rejects `.item:global(> span)` / `.dataItem:global(> span)` with "Invalid empty selector". Replaced with plain `.item > span` / `.dataItem > span` — behavior-identical, since tag selectors are never scoped by CSS Modules, and the sibling rules in the same file already use `> span:first-of-type`. A scan of all `.module.css` files confirmed these two were the only malformed instances.

### 4. `AppRouterCacheProvider` — emotion hydration fix
Under Turbopack the app hydration-mismatched on every page (server rendered an emotion `<style data-emotion=css-global>` where the client rendered a CSS-Module div, first hit at `Notifications.tsx`). Root cause: the app had no MUI App-Router emotion cache setup, so emotion injected its global style inline during SSR; webpack's ordering hid it, Turbopack exposed it (confirmed absent on `develop`). Wrapping the app in `AppRouterCacheProvider` flushes emotion styles into `<head>` via `useServerInsertedHTML`, removing the mismatch.

## Deployment note

`apps/web` and the pulse worker run the **same Docker image** (`apps/web/Dockerfile`), which keeps a full, unpruned `node_modules` (no `output: 'standalone'`). The worker executes SFTP jobs via `ts-node` at runtime — a path Turbopack does not touch — so `serverExternalPackages` only changes what the web build bundles, not what is present in the runtime image.
