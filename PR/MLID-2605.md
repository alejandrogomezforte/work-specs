# [MLID-2605] fix(orders-tracker): decouple review tabs so each follows only its own feature flag

## Summary

- The Order Details review panel hosts three sub-tabs — Clinical Review, Prior Auth, Financial Review — but they were coupled to the `CLINICAL_REVIEWS` flag. With `CLINICAL_REVIEWS` off, the entire panel was hidden, so Prior Auth and Financial Review disappeared even when their own flags were on.
- Each tab now follows only its own flag: Clinical Review → `CLINICAL_REVIEWS`, Prior Auth → `ORDER_DETAILS_AUTH_REVIEW`, Financial Review → `ORDER_FINANCIAL_CALCULATION`. `ORDER_DETAILS_ACTIONS` remains the panel-level master switch (whole review panel on/off).
- Renamed the misleadingly-named host component `OrderClinicalReviewsPanel` → `OrderDetailsReviewsPanel`, since it is a generic wrapper for all three review tabs, not a clinical-only panel.

## Root Cause

The review sub-tabs all live inside one shared component. In the order-detail layout, the dashboard mount of that component required `ORDER_DETAILS_ACTIONS && CLINICAL_REVIEWS`. Because the dashboard is the landing path for any order, turning `CLINICAL_REVIEWS` off removed the whole panel — and with it the Prior Auth and Financial Review tabs — regardless of their own flags. Inside the panel, the Clinical Review tab was rendered unconditionally and the active-tab fallback always defaulted to `'clinical'`, so even mounting the panel without `CLINICAL_REVIEWS` would have shown clinical content.

The component's name reinforced the bug: a panel called "ClinicalReviews" naturally got gated behind the clinical flag.

## What Changed and Why

### 1. Dashboard panel mount (layout.tsx)

Before — the panel only mounted when the clinical flag was on:
```tsx
{isDashboardPath && isOrderDetailsActionsEnabled && isClinicalReviewsEnabled && !isLoading && !error ? (
  <OrderDetailsReviewsPanel />
) : null}
```

After — the master switch plus any enabled sub-tab mounts the panel:
```tsx
{isDashboardPath &&
isOrderDetailsActionsEnabled &&
(isClinicalReviewsEnabled || isAuthReviewEnabled || isFinancialReviewEnabled) &&
!isLoading && !error ? (
  <OrderDetailsReviewsPanel />
) : null}
```

### 2. Auth-review route under the master switch (layout.tsx)

The `/auth-review` routed content previously required only `ORDER_DETAILS_AUTH_REVIEW`. It now also requires `ORDER_DETAILS_ACTIONS`, so all three review routes consistently sit under the same panel-level master switch.

### 3. Tab gating and default tab (OrderDetailsReviewsPanel.tsx)

- Added `useFeatureFlag(FeatureFlag.CLINICAL_REVIEWS)`.
- The Clinical Review tab is now conditional on `CLINICAL_REVIEWS` (Prior Auth and Financial Review were already conditional on their own flags).
- `activeReviewTab` falls back to the first enabled tab (clinical → auth → financial) instead of always `'clinical'`, so the panel never renders clinical content when `CLINICAL_REVIEWS` is off.

### 4. Avoid a wasted clinical-reviews fetch (OrderDetailsReviewsPanel.tsx)

The clinical-reviews hook only skips fetching when both its `patientId` and `lisaOrderId` arguments are absent. Because the panel always passed the order id, the fetch ran even with the Clinical Review tab disabled. Both arguments are now gated on `CLINICAL_REVIEWS`, so no clinical-reviews request is made when that feature is off.

### 5. Component rename

`OrderClinicalReviewsPanel` → `OrderDetailsReviewsPanel` across the component, its test, its CSS module, the four importers (the order-detail layout and the three review route pages), the test-id (`order-clinical-reviews-panel` → `order-details-reviews-panel`), and the layout test's mock module path. Pure rename, no behavior change.

### Design decisions

- **`ORDER_DETAILS_ACTIONS` kept as the master switch** rather than removed, so the whole review area can still be toggled as a unit; each tab is independent only beneath it.
- **Auth-review pulled under the master switch** for consistency — a deliberate behavior change so all three routes share one container gate.
- **No backend change**: an audit of the financial calculator UI (`financial-review/_components`, `components/Modules/financial-calculator`) and its API routes (`api/orders/{new,maintenance}/[orderId]/financial-calculations`, `pages/api/patients/[id]/financialCalculation.ts`) confirmed zero `CLINICAL_REVIEWS` coupling. The only coupling was the structural panel mount fixed here.
