# [MLID-1955] fix(locations): change isPharmacyEligible default from true to false

## Summary
- Fix data inconsistency where locations missing `isPharmacyEligible` in MongoDB showed "Enabled" in the UI but "No" in the Orders Tracker pharmacy eligibility pipeline
- Align all code paths to treat missing `isPharmacyEligible` as `false` (opt-in model)
- Add `$ifNull` backfill in the Looker webhook so new locations explicitly get `isPharmacyEligible: false`
- Include a one-time migration script to patch existing documents

## Root Cause
The `patientLocations` collection has two data sources: the Looker webhook (raw MongoDB driver) and the admin UI (Mongoose). The Mongoose schema defined `isPharmacyEligible` with `default: true`, but the Looker webhook never wrote this field when creating/updating locations.

This caused a split-brain problem:
- **Locations UI** showed "Enabled" â€” Mongoose applied the schema default on read, and the service layer's `normalizePharmacyEligible` function used `?? true`
- **Orders Tracker pipeline** showed "No" â€” the aggregation used `$ifNull: ['$site.isPharmacyEligible', false]`, reading directly from MongoDB without Mongoose defaults

3 locations in the database were affected: Deerwood (FL), Cary (NC), and Durham (NC).

## What Changed and Why

### 1. Mongoose Schema Default (`Location.ts`)
Changed `isPharmacyEligible: { type: Boolean, default: true }` to `default: false`. Locations should be opt-in for pharmacy eligibility, not opt-out.

### 2. Looker Webhook Backfill (`webhookUpsertLocations.ts`)
Added `isPharmacyEligible: { $ifNull: ['$isPharmacyEligible', false] }` to the `$set` block, following the existing `location_type` backfill pattern. This ensures:
- New locations from WeInfuse get `isPharmacyEligible: false` explicitly
- Existing locations configured via admin UI preserve their value

### 3. Service Layer Normalization (`locationDbService.ts`)
The `normalizePharmacyEligible` function was converting `undefined` â†’ `true` for all locations fetched via the raw MongoDB driver. Changed to `?? false` to match the corrected default.

### 4. UI Fallbacks (active + deprecated pages)
All `?? true` fallbacks in UI components and form defaults changed to `?? false`. This covers:
- Active: `/users-locations` page (via `locationDbService.ts` + `AddEditLocationModal.tsx`)
- Deprecated: `/admin/locations` pages (direct `?? true` in JSX)

### 5. Migration Script
`scripts/migrate-locations-pharmacy-eligible.ts` patches existing documents where `isPharmacyEligible` doesn't exist, setting it to `false`. Run after deployment: `npx ts-node scripts/migrate-locations-pharmacy-eligible.ts`

**No changes to `ordersTracker.ts`** â€” the pipeline already uses `$ifNull: ['$site.isPharmacyEligible', false]`, which was correct all along.
