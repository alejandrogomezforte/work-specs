# [MLID-2439] fix(notifications): disable backfill lookback window

## Summary

- PO direction (2026-05-27): drop the 30-day document-age filter that `notifyOrderOfRecentDocuments` applies when surfacing the patient's existing documents to a new order. The filter caused recently-created orders for patients with older documents to receive zero notifications and zero auto-attach audit rows — assignees saw no badge / "New" chip on the order's Documents tab even though the patient had several documents ready to be worked.
- Implementation uses a sentinel pattern: `BACKFILL_LOOKBACK_DAYS: number = -1` disables the filter, any positive value re-enables it with the original behavior. The date-filter code path is preserved (not deleted) so PO can re-enable by flipping the constant back to a positive number — no further code change required.
- One unrelated piggyback commit on this branch: a single-line `turbo.json` change switching the dev UI from `tui` to `stream` so that selecting text in the dev-server terminal no longer kills the running app. PE pre-approved bundling this onto a random task branch (no dedicated Jira ticket).

## Root Cause

`notifyOrderOfRecentDocuments` (the helper called from `POST /api/orders/new` at order creation AND from the `handleOrderReassignment` initial-assignment backfill path) computed a `since` cutoff as `now - 30 days` and passed it to `getDocumentsByPatientId({ patientId, since })`. The Mongo query then applied a `createdAt >= since` filter, dropping any document older than 30 days from the candidate set. Documents that aged out of the window were never surfaced again — even when a new order arrived for a patient whose intake-time uploads had crossed the 30-day mark.

The 30-day value was an inline constant introduced as a reasonable default during the original MLID-2011 work. There is no operational reason for the cutoff itself — it was a guess, not a policy.

## What Changed and Why

### Sentinel pattern instead of removal

Two alternatives were considered:

- **Delete the lookback entirely** (remove the constant + math, always pass `undefined`). Smallest diff. Loses the ability to re-enable the filter without re-introducing the math by hand.
- **Promote to env var** (`process.env.BACKFILL_LOOKBACK_DAYS`). Configurable per environment, but the cost is meaningful: code change + `.env.local.example` + terraform updates on stage AND prod + ops needs to know about the new var. PO has not asked for per-env behavior — they made a global "disable" call. The env-var setup is over-engineering for a value that has changed once in the project's history.

The sentinel pattern lands in the middle: the value is one number to flip back to (or to tune), no infra coordination needed, and the date-filter code is fully exercised by the codebase the moment someone sets a positive value again.

### TypeScript widening

A subtle pitfall on the sentinel form: declaring the constant as `const BACKFILL_LOOKBACK_DAYS = -1` would narrow its type to the literal `-1`, which then makes TypeScript treat `BACKFILL_LOOKBACK_DAYS > 0` as a statically-false condition. ESLint / TS might flag the kept `since` computation as unreachable, and the linter on the `: Date | undefined` ternary's positive branch would warn. Annotating the type explicitly as `: number` widens it back to the general numeric type, keeps the comparison meaningful at the type level, and preserves the lint-clean state when the constant is flipped.

### Behavior table

| Scenario | Before this fix | After this fix |
|---|---|---|
| New order created, patient has docs uploaded in the last 30 days | Docs propagated, notifications written | Same (unchanged) |
| New order created, patient has docs older than 30 days | Docs filtered out, **zero** notifications | All non-Order docs propagated, notifications written |
| Initial-assignment backfill (`handleOrderReassignment` empty→populated path), patient has any docs | Same age cutoff applied | All non-Order docs propagated |
| Reassignment between two users | Unchanged (retarget path doesn't go through propagator) | Unchanged |
