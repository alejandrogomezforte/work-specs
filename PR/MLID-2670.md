# [MLID-2670] fix(prior-auth): match drug names ignoring separators in order-scoped Auth Review

## Summary
- The Order Details → Prior Auth (Auth Review) tab was hiding legitimate intake documents when the catalogue drug name and the OCR-extracted drug name differed only in separator characters (for example a hyphen versus a space).
- The drug-name comparison now ignores separators and is case-insensitive, so the catalogue name (the source of truth) matches the OCR text regardless of how it was punctuated.
- Fix is internal to the prior-auth query builder; both order-scoped routes benefit with no route changes.

## Root Cause
The order-scoped prior-auth list builds its drug match in `buildContainsClause`, which performed a raw case-insensitive substring test (`$indexOfCP` over a `$toLower` field). The order term was only lowercased, with separators left intact.

For the reported case the catalogue `lidrugs.name` is `Ocrevus-Zunovo` (hyphen) while the OCR `brandDrugName.value` is `Ocrevus Zunovo` (space). `"ocrevus-zunovo"` is never a substring of `"ocrevus zunovo"`, so the brand fuzzy clause returned false. The other drug branches could not rescue it (this drug has no generic name in the catalogue, the order has no generic name, and the OCR brand value is present, so no passthrough applied). Every other gate — patient, status, document type, and the decision-date-versus-order-creation-date rule — passed, so the document was silently excluded.

## What Changed and Why

### Separator-agnostic, case-insensitive matching
The catalogue drug name is treated as the source of truth and turned into a regular expression matched against the OCR field with `$regexMatch` (`options: 'i'`). The name is split on `[^a-z0-9]+` into words and rejoined with the separator class `[^a-z0-9]*` (zero or more), so any run of separators — or none — is allowed between words.

Before:
```
$indexOfCP over $toLower(brandDrugName)  vs  "ocrevus-zunovo"
  -> "ocrevus-zunovo" not found in "ocrevus zunovo" -> no match
```

After:
```
$regexMatch( brandDrugName , /ocrevus[^a-z0-9]*zunovo/i )
  -> matches "Ocrevus Zunovo", "Ocrevus-Zunovo", "OcrevusZunovo"
```

### Design decisions
- **Build the regex from the catalogue name rather than normalizing both strings.** MongoDB aggregation has no regex-replace operator (`$replaceAll` is literal-only and cannot collapse arbitrary separator runs), so normalizing both sides to a canonical form is awkward. Building the pattern from the trusted catalogue name expresses the same intent directly.
- **Allow glued words (`[^a-z0-9]*`, zero or more).** OCR may drop a separator entirely (for example catalogue `Vitamin B-12` versus OCR `Vitamin B12`), so the separator between words is optional. The catalogue words must still appear, in order, so unrelated drugs do not match.
- **Contains semantics preserved.** `$regexMatch` is unanchored, so OCR values with extra surrounding text (a dosage suffix or a label prefix) still match, matching the previous behavior.

### Safety
- The `$type`-string guard is kept, so a non-string OCR field (for example a malformed array) is coerced to an empty string instead of throwing inside `$regexMatch`.
- After splitting on `[^a-z0-9]+`, every word contains only `[a-z0-9]`, so the built pattern carries no regular-expression metacharacters — it is injection-safe. The order term is resolved server-side from the catalogue, not from request input.
- Fuzzy clauses are only emitted when `normalizeText(term)` is non-empty, preventing an empty pattern that would otherwise match every document.

### No false-positive loosening
The match still requires the catalogue's words to appear in order within the OCR field; only the separators between those words are relaxed. It does not make unrelated drugs match.
