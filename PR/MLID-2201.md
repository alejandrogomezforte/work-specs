# [MLID-2201] feat(file-viewer): add fullscreen PDF toggle and sunset legacy FileViewer wrapper

## Summary

- Adds a fullscreen toggle button to the embedded PDF viewer's toolbar so users can read documents at full size instead of being constrained to a column-width panel.
- Sunsets the legacy `FileViewer` wrapper (`apps/web/components/UI/FileViewer/index.tsx`) and migrates all four consumers (`patientDocumentRecords`, `patientDocuments`, `intakes/[id]/page`, `PatientPriorAuth`) to a new `EmbeddedFileViewer` **dialog mode**. This eliminates the Dialog-inside-Dialog stacking that surfaced on document-records and document-list pages after the fullscreen toggle landed.
- One coherent `@repo/ui`-based Dialog stack across every PDF-viewing flow in the app.

**Branch:** `feature/MLID-2201-fullscreen-pdf-viewer` → `develop`

**Jira:** [MLID-2201](https://localinfusion.atlassian.net/browse/MLID-2201)

---

## Root Cause (the bundled bug fix)

After the fullscreen toggle from Part 1 landed, four routes still wrapped `EmbeddedFileViewer` in the legacy react-modal-based `FileViewer` wrapper. Clicking the toolbar's new fullscreen icon inside that legacy Dialog mounted a *second* `@repo/ui` Dialog on top of it via its own portal — two close buttons, two title bars, two scroll surfaces. Functional but visually broken.

Rather than coexist with the stack, this PR replaces the legacy wrapper entirely. `EmbeddedFileViewer` gains a first-class dialog mode (`open`/`onClose` props); callers that need a popup mount it directly with no intermediate wrapper. Inside dialog mode the inline fullscreen button is suppressed — you're already in the popup, so no popup-inside-popup.

---

## Design Pattern — callback prop + lifted state (Part 1)

The fullscreen toggle uses the same dual-instance pattern the PoC validated:

```
EmbeddedFileViewer  (owns isFullscreen state + @repo/ui Dialog)
├── ReactPdfRenderer  (inline)        ← onRequestFullscreen passed → button visible
└── Dialog (when isFullscreen)
    └── ReactPdfRenderer  (popped-out)   ← no callback → button hidden
```

`ReactPdfRenderer` stays focused on rendering PDFs — it never knows about Dialogs. Same `objectUrl` (lifted state) is reused across both instances, so the blob is fetched once and shared. The two renderer instances mount independently, each seeding its own internal zoom state from `initialZoom` on first mount — which is why the inline default stays at 50% while the popped-out and dialog-mode defaults are 100%.

## What Changed and Why

### 1. Toolbar fullscreen button (`ReactPdfRenderer.tsx`)

The renderer gets one new optional prop: `onRequestFullscreen?: () => void`. The button renders only when this prop is **defined**, never based on its truthiness. Two consequences:

- Callers in inline mode pass a setter callback — button visible, click fires the callback.
- The dialog-mode wrapper instance and the popped-out fullscreen instance both omit the prop — button absent automatically. No `showFullscreenButton` boolean, no recursive button.

Reuses the existing `styles.toolbarButton` class. No new CSS.

### 2. Sibling fullscreen Dialog (`EmbeddedFileViewer.tsx`, inline path)

When `onRequestFullscreen` fires, `isFullscreen` flips to true and an `@repo/ui` Dialog mounts a *second* `ReactPdfRenderer` with the same `objectUrl`. Both renderer instances are wrapped in the same `<ErrorBoundary>` pattern the inline path already used.

**The Dialog uses `@repo/ui`, not the legacy `@/components/UI/Dialog`.** This was an explicit constraint from the Jira ticket: standardize on the design system. The legacy primitive is still used elsewhere (e.g. `admin/JobsManagement`) — out of scope to fully retire here.

### 3. Dialog mode (`EmbeddedFileViewer.tsx`, second feature)

Two new optional props:

```ts
open?: boolean;
onClose?: () => void;
```

When `open` is defined, `EmbeddedFileViewer` wraps its body in an `@repo/ui` Dialog directly. The inner `ReactPdfRenderer` receives no `onRequestFullscreen` callback (so its toolbar button is automatically hidden — no popup-inside-popup), and the sibling fullscreen Dialog is skipped entirely. The blob fetch is gated on `open !== false` so a closed dialog mounted with an empty `fileUrl` (a pattern used by `patientDocumentRecords` while the user hasn't selected a document) never fires a bogus request.

When `open` is undefined (default), behavior is unchanged: inline rendering + toolbar fullscreen button + sibling popup Dialog.

### 4. Initial zoom defaults

```ts
const effectiveInitialZoom = pdfInitialZoom ?? (isDialogMode ? 100 : 50);
const fullscreenInitialZoom = pdfInitialZoom ?? 100;
```

- Inline view defaults to 50% — fits in a narrow column.
- Dialog mode (parent passes `open` prop) and the popped-out fullscreen Dialog both default to 100% — they fill most of the screen, so a higher zoom is actually readable.
- Caller's `pdfInitialZoom` always wins for both.
- Two separate named values rather than a unified flag because the two `ReactPdfRenderer` instances mount independently and each seeds its own internal zoom state from `initialZoom` on first mount. A single unified value that flipped to 100 when the popup opened would also flip the already-mounted inline renderer's prop (harmless in production since `ReactPdfRenderer` only reads `initialZoom` on mount, but harder to reason about and harder to test).

### 5. Migration of 4 consumers + removal of `FileViewer` wrapper

Each of the four routes that used the legacy `FileViewer` wrapper became a direct `EmbeddedFileViewer` call in dialog mode:

```tsx
<EmbeddedFileViewer
  open={fileViewerOpen}
  onClose={closeFileViewer}
  fileUrl={selectedFile?.url || ''}
  fileName={selectedFile?.name || ''}
  fileType={...}
/>
```

The legacy `FileViewer` wrapper supported an `onOpen` prop that rendered a header button — used only by `PatientPriorAuth` to navigate to the full prior-auth review page. **Dropped during migration**: clicking the table row (instead of the small preview icon) already navigates to the same destination, which embeds `EmbeddedFileViewer` inline. The popup's "Open" button was redundant.

After all four migrations, `index.tsx` and `index.test.tsx` had zero remaining consumers. Replaced `index.tsx` with a small `index.ts` barrel that re-exports `EmbeddedFileViewer` and `ContentLibraryFileViewer`. `apps/web/components/UI/index.ts` still does `export * from './FileViewer'` so app-wide imports continue to resolve.

### 6. Test infrastructure — `LiThemeProvider`

The `@repo/ui` Dialog's styled component reads from Emotion theme tokens (`t.radii.xl`, `t.elevation.lg`, etc.) on every render, so mounting outside a `LiThemeProvider` throws `Cannot read properties of undefined (reading 'radii')`. Both `EmbeddedFileViewer` test files now alias `render` from `@testing-library/react` to auto-wrap children in `LiThemeProvider` via the `wrapper` option — no changes to the 45+ existing render call sites. Matches the precedent in `DrugSwitchContextNoteDialog.test.tsx`.

## Alternatives Considered

| Alternative | Why not |
|---|---|
| Make `ReactPdfRenderer` itself open the Dialog (recursion) | Would force the renderer to know about Dialogs, and recursion creates lifecycle ambiguity between the inline and popped-out instances. Callback + lifted state keeps each component's responsibility clean. |
| Keep the legacy `FileViewer` wrapper, just unify the Dialogs underneath | Leaves a dead abstraction in place — every new consumer would have to choose between two Dialog patterns. Sunset is simpler long-term. |
| Build a separate `FileViewerDialog` component wrapping `EmbeddedFileViewer` | Two ways to render the same thing. Dialog mode on the existing component is one prop interface, one component, one mental model. |
| Migrate `PatientPriorAuth`'s "Open" button into the new Dialog via `@repo/ui` Dialog's `actions` slot | Confirmed with PO during planning: row-click already reaches the same destination, so the button is redundant. Deleted instead. |
| Unified `effectiveInitialZoom = pdfInitialZoom ?? (isFullscreenContext ? 100 : 50)` based on `isDialogMode \|\| isFullscreen` | Worked in production (the renderer only reads `initialZoom` on mount) but coupled the two instances through a single prop that changed value mid-lifetime — surprising in tests, brittle to reason about. Two named values per instance is more explicit. |
