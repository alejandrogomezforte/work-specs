# [MLID-2159] fix(orders-tracker): return documents on order details page

## Summary
- The Documents tab in the Order Tracker order details page (`/orders-tracker/[category]/[orderId]/documents`) always rendered an empty list, even for orders whose patient had real documents in the `documents` collection.
- Root cause: `getPatientMongoId()` in the details API route checked `typeof _id === 'string'` but, after the aggregation's patient `$lookup`, `_id` arrives as a BSON `ObjectId` instance — so the helper returned `null`, the documents fetch was skipped, and the API unconditionally returned `documents: []`.
- The fix coerces an `ObjectId` `_id` to its hex string via `toHexString()` before handing it to `getDocumentsByPatientId`, while preserving the existing string branch for tests and any legacy code paths.

## Bug: Documents Tab Always Empty

### Problem

`GET /api/orders/details` hydrates the order via `getOrderById()`, which runs the `getOrdersAggregate()` pipeline. For `new` and `maintenance` categories that pipeline does a `$lookup` into the `patients` collection, replacing the raw `patient` string with a populated patient subdocument. After the lookup the subdocument's `_id` is a BSON `ObjectId` instance, **not** a string.

The helper that extracted the patient id for the documents query was:

```typescript
function getPatientMongoId(patient: unknown): string | null {
  if (!isObject(patient)) {
    return null;
  }

  const rawId = patient._id;
  return typeof rawId === 'string' ? rawId : null;
}
```

`typeof ObjectId === 'object'`, so the helper returned `null` for every real order. The route's fetch then short-circuited:

```typescript
const [documents, weInfuseOrder] = await Promise.all([
  patientMongoId
    ? getDocumentsByPatientId(patientMongoId, { limit: 100 })
    : Promise.resolve([]),
  // ...
]);
```

And the API responded with `documents: []`.

### Why It Went Unnoticed

Most existing documents in the DB are `jotform-webhook` intake-form writes whose `patientId` field is the patient's numeric `we_infuse_id_IV` (e.g. `"1078388"`), not the Mongo `_id`. Those rows would have been invisible to the Documents tab even with a working helper — because `getDocumentsByPatientId` queries by the `_id` form. The empty Documents tab therefore "looked right" for many orders. The bug only surfaces on patients who have documents keyed by Mongo `_id` (e.g. the QA `Test Request` patient), which is the shape MLID-2011 test data seeding exposes.

### Fix

Keep the existing `string` branch untouched (safe for tests and any code path that passes a pre-stringified id), and add a branch that calls `toHexString()` when `_id` is an `ObjectId`:

```typescript
function getPatientMongoId(patient: unknown): string | null {
  if (!isObject(patient)) {
    return null;
  }

  const rawId = patient._id;
  if (typeof rawId === 'string') {
    return rawId;
  }
  // After $lookup, `_id` comes back as a BSON ObjectId instance, not a string.
  const hex = (rawId as { toHexString?: () => string } | null)?.toHexString?.();
  return typeof hex === 'string' ? hex : null;
}
```

### Why Existing Tests Still Pass

`apps/web/app/api/orders/details/route.test.ts` seeds the mocked order with `patient: { _id: 'patient-1', ... }` — a plain string. The fix takes the `typeof rawId === 'string'` branch first, so behavior for that test is unchanged. No test updates required.
