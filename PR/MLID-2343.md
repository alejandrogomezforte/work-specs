# [MLID-2343] feat(orders/documents): in-page PDF viewer with paginated footer

## Summary

- Replaces the "Document Name" external link in the Orders Documents tab (`/orders-tracker/[category]/[orderId]/documents`) with an in-page PDF viewer dialog. Clicking a document name now opens `EmbeddedFileViewer` instead of `target="_blank"` to a new browser tab.
- Adds a paginated footer to the dialog (prev/next arrows at the edges, "Document N of total" + category chip + "Received {date}" centered) so users can step through the order's documents without closing and reopening the viewer.
- Sunsets the `/api/orders/details/document-view` 302-redirect endpoint. The new viewer expects an Azure blob URL directly (matching how `patientDocuments` already works), so the document API now returns `document.blobPath` as `viewUrl` and the redirect route's only caller is gone.
- Generalizes `EmbeddedFileViewer` with one optional `dialogActions` prop forwarded to the dialog-mode `Dialog`'s `actions` slot. The viewer stays domain-agnostic — all navigation state and footer content live in the page.

**Branch:** `feature/MLID-2343-documents-tab-pdf-viewer` → `develop`

**Jira:** [MLID-2343](https://localinfusion.atlassian.net/browse/MLID-2343)

## Architecture

`EmbeddedFileViewer` already renders as an `@repo/ui` `Dialog` when `open`/`onClose` are passed (introduced in MLID-2201). This task adds one optional prop, `dialogActions: React.ReactNode`, forwarded as the `actions` slot of that dialog-mode `Dialog`. The inline-mode internal fullscreen `Dialog` is intentionally unchanged — it serves generic consumers across other routes and must remain domain-agnostic.

The page owns all navigation state via a single `openIndex: number | null` value. Swapping `fileUrl` and `fileName` on the viewer triggers its existing `useEffect([fileUrl, ...])` re-fetch — no new logic inside the viewer.

`DocumentViewerFooter` is a pure presentational component co-located with the page. It receives `currentIndex`, `total`, `category`, `receivedDate`, `onPrev`, and `onNext` as props — no internal state, no domain data imports. Its layout is a 3-column flex row: prev IconButton pinned left, a `Layout.Item grow="1"` middle that centers the count + chip + date via `justify="center"`, next IconButton pinned right. Outer Layout uses `maxWidth="full"` so the footer fills 100% of the dialog width.

## Root Cause (the bundled API refactor)

After wiring the page to `EmbeddedFileViewer`, the dialog opened but the PDF failed to load with a 400 from `/api/files/stream` ("Invalid blob URL format").

**Why:** the page was passing `viewUrl` (e.g. `/api/orders/details/document-view?documentId=…&orderId=…&category=…`) as the viewer's `fileUrl`. The viewer pipes that through `/api/files/stream?url=<encoded fileUrl>`, which expects the URL to match the Azure blob pattern `https://{account}.blob.core.windows.net/{container}/{path}`. A relative `/api/...` path doesn't match → 400.

The legacy `target="_blank"` flow worked because the browser navigated directly to `/api/orders/details/document-view`, which 302-redirects to the actual blob URL. The viewer's stream-based flow has no equivalent indirection.

**Fix:** change `viewUrl` to be the Azure blob URL directly (`document.blobPath`), matching the existing `patientDocuments` pattern (where the same `EmbeddedFileViewer` consumes `selectedFile.url` as a blob URL). The `document-view` route had no other callers, so it's removed in the same commit.

Authorization is preserved via `/api/files/stream`'s session auth + HIPAA logging.

## What Changed and Why

### `EmbeddedFileViewer.dialogActions` prop

One new optional prop. Forwarded only in the dialog-mode branch:

```tsx
if (isDialogMode) {
  return (
    <Dialog
      open={open === true}
      onClose={onClose ?? (() => {})}
      title={fileName}
      size="lg"
      maxWidth="xl"
      fixedHeight={96}
      actions={dialogActions}  // new
    >
      {open ? inlineContent : null}
    </Dialog>
  );
}
```

The inline-mode internal fullscreen Dialog (used by other consumer routes via the renderer's `onRequestFullscreen` callback) is intentionally not touched — adding domain-specific footer content there would leak into routes that have no concept of order documents. The regression test asserts `dialogActions` does **not** appear in inline mode.

### `viewUrl` semantic change + sunset

Before:
```ts
viewUrl: `/api/orders/details/document-view?documentId=${...}&orderId=${...}&category=${...}`
```

After:
```ts
viewUrl: document.blobPath ?? ''
```

The `document-view` route's only purpose was to 302-redirect the (documentId, orderId, category) triplet to `document.blobPath` while doing a per-document authz check (verifying the doc's patient matches the order's patient). With `target="_blank"` gone, no caller follows that redirect anymore — the new viewer flow goes directly to the blob via `/api/files/stream`, which has its own session auth + HIPAA logging.

Empty string fallback for missing `blobPath` is intentional: `EmbeddedFileViewer` short-circuits with `if (!fileUrl) return` and shows its existing "No file content available" overlay. No worse than the old flow, which would 404 in the new tab.

### Document Name → button

```tsx
renderCell: (params) => {
  const docIndex = data.documents.findIndex((d) => d.id === params.row.id);
  return (
    <Link
      component="button"
      onClick={() => setOpenIndex(docIndex)}
      sx={{ color: colors.status.success, textDecoration: 'underline' }}
    >
      {params.value as string}
    </Link>
  );
},
```

Uses `@mui/material/Link` (not `@repo/ui` `Link`) because the design-system `Link` only supports `'primary' | 'secondary' | 'inherit'` colors, and the spec calls for `colors.status.success` (`#1d8474`). `feedback_use_repo_ui.md` explicitly allows the MUI fallback when @repo/ui can't express the requirement.

### Navigation state

A single `openIndex: number | null` in the page component:

```tsx
const [openIndex, setOpenIndex] = useState<number | null>(null);
```

Conditional render — viewer mounts only when a document is open:

```tsx
{openIndex !== null && (
  <EmbeddedFileViewer
    fileUrl={data.documents[openIndex].viewUrl}
    fileName={data.documents[openIndex].name}
    fileType="pdf"
    open={true}
    onClose={() => setOpenIndex(null)}
    dialogActions={
      <DocumentViewerFooter
        currentIndex={openIndex}
        total={data.documents.length}
        category={data.documents[openIndex].category}
        receivedDate={data.documents[openIndex].receivedDate}
        onPrev={() =>
          setOpenIndex((current) => Math.max(0, (current ?? 0) - 1))
        }
        onNext={() =>
          setOpenIndex((current) =>
            Math.min(data.documents.length - 1, (current ?? 0) + 1)
          )
        }
      />
    }
  />
)}
```

Navigation is clamped at both ends. Closing the viewer resets `openIndex` to `null`, so reopening any row starts fresh.

### Footer layout

3-column flex that fills 100% of the dialog actions width via `maxWidth="full"`. Middle column is `Layout.Item grow="1"` with an inner `Layout justify="center"`, so the count + chip + date group is **true-centered** regardless of the icon button widths:

```tsx
<Layout direction="row" align="center" maxWidth="full">
  <IconButton aria-label="Previous document" ...><ChevronLeftIcon /></IconButton>
  <Layout.Item grow="1">
    <Layout direction="row" align="center" justify="center" gap={2}>
      <Typography variant="bodySmall">Document {currentIndex + 1} of {total}</Typography>
      <Chip label={category} size="sm" variant="outlined" />
      <Typography variant="bodySmall">Received {formatReceived(receivedDate)}</Typography>
    </Layout>
  </Layout.Item>
  <IconButton aria-label="Next document" ...><ChevronRightIcon /></IconButton>
</Layout>
```

### `formatReceived` extraction

Pulled out of `page.tsx` into a sibling `formatReceived.ts` so the footer can import it without circular dependencies. No behavior change. Kept co-located with the documents directory rather than promoted to a global utility — its local-timezone formatting semantics are specific to this surface.

### Navigation order

Follows the `data.documents` array order returned by the API, **not** the DataGrid's current sort order. PO confirmed this in the Jira spec — sort-order navigation is a separate feature if ever requested.

## Notes / Open Questions

- **Loading state between documents** — each prev/next click triggers a fresh `axios` fetch inside `EmbeddedFileViewer` (the viewer's `useEffect([fileUrl, ...])` fires on `fileUrl` change). A brief loading overlay is expected between documents. Documented behavior, not a bug.
- **`ReactPdfRenderer` toolbar** — the viewer always renders its own zoom/rotation toolbar. This toolbar appears inside the dialog alongside the new footer. If the design team wants the toolbar suppressed in dialog mode, that's a separate task.
- **Green link color** — using `colors.status.success` (`#1d8474`) which is the standard success green in the design system with WCAG contrast verified. If the playground's reference is a different shade, `colors.status.successDark` (`#065f46`) is the next-darker alternative.
