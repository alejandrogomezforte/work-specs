# [MLID-2752] fix(orders-financial-review): apply admin cost before drug cost to deductible and OOP

## Summary

- The orders-tracker financial review was processing the Drug leg before the Admin leg when computing patient responsibility against the remaining deductible and OOP
- This produced different per-leg results from the patient section, which was corrected in MLID-2056 (PR #1571) to run Admin first
- This fix aligns the orders-tracker calculation engine to the same Admin-first order, resolving the discrepancy QA reported in MLID-2752

## Root Cause

Both calculation engines share the same per-leg formula, but the order in which the two legs consume the shared `remainingDeductible` and `remainingOOP` state diverged.

Alberto's MLID-2056 fix (patient side, May 2026) flipped the patient-section handler from Drug-first to Admin-first. The orders-tracker service (`runOrdersFinancialCalculation`) was introduced after that fix and independently reimplemented the algorithm with Drug-first ordering, inadvertently recreating the pre-MLID-2056 behavior.

Because the first leg burns deductible/OOP credits before the second leg runs, a different processing order produces different per-leg `drugResp`/`adminResp` figures whenever the deductible or OOP cap is not already exhausted. Example with the QA test case ($50 deductible remaining, 30% coinsurance):

| | Drug-first (old) | Admin-first (fixed) |
|---|---|---|
| Admin leg: deductible portion | $0 | $50 |
| Admin leg: coinsurance on remainder | $30 (30% × $100) | $30 (30% × $100) |
| **Admin responsibility** | **$45** | **$80** |
| Drug leg: coinsurance (covered by assistance) | $0 | $0 |
| **Drug responsibility** | **$0** | **$0** |
| **Total** | **$45** | **$80** |

QA expected $80 total; the old code returned $45.

## What Changed and Why

### `runOrdersFinancialCalculation` — leg order flipped

**Before:**
```ts
const drugRun = processOrdersFinancialCalculationPart('Drug', drugCost, ...initialState);
let adminRun;
if (drugRun.newState.remainingOOP === 0) {
  adminRun = { bullets: ['Skipped ... during Drug.'], ... };
} else {
  adminRun = processOrdersFinancialCalculationPart('Admin', adminCost, ...drugRun.newState);
}
const drugResp  = drugRun.patientPays;
const adminResp = adminRun.patientPays;
summary: ['Patient Drug Responsibility: ...', 'Patient Admin Responsibility: ...', ...]
```

**After:**
```ts
const adminRun = processOrdersFinancialCalculationPart('Admin', adminCost, ...initialState);
let drugRun;
if (adminRun.newState.remainingOOP === 0) {
  drugRun = { bullets: ['Skipped ... during Admin.'], ... };
} else {
  drugRun = processOrdersFinancialCalculationPart('Drug', drugCost, ...adminRun.newState);
}
const adminResp = adminRun.patientPays;
const drugResp  = drugRun.patientPays;
summary: ['Patient Admin Responsibility: ...', 'Patient Drug Responsibility: ...', ...]
```

The `processOrdersFinancialCalculationPart` function itself is unchanged — only the call order and state threading changed.

### Test update — stale OOP-skip test replaced

The test `"should skip the Admin leg entirely when remainingOOP hits zero during Drug"` had:
- `drugCost: 500, adminCost: 200` — Drug exhausts OOP
- `expect(result.drugResp).toBe(50), expect(breakdown.admin).toMatch(/Skipped/)`

After the fix Drug no longer runs first, so Admin (with `adminCost: 200`) now consumes the OOP. The test was replaced with `"should skip the Drug leg when remainingOOP hits zero during Admin"`, swapping cost fields and assertions. No scenario coverage was lost.
