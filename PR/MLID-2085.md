# [MLID-2085] fix(documents): allow current order assignee to see and act on stale notifications

## Summary

QA found that a newly assigned Infusion Guide cannot see the *New* chip, the per-row *Mark as Read* button, or the *Mark all as Read* bulk button on the Documents tab — and that even when the chips were forced visible, clicking *Mark as Read* returned `404` and never persisted. Root cause: document notifications snapshot `targetUserId` at write-time and never update when the order is reassigned, and every gate (UI predicate + server authz) was strict-equal to that stale snapshot.

This PR relaxes the gate in three places — two on the client, two on the server — so the *current* order assignee is authorized regardless of who the notification's `targetUserId` names. The order id used by the server fallback is read from the notification's `entities`, never from client input, so the API contract is unchanged and existence-probing protection is preserved.

The long-term data-side fix (cascade `targetUserId` on reassignment) is tracked in **[MLID-2391](https://localinfusion.atlassian.net/browse/MLID-2391)**. The OR-branches added here are kept after that ships, as belt-and-suspenders against assignment races.

**Branch:** `feature/MLID-2085-documents-tab-relaxed-guard` → `develop`

**Jira:** [MLID-2085](https://localinfusion.atlassian.net/browse/MLID-2085)

## Root Cause

Every document notification is created with `targetUserId = order.assignedTo` at the moment the document arrives (`apps/web/services/notifications/documentNotification.ts:85-88`). When the order is later reassigned, the notification document is not touched — `targetUserId` keeps pointing at the prior assignee.

Three gates depended on a strict `targetUserId === userId` match:

| Gate | Surface | Effect when stale |
|---|---|---|
| `isDocumentUnreadForUser` | *New* chip + per-row *Mark as Read* button | controls hidden |
| `useDocumentsTabNotifications.unreadNotificationIdsForCurrentUser` | drives `hasUnreadForCurrentUser` (bulk button enable state) and the bulk PATCH payload | bulk button disabled, bulk payload empty |
| `markUserNotificationAsRead` / `bulkMarkUserNotificationsAsRead` | server-side authz on the PATCH endpoints | endpoints silently no-op (`404`) |

The bug presented as "everything is broken" because the UI gate hid the controls; if the UI was forced past it, the server gate then silently rejected the click. Both layers needed the same OR-branch.

## Architecture

```
Notification (write-time snapshot)
   targetUserId  ←  order.assignedTo at the moment the document arrives
                    (NEVER updated; MLID-2391 will cascade)
        |
        v
─── UI gates ────────────────────────────────────────────────────────────
isDocumentUnreadForUser(summary, userId, orderAssignedTo):
   summary.targetUserId === userId  OR  orderAssignedTo === userId

useDocumentsTabNotifications(orderId, orderAssignedTo):
   unreadNotificationIdsForCurrentUser filter:
     n.targetUserId === userId  OR  orderAssignedTo === userId
   unreadCount stays GLOBAL (it's the tab badge)
─── Server gates ────────────────────────────────────────────────────────
markUserNotificationAsRead(id, userId):
   strict findOne({ _id, targetUserId })          ─┐
   on miss:                                        ├─ allowed if either succeeds
     fetch notification, find {entityType:'order'},│
     getOrderById('new') ?? getOrderById('maint'), │
     order.assignedTo === userId                  ─┘

bulkMark…(ids, userId):
   strict find({ _id: $in, targetUserId })  → strictIds
   for the rest:
     fetch notifications, derive unique orderIds,
     getOrderById per orderId (cached),
     keep ids whose order.assignedTo === userId
   write NotificationRead.insertMany(strictIds ∪ fallbackIds)
```

Key invariants that made the relaxation safe to ship without the data-side cascade:

- **Source of truth for the order id is the notification document.** The server reads `notification.entities` to locate the order id; there is no new request body field, so callers cannot forge it. The strict-only contract is preserved at the API surface.
- **404 on every authorization failure.** Routes still return `404` (not `403`) when the fallback also fails. Callers cannot distinguish "exists but not owned" from "does not exist", preserving existence-probing protection.
- **Strict path remains primary.** `getOrderById` is only called when the strict `findOne` / `find` misses. Strict-owned ids cost zero extra DB work.
- **Tab badge stays global.** `unreadCount` (the tab header badge) is intentionally global per the 2026-04-14 product decision and is *not* relaxed; only the per-user-action filter changes.

## What Changed and Why

### 1. Two UI predicates, one OR-branch

The plan originally identified `isDocumentUnreadForUser` as the only gate, but tracing through `page.tsx` showed *two* separate predicates contributing to the broken state. `isDocumentUnreadForUser` gates the chip and the per-row button (lines 136 and 168). `useDocumentsTabNotifications.unreadNotificationIdsForCurrentUser` is a different predicate that drives `hasUnreadForCurrentUser` (the bulk button's `disabled` prop) and the bulk PATCH payload. Both received the OR-branch.

`unreadCount` (which the tab header reads) is intentionally global — the badge counts every unread notification regardless of assignee, per the 2026-04-14 product decision. It is *not* relaxed.

### 2. Server fallback reads the order id from the notification

```ts
async function isCallerOrderAssignee(
  notification: { entities?: INotificationEntity[] | null },
  userId: string,
  orderAssigneeCache?: Map<string, string | null>
): Promise<boolean> {
  const orderEntity = notification.entities?.find(
    (entity) => entity.entityType === 'order'
  );
  if (!orderEntity) return false;

  const { entityId: orderId } = orderEntity;
  let assignedTo: string | null;
  if (orderAssigneeCache && orderAssigneeCache.has(orderId)) {
    assignedTo = orderAssigneeCache.get(orderId) ?? null;
  } else {
    const order =
      (await getOrderById(orderId, 'new')) ??
      (await getOrderById(orderId, 'maintenance'));
    assignedTo = order?.assignedTo == null ? null : String(order.assignedTo);
    orderAssigneeCache?.set(orderId, assignedTo);
  }

  return assignedTo !== null && assignedTo === String(userId);
}
```

The order id is *always* derived from `notification.entities` — never accepted as a request body field. A caller cannot trick the server into authorizing reads for notifications they aren't entitled to, because there is no body to lie about.

Lead orders are skipped (no Documents tab there). Worst-case extra cost: one additional Mongo lookup per click when the order is a maintenance order.

The bulk path passes a `Map<string, string | null>` cache so multiple notifications that point at the same order share a single `getOrderById` call; the dedup is asserted in `dedupes order lookups when multiple notifications point at the same order`.

### 3. Existence-probing protection preserved

The fallback succeeds only when (a) the notification exists, (b) it has an order entity, AND (c) the caller is the order's current `assignedTo`. A caller who is not the assignee of any order the notification belongs to gets the same `false` result they got before. Routes continue to return `404` on every authorization failure, so the response signature is uniform — clients cannot distinguish "wrong owner" from "does not exist".

### 4. ObjectId vs string normalization (post-deploy bug found in dev)

After the first round of changes the UI correctly showed the chip and button, but clicking *Mark as Read* still returned `404`. The dev-server log for the call site:

```
[INFO] Notification Service: Marking user notification as read {
  notificationId: '6a073016c3a15ed77b2e257a',
  userId: new ObjectId('695c25a530f7a75a8b1ed8c8')
}
PATCH /api/notifications/6a073016c3a15ed77b2e257a/read 404
```

`session.user.id` arrives as a Mongoose `ObjectId`, but `order.assignedTo` is stored as a hex string (the order pipeline does `$convert: { input: '$assignedTo', to: 'objectId' }` only for the join — the underlying field is a `String`). JavaScript `===` between an `ObjectId` and a string is always `false`, so the fallback authz silently denied every click.

The strict path escapes this because Mongoose auto-casts the `ObjectId` to a string when querying against the schema's `String`-typed `targetUserId` field; only our in-process equality needed manual normalization. Fix: `String(...)` both sides at the storage boundary, with an explicit `null` short-circuit so missing assignees stay denied. Three additional tests cover the ObjectId-as-userId, ObjectId-as-assignedTo, and bulk-path-with-ObjectId cases.

## Security Notes

- **No new request body fields.** The server derives the order id from the notification's `entities` array, not from any client input.
- **Routes return `404` on every authorization failure.** The fallback only changes *who* the server authorizes, not the response signature for unauthorized callers.
- **Session userId is sourced from `getServerSession(authOptions)`** — clients cannot forge it.
- **PHI handling unchanged.** Only `notificationId`, `orderId`, and `userId` strings flow through the new code paths.
- **Race-safe.** UI and server both consult the same source of truth (`order.assignedTo`); if a reassignment happens between the UI check and the server check, the server's later read wins.
