# [MLID-2194] fix(pharmacy-eligibility): add "Medicare Supplement" to auto-pass allow-list

## Summary

- PM identified a missing `payer_type_name` value in the pharmacy eligibility logic added by the MLID-1492 epic. Orders whose primary insurance has `payer_type_name = "Medicare Supplement"` should auto-pass the insurance check (like Medicare and Medicare Advantage), but the current regex only matched the latter two.
- One-line fix to the anchored regex inside `getEligibilityPipelineStages()` plus matching test updates. No type, schema, feature-flag, or UI changes.
- Behaviour for unrelated values (`Medicare Part D`, arbitrary suffixes, `Commercial`, `Medicaid`) is unchanged.

## Root Cause

The pharmacy eligibility `$switch` in the New Orders aggregation pipeline encodes the auto-pass insurance allow-list as a single anchored regex inside `branches[2].case.$and[2].$or[0]`:

```
^\s*medicare(\s+advantage)?\s*$
```

This matches `Medicare` and `Medicare Advantage` only. `Medicare Supplement` falls through to the second `$or` branch (Commercial / Medicaid), which additionally requires `_insurancePlan.isPharmacyEligible == true` — a flag that is generally not set on Medicare Supplement plans, producing an incorrect `"No"` outcome.

## What Changed and Why

### `ordersTracker.ts` — extend the Medicare auto-pass regex

Before (line 174):

```ts
regex: '^\\s*medicare(\\s+advantage)?\\s*$',
```

After:

```ts
regex: '^\\s*medicare(\\s+(advantage|supplement))?\\s*$',
```

The optional suffix group `(\s+(advantage|supplement))?` keeps bare `Medicare` matching while adding `Medicare Supplement` alongside `Medicare Advantage`. Anchoring (`^…$`) and case-insensitivity (`options: 'i'`) are unchanged, so the negative cases (`Medicare Part D`, `Medicare Whatever`) continue to fail as required.

**Alternatives considered:**

- A `$in` array on the raw payer_type_name string — rejected because the existing implementation is regex-based and case-insensitive with whitespace tolerance; switching the encoding for one new value would needlessly enlarge the diff.
- Two separate `$regexMatch` clauses — rejected because the single-regex form is already in place and a one-token additive change preserves the structure.

### `ordersTracker.test.ts` — flip one test, add one test

The existing `describe('pharmacyEligible Medicare regex anchoring', …)` block extracts the live regex from the pipeline stage and exercises it against representative inputs. Two changes:

1. The pre-existing assertion that documented the *old* (incorrect) behaviour was flipped:

   Before:

   ```ts
   it('should NOT match "Medicare Supplement"', async () => {
     const regex = await getMedicareRegex();
     expect(new RegExp(regex, 'i').test('Medicare Supplement')).toBe(false);
   });
   ```

   After:

   ```ts
   it('should match "Medicare Supplement"', async () => {
     const regex = await getMedicareRegex();
     expect(new RegExp(regex, 'i').test('Medicare Supplement')).toBe(true);
   });
   ```

2. A whitespace mirror test was added after the existing `Medicare Advantage` whitespace test, matching the established pattern:

   ```ts
   it('should match "Medicare Supplement" with leading/trailing whitespace', async () => {
     const regex = await getMedicareRegex();
     expect(new RegExp(regex, 'i').test(' Medicare Supplement ')).toBe(true);
   });
   ```

The other 6 tests in the describe block (anchoring, `Medicare`, `Medicare Advantage`, `Medicare Part D` negative, plus the two existing whitespace variants) are untouched and continue to pass — they form the regression net for this change.
