# [MLID-2742] feat(masked-input): in-house MaskInput in @repo/ui, replacing the @mona-health/react-input-mask fork

## Summary

- Builds a purpose-built, headless masked-input component (`MaskInput`) and caret hook (`useMaskedCaret`) in the `@repo/ui` design system, then swaps the two app-level wrappers to it and removes the third-party fork entirely.
- Resolves the PE request to stop production depending on the `@mona-health/react-input-mask` fork (adopted during the Next.js 16 / React 19 migration, MLID-364, because the original `react-input-mask` used the removed `ReactDOM.findDOMNode`).
- Delivered as three sub-tasks on this branch: **MLID-2768** (build the component), **MLID-2769** (swap the wrappers), **MLID-2770** (remove the dependency + shim). This is the single PR for all three.

## Background / Why

The app uses masking in a deliberately narrow way — only **3 digit-only masks** exist across the whole codebase:

- `+1 (999) 999-9999` (phone)
- `99/99/9999` (date)
- `999999` (6-digit code)

Only the `9` digit placeholder is used — no alphabetic, regex, or optional-section masks, and none of the fork's advanced features (`beforeMaskedStateChange`, custom `maskChar`). So rather than vendor the fork's ~1,900-line source, this delivers a small component we own, covering exactly that surface. The genuinely hard part is caret management (keeping the cursor correct while typing, deleting across fixed characters, pasting, and range-selecting); the caret algorithm was ported in trimmed form from the MIT-licensed fork, with attribution retained in the hook.

## What changed

### 1. `@repo/ui` — new `MaskInput` component + `useMaskedCaret` hook (MLID-2768)

- Headless, unstyled `forwardRef<HTMLInputElement>` that renders a bare `<input>` (no visual chrome), so it works both standalone and dropped into MUI's `inputComponent` slot.
- `useMaskedCaret` owns mask parsing, value formatting, and caret positioning (auto-advance past fixed characters, delete-across-fixed-character, paste, range-replacement) applied via `useLayoutEffect`.
- Unfilled slots render as `_` (e.g. an empty phone shows `+1 (___) ___-____`), matching the fork's default output so the swap stays behavior-identical.
- Ships with a Storybook story (all 3 masks) as the primary real-browser caret verification surface, plus unit tests.

### 2. Swap the wrappers to `MaskInput` (MLID-2769)

- `MuiPhoneField` — `PhoneMaskAdapter` now renders `MaskInput` instead of the fork; public `MuiPhoneFieldProps` and the `hasUserInput` logic are unchanged.
- `MaskedInput` — reworked from the fork's child-clone pattern to render `MaskInput` directly, keeping the shared `styles.input` class so downstream consumers look and behave the same.
- `StyledInput` — stops importing the fork's `InputState` type; now uses a local `StyledInputProps` (so the fork type shim can be deleted).
- No downstream consumer files were changed. A characterization test was added for `MaskedInput` (green against both the old fork impl and the new one).

### 3. Remove the fork dependency + type shim (MLID-2770)

- Uninstalled `@mona-health/react-input-mask` from `apps/web` (removed from `package.json` and the root lockfile).
- Deleted the ambient type shim `apps/web/types/mona-health-react-input-mask.d.ts`.
- Kept the MIT-license attribution comment in `useMaskedCaret.ts` (removing the package does not remove the attribution obligation for the ported algorithm).

### Incidental bug fix — AbbVie / Kedrion SFTP fields now actually disable

While tightening `StyledInput`'s type, a pre-existing latent bug surfaced: the AbbVie and Kedrion admin integration pages passed an invalid `isDisabled` prop to `StyledInput` (which never disabled the fields — it landed on the DOM as an unrecognized attribute). Fixed the 10 call sites (`isDisabled` → `disabled`), so the SFTP Host/Port/Username/Password/Remote-Path fields are now correctly disabled while a save/test request is in flight. Covered by new tests that render the real `StyledInput` and assert the disabled state.

## Follow-up (separate ticket, not in this PR)

Tightening `StyledInput`'s type surfaced ~19 additional pre-existing latent-bug sites across ~8 files unrelated to masking (the same invalid `isDisabled` on argenx/jotform/spruce/notion, and an unsupported custom `error` prop on AddEditLocationModal / FormItem).
