# [MLID-2684] fix(prior-auth): preview parsed document instead of whole fax

## Summary
- The Patient card Prior Auth tab and the Order Details Auth Review tab previewed the entire inbound fax instead of the split, parsed prior-auth document that the Documents tab already shows.
- Both surfaces render through the same shared transform, so the fix is concentrated there plus one batched lookup per list page â€” no schema change and no change to the review/edit workflow.
- New rule: if the intake has a `documents` row with `category === "Prior Authorization"`, preview that row's `blobPath`; otherwise preview the whole intake PDF. Both open on page 1.

## Root Cause
`transformIntakeToPriorAuthLetter` (`apps/web/services/priorAuth/transforms.ts`) always built the preview `documentUrl` from `intake.documents[0].pdfDocument`, which is the whole consolidated fax. The Documents tab instead reads the `documents` collection, whose prior-auth row `blobPath` points at the split file produced during intake processing â€” which is why only that tab showed the parsed document. Because both the Patient card and the Order Details Auth Review reuse the shared transform, they inherited the same wrong preview source.

Separately, the old `documentPage` value was derived by matching `documentDefinitions[].type === 'prior-authorization'`. Production writes that field as `'Insurance Authorization'`, so the page jump was already silently non-functional against real intakes (masked only by test fixtures). This change removes that dead path.

## What Changed and Why

### 1. Preview source moves from the whole fax to the parsed document
The core of the fix. `transformIntakeToPriorAuthLetter` now receives the intake's `documents` rows and resolves the preview URL through `selectPriorAuthSplitDocument`:

Before:
```ts
const pdfDocument = intake.documents?.[0]?.pdfDocument; // the whole fax
const documentUrl = pdfDocument || undefined;
```

After:
```ts
const splitBlobPath = selectPriorAuthSplitDocument(documents);
const documentUrl =
  splitBlobPath || intake.documents?.[0]?.pdfDocument || undefined;
```

The `documents` argument is **required** (no default) so every call site must explicitly decide what to pass, preventing a caller from silently reverting to the whole-fax fallback by omission.

### 2. `documentPage` removed â€” preview always opens on page 1
The split prior-auth document contains only the parsed pages (page 1 is correct), and the whole-intake fallback opens from page 1 as well. So the transform stops setting `documentPage` and the `documentDefinitions` page-extraction block is deleted. `ReactPdfRenderer` defaults `pageNumber` to `1`, so `undefined` renders page 1 with no UI change. This also removes the latent dead path described in Root Cause.

### 3. Selection rule (presence check, most-recent tiebreak)
`selectPriorAuthSplitDocument` filters to `category === "Prior Authorization"` and returns the most recently created match's `blobPath`, or `undefined`. The "most recent" tiebreak covers the rare intake with more than one prior-auth document and is deterministic regardless of input order.

### 4. Batched lookup for the list routes
The detail path already fetched the intake's documents, so it just threads them in. The two list routes add one batched query per page â€” `getPriorAuthDocumentsByIntakeIds` runs `find({ intakeId: { $in: pageIntakeIds }, category: "Prior Authorization" })` and groups by `intakeId` â€” instead of one query per row. On query failure it logs and returns an empty map, so a lookup error degrades to the whole-fax fallback rather than failing the whole list response.

### Design decisions / alternatives considered
- **Match `category === "Prior Authorization"` only** (the canonical value written by the split flow). Rows literally labeled `"Insurance Authorization"` (a data mislabel) intentionally fall back to the whole fax; relabeling them is separate data cleanup â€” see Test Plan note.
- **No `Document` schema change and no change to the review/edit workflow** â€” all OCR and verify data still live on the intake. A full "edit from documents" migration was considered and rejected as disproportionate to a preview-source bug.
- **Join by the intake's own `_id`, never by `linkedOrderIds`** â€” keeps the preview scoped to the same intake already resolved for the patient/order.
