# [MLID-2086] refactor(document-notifications): filter unlinked Order-category docs and inverted-control orchestrator

## Summary

QA rejected the previous MLID-2086 work because documents of `category: 'Order'` that were uploaded against a patient but not yet linked to a specific LISA order were generating notification rows, audit-log entries, and SignalR broadcasts on every patient order. The business rule is that Order-category documents must propagate only to the specific orders listed in their `linkedOrderIds` array ‚Äî until a user links them via `/patient/[id]/document-records`, they are patient-level only.

This PR unifies the three triggers that propagate documents to orders behind a single orchestrator that owns the eligibility decision. The eligibility predicate is extracted to its own module and matches the precedent the Documents tab already uses.

**Branch:** `feature/MLID-2086-unlinked-order-document-filter` ‚Üí `develop`

**Jira:** [MLID-2086](https://localinfusion.atlassian.net/browse/MLID-2086)

---

## Root Cause

Three propagation surfaces were independently doing patient-level fan-out without any eligibility check:

| Function | What it did | Problem |
|---|---|---|
| `notifyDocumentCreated` | Called `getOrdersByPatient` and wrote a Notification + SignalR for every patient order | Wrote rows for unlinked Order-category docs |
| `recordDocumentAddedToOrders` | Called `getOrdersByPatient` and wrote an audit log for every patient order | Same ‚Äî Order-category docs ended up in every order's history |
| `notifyOrderOfRecentDocuments` | Called when a new order is created ‚Äî wrote a Notification for every recent patient document | Same ‚Äî surfaced unlinked Order docs on the new order |

The Documents tab list already filtered the unlinked Order docs (`apps/web/app/api/orders/details/route.ts:435-451`), but the sibling propagation surfaces did not ‚Äî so the doc disappeared from the tab list while still inflating the counter badge and order history.

---

## Architecture ‚Äî inverted control

The previous design had each side-effect function do its own patient fan-out via `getOrdersByPatient` internally, with no eligibility check. This PR inverts that:

```
Trigger 1 (upload)            Trigger 2 (order created)        Trigger 3 (doc linked)
notifyDocumentCreated      notifyOrderOfRecentDocuments     firePropagationForNewlyLinkedOrders
  (enumerates patient        (enumerates patient                (enumerates the delta
   orders, hands each to      docs, hands each to                newlyAdded = newIds ‚àí oldIds,
   the orchestrator)          the orchestrator)                  hands each to the orchestrator)
        \                            |                                    /
         \                           |                                   /
          \                          v                                  /
           ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚ñ∫ propagateDocumentNotificationToOrder(order, document, ...) ‚óÑ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
                              ‚îÇ
                              ‚îÇ  Eligibility gate (single source of truth):
                              ‚îÇ    document.category !== 'Order'
                              ‚îÇ    || (linkedOrderIds ?? []).includes(String(order._id))
                              ‚îÇ
                              ‚îÇ  Three independent steps, each in its own try/catch
                              ‚îÇ  with own log lines, awaited sequentially:
                              ‚îú‚îÄ‚ñ∫ Step 1: writeOrderDocumentNotification(order, document)
                              ‚îú‚îÄ‚ñ∫ Step 2: writeOrderDocumentAuditEntry(order, document)
                              ‚îî‚îÄ‚ñ∫ Step 3: broadcastDocumentAddedEvent(order, document)
```

Eligibility lives in a new module `apps/web/services/notifications/documentEligibility.ts` so the same predicate governs every propagation surface AND matches the Documents tab filter at `apps/web/app/api/orders/details/route.ts:435-451`. Single rule, single place.

## Trigger details

### 1. Document uploaded via /intakes

`notifyDocumentCreated` becomes a thin caller-side fan-out across `getOrdersByPatient`. Called from:

- `apps/web/app/api/documents/split-pdf/route.ts` ‚Äî both the full-upload and split-upload paths
- `apps/web/services/jobs/definitions/jotFormSubmission/weinfuseUploader.ts` ‚Äî webhook-driven upload (JotForm / Spruce)

Order-category documents are unlinked at upload time (verified by grep: `linkDocumentToOrders` is the only writer of `linkedOrderIds`), so the orchestrator's eligibility gate naturally suppresses them. No special-case logic at the upload sites.

The previously parallel `recordDocumentAddedToOrders` calls at these sites are dropped; their `actorId` and `auditSource` values are now passed through `notifyDocumentCreated` to the orchestrator.

### 2. New order created ‚Äî backfill

`notifyOrderOfRecentDocuments` is called from `app/api/orders/new/route.ts:154` when an order is created. It is rewritten to enumerate the patient's recent (30-day) documents and call the orchestrator for each `(newOrder, existingDocument)` pair.

The previous coalesced single SignalR broadcast pattern is dropped: each call now emits its own broadcast. Per a team convention recorded as memory `feedback_signalr_cheap_mongo_expensive`, SignalR events are cheap and Mongo writes are the real cost ‚Äî we do not coalesce broadcasts to save events.

### 3. Document linked via /patient/[id]/document-records

A new helper `firePropagationForNewlyLinkedOrders` is added to `documentNotification.ts` and fired from `PUT /api/documents/[documentId]/link-orders` after a successful `linkDocumentToOrders`.

Delta computation prevents duplicates:

```ts
const previousLinkedOrderIds = doc.linkedOrderIds ?? [];
const updated = await linkDocumentToOrders(documentId, sanitizedIds);
const newlyAddedOrderIds = sanitizedIds.filter(
  (id) => !previousLinkedOrderIds.includes(id)
);
if (newlyAddedOrderIds.length > 0) {
  firePropagationForNewlyLinkedOrders({
    documentId,
    document: { _id: updated._id, category: updated.category, linkedOrderIds: updated.linkedOrderIds },
    newOrderIds: newlyAddedOrderIds,
    actorId: session.user?.id || '',
    auditSource: 'user',
  }).catch((error) => {
    logger.error('Link-time propagation failed', { documentId, error });
  });
}
```

- Re-saving the same selection ‚Üí `newlyAddedOrderIds` is empty ‚Üí no propagation
- Removing an order ‚Üí `newlyAddedOrderIds` is empty ‚Üí no propagation
- Adding A to `[]`, then B to `[A]` ‚Üí only B propagates the second save

Order lookup tries `'new'` first and falls back to `'maintenance'`, mirroring the MLID-2085 fallback pattern.

## Deletions

- `recordDocumentAddedToOrders` ‚Äî replaced by `writeOrderDocumentAuditEntry` (the per-order primitive the orchestrator calls). No callers remain after the upload-site refactor.
- `dispatchOrderDocumentNotification` ‚Äî internal helper that was only used by the old `notifyDocumentCreated`. Replaced by the orchestrator path.

Per `feedback_delete_unused_code`, these are deleted in the same PR rather than left as dead code.

---

## Step independence

Each side-effect primitive (`writeOrderDocumentNotification`, `writeOrderDocumentAuditEntry`, `broadcastDocumentAddedEvent`) owns its own try/catch and never throws back to the orchestrator. The orchestrator awaits them sequentially in the order **Notification ‚Üí Audit ‚Üí SignalR**.

If any step fails:
- It is logged with `orderId`, `documentId`, and the error
- The subsequent steps still run
- The orchestrator resolves cleanly

Each step also logs an info line on success so we can correlate end-to-end execution in Application Insights.

## Eligibility predicate

`isDocumentEligibleForOrder(document, order)` in `apps/web/services/notifications/documentEligibility.ts`:

```ts
export function isDocumentEligibleForOrder(
  document: EligibilityDocument,
  order: EligibilityOrder
): boolean {
  if (document.category !== 'Order') return true;
  return (document.linkedOrderIds ?? []).includes(String(order._id));
}
```

- `category !== 'Order'` ‚Üí always eligible
- `category === 'Order'` ‚Üí eligible only if `linkedOrderIds` includes `String(order._id)`

`String(order._id)` lets callers pass either a Mongo ObjectId or a string ‚Äî the predicate keeps working either way.

## Behavior change worth flagging

Previously `notifyDocumentCreated` and `notifyOrderOfRecentDocuments` early-returned for any order with no `assignedTo`. After this refactor, an unassigned order still receives the **audit entry** and **SignalR broadcast** ‚Äî only the notification step is skipped (since there is no `targetUserId` to target).

Order history is per-order, not per-user, so audit entries should land regardless of whether an assignee exists at the moment of propagation. The relevant test has been updated to assert the new contract.

## Out of scope (called out for QA / PO)

- **Orders-list global badge vs Documents-tab per-user badge inconsistency.** After T4, the orders-list shows 6 unread while the Documents tab shows 5 for the same order. The orders-list badge is intentionally global (keyed off `targetUserId`, per the 2026-04-14 "team dashboard" decision in `apps/web/services/mongodb/notification.ts:176-205`) and predates MLID-2085's relaxed guard. The inconsistency exists on `develop` today. Recommend a follow-up ticket if user-visible.
- **`targetUserId` cascade on reassignment.** Already tracked as the [MLID-2391](https://localinfusion.atlassian.net/browse/MLID-2391) follow-up that was noted in MLID-2085's plan.
- **"Document unlinked" audit entries.** Removing an order from `linkedOrderIds` does not produce a history record. Not requested in the spec.
- **Production data backfill / cleanup.** Existing stray notification or audit rows written by the pre-fix code remain in the database. No migration in this PR.

## How to validate in browser

1. Pick a patient with at least one active order.
2. Upload via Intake Analyser: a Clinical doc ‚Üí propagates to every patient order; an Order-category doc ‚Üí does NOT propagate anywhere.
3. Go to `/patient/[id]/document-records`, find the Order-category doc, click **Update Order Linking**, select the order, save. The order's Documents tab counter increments by 1; Order History gains `"New Document received (Document type = Order)"`.
4. Re-save the same link selection: no duplicate entries.
5. Add a second order to the link: only that new order receives an entry.
6. Create another new order on the same patient: the backfill writes entries for every non-Order recent doc and for every Order-category doc whose `linkedOrderIds` includes the new order; everything else is filtered.
