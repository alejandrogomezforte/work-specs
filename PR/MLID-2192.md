# [MLID-2192] fix(orders-tracker): include "Undetermined" insurance status in primary insurance lookup

## Summary

- The `patient_insurances` `$lookup` in `getOrdersAggregate()` filtered status strictly to `"Active"`, silently dropping records whose status is `"Undetermined"`. This caused three downstream regressions for the 94 priority-1 patients with Undetermined insurance in production.
- One-line fix to widen the lookup's `$match.$expr.$and` clause from `$eq: ['$status', 'Active']` to `$in: ['$status', ['Active', 'Undetermined']]`. No type, schema, feature-flag, or UI changes.
- Behaviour for `"Inactive"` and `"Archived"` status values is unchanged (still excluded).

## Root Cause

The shared primary-insurance lookup at `apps/web/services/mongodb/ordersTracker.ts:362–381` runs whenever `category === 'new'` or `category === 'maintenance'` and populates `patient.primaryInsurance`, which is consumed by three downstream features:

1. The "Insurance" display column on the New Orders table (`apps/web/app/orders-tracker/Orders/NewOrders.tsx`, behind `ORDERS_TRACKER_IG_COLUMNS`).
2. The pharmacy eligibility `$switch` in `getEligibilityPipelineStages()` (behind `ORDER_ELIGIBILITY`).
3. The 340B eligibility `$switch` for both new and maintenance categories.

Because the inner `$match` had `{ $eq: ['$status', 'Active'] }`, any insurance whose `status` was `"Undetermined"` failed the lookup and `patient.primaryInsurance` resolved to `null`. The "Insurance" column rendered empty, and both eligibility `$switch` blocks short-circuited to "Waiting on Insurance Input" / `null` instead of evaluating the treatment / Medicare / hospital-system branches.

Per the WeInfuse OpenAPI spec (`apps/web/apis/weinfuse/openapi.json:898`) and the existing TypeScript unions in `apps/web/types/patientOrderReview.ts:107`, `apps/web/components/Modules/document-intake/schemas/types.ts:148`, and `apps/web/services/intakes/intakeAiService.ts:93`, `"Undetermined"` is a canonical status alongside `"Active"` and `"Inactive"` — not a transitional or null-marker value. Live MongoDB confirms 94 priority-1 records carry the literal string `"Undetermined"` (no nulls, no missing values, no empty strings).

## What Changed and Why

### `ordersTracker.ts` — widen the status filter

Before (line 372):

```ts
{ $eq: ['$status', 'Active'] },
```

After:

```ts
{ $in: ['$status', ['Active', 'Undetermined']] },
```

The change is intentionally minimal: the surrounding `$and` array, `let.patient_id`, and `$limit: 1` are unchanged, so `patient.primaryInsurance` still resolves to a single doc (priority-1, status now in the allow-list) or remains `null` if no qualifying insurance exists. All three downstream consumers (display, pharmacy eligibility, 340B) automatically pick up the wider allow-list without any change to their own logic.

**Alternatives considered:**

- `$or` with two `$eq` clauses — rejected. `$in` is shorter, idiomatic, and easier to extend if PM later wants to add more statuses.
- Adding a separate feature flag — rejected. This is a bug fix, not a new feature; the existing `ORDER_ELIGIBILITY` and `ORDERS_TRACKER_IG_COLUMNS` flags already gate the consuming UI and computations, and toggling those off restores pre-fix behaviour.
- Including `"Archived"` as well — rejected. The Jira AC explicitly lists only `"Active"` and `"Undetermined"`. There are 2,811 priority-1 `"Archived"` records in production; expanding to include them would be out of scope and may surprise PM. (Worth flagging to PM separately.)

### `ordersTracker.test.ts` — pipeline-structure assertion

Existing tests in `with includeEligibility` (lines 596–928) check stage presence/shape and field names but never assert on the inner `$status` clause value, so none of them would have caught this bug and none of them break with the fix. To establish a regression net, a new describe block captures the executed pipeline from a mocked `aggregate` call, finds the `patient_insurances` `$lookup`, and asserts both halves of the migration:

```ts
expect(hasInClause).toBe(true);    // new: $in ['$status', ['Active', 'Undetermined']]
expect(hasOldEqClause).toBe(false); // old: $eq ['$status', 'Active'] is gone
```

Asserting both halves prevents an accidental future revert and documents the explicit migration. The new test is placed inside `with includeEligibility` (rather than at the top level) to reuse the existing `setupMockDb` helper and to keep all eligibility-pipeline coverage co-located.
