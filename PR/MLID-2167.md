## Update — safety-net additions on top of the original fix

The immediate `message.save()` above closes the deferred-save race in ~99% of cases, but leaves three smaller gaps open: the microtask window between Spruce 200 and the DB write, Spruce redelivering the webhook (which would double-post "Processed by LISA"), and any miss with no recovery path (process crash, transient Mongo error, Spruce-side delivery failure). The commits below close those.

### 1. Stamp `spruce.archivedAt` on successful archive + webhook idempotency

- New optional `spruce.archivedAt: Date` field on the Message schema and DTO.
- `noteAndArchiveConversation` now returns `Promise<boolean>` — `true` only when `archiveSpruceConversation` itself resolves. Every other branch (flag off, note failed, archive threw) returns `false`.
- `archiveSmsConversation` adds `'spruce.archivedAt': { $exists: false }` to its `findOne` filter. A redelivered webhook for an already-archived message misses the lookup and silently skips — no second internal note posted.
- After a successful archive, `Message.updateOne` stamps `spruce.archivedAt`, scoped with `$exists: false` so two racing callers cannot both win.
- Exported `ARCHIVABLE_SOURCE_LOG_PREFIXES` and `noteAndArchiveConversation` so the sweeper reuses the same flow (including the 2 s internal-note wait) without duplicating it.

Files: `apps/web/models/Message.ts`, `apps/web/types/messageDTO.ts`, `apps/web/services/webhooks/handlers/spruce_conversation.ts`, `apps/web/services/webhooks/handlers/spruce_conversation.test.ts`.

### 2. `SPRUCE_ARCHIVE_SWEEPER` feature flag

- New flag in `utils/featureFlags/featureFlags.ts`, default `false`.
- Gates the sweeper handler so the Pulse schedule can fire unconditionally while the work stays dormant until an admin toggles the flag. Same pattern as `LINKED_ORDERS_EXPORT`.

### 3. Safety-net Pulse job (`spruceArchiveSweeper`)

Reconciles any outbound SMS whose post-send webhook missed. The webhook remains the real-time primary path; this job catches the tail with a few-minutes-worst-case latency.

- Runs every 5 minutes. Scans the last 1 h, skipping the most recent 2 min so the webhook wins the normal race.
- Candidate filter: `source` in the archivable set, `spruce.conversationId` present (webhook already linked it), `spruce.archivedAt` absent, `sentDate` inside the window.
- Batch cap of 100 messages per tick — fits comfortably inside the 5-min `lockLifetime` and bounds Spruce API pressure.
- Each candidate is atomically claimed via `findOneAndUpdate` keyed on `archivedAt` absence. A concurrent late webhook and the sweeper can never both post the "Processed by LISA" note — only one wins the claim, the other's filter misses.
- Calls the same `noteAndArchiveConversation` the webhook uses. On archive failure, `$unset`s `archivedAt` so the next tick can retry; on success, leaves it stamped.

Files: `apps/web/services/jobs/definitions/spruceArchiveSweeper.ts` (new), `apps/web/services/jobs/__tests__/spruceArchiveSweeper.test.ts` (new), `apps/web/services/jobs/definitions/index.ts`, `apps/web/services/jobs/definitions/index.test.ts`, `apps/web/services/jobs/scheduler.ts`, `apps/web/services/jobs/scheduler.test.ts`.

## Rollout plan

The sweeper ships gated off because of one first-deploy edge case: messages sent in the hour **before** the deploy have `spruce.conversationId` from the old webhook but no `archivedAt`. If the sweeper ran against them immediately, it would try to archive conversations the old path had already archived in Spruce and post a duplicate "Processed by LISA" note on each.

Two-month-old historical messages are **not at risk** — the `sentDate: { $gte: now − 1 h }` filter already excludes anything older than an hour. The historical backlog is invisible to this job.

Recommended sequence:

1. Merge + deploy. Flag is off, sweeper handler short-circuits with zero side effects.
2. Verify the webhook changes on live traffic: new outbound SMS shows ↗ in LISA, `spruce.archivedAt` populated in Mongo, conversation archived in Spruce.
3. **Wait ≥ 1 h** after deploy so every message whose `sentDate` is inside the sweeper's lookback window is post-deploy and already carries `archivedAt` from the new webhook code.
4. Toggle `SPRUCE_ARCHIVE_SWEEPER` on via the admin panel. First real run should log `scanned: 0` or a very low count.
5. Repeat for production.

If anything misbehaves in production, toggle the flag off — no redeploy needed.
