# [MLID-2669] fix(prior-auth): stop Verify Data step from freezing on an infinite render loop

## Summary
- On Order Details → Prior Auth → open a letter → **Verify Data** (step 2), the screen froze: the header **Back** button and every other control stopped responding.
- The reported symptom was a dead Back button, but the button handler was correct — the whole React tree was frozen by an infinite render loop in the shared `VerifyDataForm`.
- Fixes the loop in the shared component, so both the order-scoped Prior Auth screen (`AuthReviewDetail`) and the patient Prior Auth screen (`PriorAuthReview`) are corrected. No parent or route changes needed.

## Root Cause
`VerifyDataForm` reports its state upward through an effect:

- `const formValues = watch();` — react-hook-form's `watch()` returns a **new object identity on every render**.
- The reporting effect depended on `formValues`, so it ran on **every** render and called `onChange(formValues, checkValidity())` unconditionally.
- Both parents store that output in state (`setStep2Data(formData)`), which always receives a fresh object reference, so the parent always re-rendered → the form re-rendered → `watch()` produced a new object → the effect fired again.

This is an unbounded feedback loop. React aborts it with *"Maximum update depth exceeded"*, leaving the tree frozen. The order screen's Back uses a client-side `router.push`, which cannot complete while the tree is frozen (hence "dead Back button"); the patient screen's Back uses `router.back()` (native browser navigation), which escaped the frozen tree — masking the same underlying loop there.

## What Changed and Why

### The fix — guard the reporting effect
The effect still runs each render (inherent to reading a `watch()` snapshot), but it now only pushes an update to the parent when the content genuinely changed, giving the feedback loop a fixed point.

Before:
```ts
const formValues = watch();
// ...
useEffect(() => {
  onChange(formValues, checkValidity());
}, [formValues, checkValidity, onChange]);
```

After:
```ts
const formValues = watch();
// ...
const lastSentRef = useRef<string | null>(null);
useEffect(() => {
  const isValid = checkValidity();
  const signature = JSON.stringify([formValues, isValid]);
  if (signature === lastSentRef.current) {
    return;
  }
  lastSentRef.current = signature;
  onChange(formValues, isValid);
}, [formValues, checkValidity, onChange]);
```

Design decisions:
- **Kept inside the shared component.** The guard does not depend on either parent passing a referentially stable `onChange`, so it is robust for both consumers and any future one.
- **No `eslint-disable`.** The dependency array stays honest (`[formValues, checkValidity, onChange]`).
- **Preserved existing behavior.** The first mount still emits once (so the wizard's Next/validity gating keeps working), and real edits still propagate value and validity.

Alternatives considered: subscribing via `watch(callback)` (more idiomatic but a larger refactor: `checkValidity` would need to accept values and mount would need an explicit initial emit), and depending on `JSON.stringify(formValues)` in the dependency array (simpler but requires an `eslint-disable` and relies on a stable parent `onChange`). The ref-guard was chosen as the smallest self-contained change.

### The regression test
Existing tests used a stable `onChange` mock that never calls `setState`, so they could not reproduce the loop. The new test adds a stateful harness that mirrors the real parents — it stores the `onChange` output in state and feeds it back as `initialData` — and asserts the render does not exceed the update depth. All prior `VerifyDataForm` tests remain green, confirming value/validity propagation is unchanged.

Note on the test's failure mode: against the buggy code this scenario *hangs* (a synchronous loop inside React's `act()` blocks the event loop, so a per-test timeout cannot interrupt it) rather than failing with a clean assertion. It protects the fix while green.
