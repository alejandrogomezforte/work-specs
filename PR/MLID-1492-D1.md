# [MLID-1492] feat: 340B Eligible Column (D1 - feature flag, pipeline, API gating, UI column)

## Summary

This PR delivers the first of two deliverables for the Order 340B & Pharmacy Eligibility epic (MLID-1492). It adds a **340B Eligible** display-only column to the New Orders DataGrid in the Referral Tracker. The eligibility value is computed at query time via MongoDB aggregation pipeline stages — no stored fields, triggers, or background jobs.

**Epic branch:** `epic/MLID-1492-order-eligibility` -> `develop`

## What Changed and Why

### Feature Flag & Types (MLID-1857)

- Added `ORDER_ELIGIBILITY` to `FeatureFlag` enum with default `false`
- Added `hospital340BEligible?: string` and `pharmacyEligible?: string` to the `Extra` type in `FullOrder<K>`, gated to `K extends 'new'` so they only exist on new orders

**Files:**
- `apps/web/utils/featureFlags/featureFlags.ts`
- `apps/web/types/orders.ts`

### 340B Pipeline Stages (MLID-1858)

Added query-time computation of 340B eligibility via 4 new aggregation pipeline stages appended to `expandIdsAggregate` inside `$facet.data`, **after `$skip/$limit`**. Only the current page of orders (25-100) gets processed.

**Pipeline stages:**

1. `$addFields: _providerNpi` — extracts NPI from the resolved provider doc or raw NPI string
2. `$lookup: hospitalSystems` — finds active hospital systems where `providers[].npi` matches the order's provider NPI
3. `$lookup: patient_insurances` — finds primary active insurance (`priority='1'`, `status='Active'`) via `patient.we_infuse_id_IV`
4. `$addFields: hospital340BEligible` — `$switch` computation with 3 outcomes:
   - **No hospital match** -> `"No"`
   - **Hospital match, no insurance** -> `"Waiting on Insurance Input"`
   - **Hospital match + Commercial insurance** -> comma-separated hospital names (e.g., `"Hospital ABC, Hospital XYZ"`)
   - **Hospital match + non-Commercial insurance** -> `"No"`

Stages are conditionally included only when `category === 'new' && includeEligibility === true`. The `_primaryInsurance` lookup is shared — it will be reused by the pharmacy eligibility stages in D2.

**Files:**
- `apps/web/services/mongodb/ordersTracker.ts`
- `apps/web/services/mongodb/__tests__/ordersTracker.test.ts`

### API Route Feature Flag Gating (MLID-1860)

The `GET /api/orders/new` route checks the `ORDER_ELIGIBILITY` feature flag via `FeatureFlagService.getFeatureFlag()` and passes `includeEligibility: true/false` to `getOrders()`. When the flag is off, the pipeline stages are not appended — zero overhead.

**Files:**
- `apps/web/app/api/orders/new/route.ts`
- `apps/web/app/api/orders/new/route.test.ts`

### UI Column + Feature Flag Wiring (MLID-1859)

- `page.tsx`: calls `useFeatureFlag(ORDER_ELIGIBILITY)` and passes `showEligibilityColumns` to `getNewOrdersColumns()`
- `NewOrders.tsx`: accepts `showEligibilityColumns` param, adds 340B column via spread-ternary pattern (same pattern as the existing delete action column)
- Column config: `field: 'hospital340BEligible'`, `headerName: '340B Eligible'`, `width: 200`, `sortable: false`, `filterable: false`
- `valueGetter` returns `row.hospital340BEligible ?? ''`

**Files:**

## Architecture Decisions

| Decision | Rationale |
|----------|-----------|
| **Query-time computation** (not stored fields) | Zero staleness — always reflects latest hospital systems, patients, and insurance data. No triggers, no background jobs, no new indexes on `neworders`. |
| **Post-pagination stages** | Only 25-100 orders per page get the extra lookups, keeping performance impact negligible. |
| **Shared `_primaryInsurance` lookup** | Looked up once, used by both 340B (this PR) and pharmacy eligibility (D2). |
| **Display-only columns** | No filtering or sorting — columns are `sortable: false`, `filterable: false`. Adding filter/sort would require moving stages before `$facet`, which is a future enhancement if needed. |
| **Feature flag at two levels** | Server: prevents pipeline stages from running. Client: prevents column from rendering. Both check `ORDER_ELIGIBILITY`. |
