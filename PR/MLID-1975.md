# [MLID-1975] feat(orders-tracker): add 340B eligible column to maintenance orders

## Summary
- Extend the existing 340B Eligible column (currently only on New Orders) to also appear on the Maintenance Orders table
- Reuses the existing `ORDER_ELIGIBILITY` feature flag, `getEligibilityPipelineStages()` pipeline, and the same UI column pattern
- Only 340B is added â€” pharmacy eligibility is intentionally NOT included for maintenance orders
- Full pipeline (340B + pharmacy) runs for maintenance but only 340B is displayed in the UI, so adding pharmacy later is a pure UI change

## What Changed and Why

### Type Widening (`orders.ts`)
`hospital340BEligible` was typed as `K extends 'new' ? string : never`, meaning it was only available on `FullOrder<'new'>`. Widened to `K extends 'new' | 'maintenance' ? string : never` so maintenance orders can carry the computed 340B value. `pharmacyEligible` remains scoped to `'new'` only.

### Pipeline Guard (`ordersTracker.ts`)
The eligibility pipeline stages were gated behind `category === 'new' && includeEligibility`. Changed to `(category === 'new' || category === 'maintenance') && includeEligibility`. This reuses the full `getEligibilityPipelineStages()` function â€” no parameterization needed. The pharmacy stages run but the result is simply not displayed in the UI.

### API Route Feature Flag (`maintenance/route.ts`)
Mirrors the pattern from `app/api/orders/new/route.ts`: checks `FeatureFlagService.getFeatureFlag(FeatureFlag.ORDER_ELIGIBILITY)` and passes the boolean as `includeEligibility` to `getOrders()`. This is the server-side enforcement â€” when the flag is off, no eligibility pipeline stages are appended.

### UI Column (`MaintenanceOrders.tsx`)
Added `showEligibilityColumns?: boolean` param to `getMaintenanceOrdersColumns()`. When true, inserts a `hospital340BEligible` column using the same spread-ternary pattern as `NewOrders.tsx`. Column is read-only, non-sortable, non-filterable, width 200. Placed before the IG columns.

### Page Wiring (`page.tsx`)
`isOrderEligibilityEnabled` already existed (from `useFeatureFlag(FeatureFlag.ORDER_ELIGIBILITY)` at line 101). Simply passed it as `showEligibilityColumns` to the maintenance columns builder. This is the client-side enforcement.

### Design Decision: Compute Both, Display Only 340B
The full eligibility pipeline (340B + pharmacy) runs for maintenance orders, but only the 340B column is rendered. This avoids parameterizing the shared `getEligibilityPipelineStages()` function and means adding pharmacy to maintenance in the future is a pure UI change with no backend work.
