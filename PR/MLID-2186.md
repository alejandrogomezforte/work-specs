# [MLID-2186] feat(pdf-viewer): add Download button to ReactPdfRenderer toolbar

## Summary

Adds a **Download** icon button to the `ReactPdfRenderer` toolbar that downloads the currently-loaded PDF blob to the user's computer. Because `ReactPdfRenderer` is the single PDF viewer used everywhere via `EmbeddedFileViewer`, this one component change enables Download on every preview surface in LISA.

## What Changed

### 1. New Download button in the PDF toolbar

`ReactPdfRenderer.tsx` — added a `<DownloadIcon>` button next to the existing rotate / save / fullscreen buttons. Clicking it builds a client-side anchor pointing at the already-loaded blob URL and triggers a download:

```ts
const anchor = document.createElement('a');
anchor.href = objectUrl;
anchor.download = normalizeDownloadFileName(fileName);
document.body.appendChild(anchor);
anchor.click();
document.body.removeChild(anchor);
```

No extra fetch is needed — `EmbeddedFileViewer` has already materialised the PDF as a `blob:` URL in browser memory.

### 2. Filename normalization

New module-level helper appends `.pdf` if missing (case-insensitive) and falls back to `document.pdf` when no filename is supplied:

```ts
function normalizeDownloadFileName(fileName: string | undefined): string {
  if (!fileName) return 'document.pdf';
  return fileName.toLowerCase().endsWith('.pdf') ? fileName : `${fileName}.pdf`;
}
```

### 3. `EmbeddedFileViewer` forwards `fileName`

`EmbeddedFileViewer` already accepts a `fileName?: string` prop (used for the Dialog title). This PR threads it through to **both** internal `ReactPdfRenderer` render sites: the inline instance and the fullscreen-popout Dialog instance. No callers needed changes — every one of the 11 call sites already passes a sensible filename.

### 4. CSS

New `.downloadButton` class in `ReactPdfRenderer.module.css`, matching the icon-only visual style of the existing rotate button.
