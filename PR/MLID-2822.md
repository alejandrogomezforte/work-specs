> - `hotfix/MLID-2822-case-insensitive-login-develop` (off `develop`) Ôćĺ targets **`develop`** (keeps develop in sync).
>

## Summary

Login email matching was exact and case-sensitive at every layer. A user whose stored email contained uppercase letters could be denied with `/?error=notAuthorized` whenever the case of the incoming email differed from the stored value. This makes the user lookup case-insensitive across both read backends (Mongo and Postgres) and the environment allow-lists, without changing any stored data.

The lookup also now deterministically prefers the **enabled** account when a case-variant pair exists, so a login resolves to the active account rather than a disabled duplicate.

## Root Cause

Every email comparison in the auth chain was an exact match with no normalization:

- `getUserByEmail` (the Google OAuth sign-in lookup) ran a plain `findOne({ email, deleted: { $ne: true } })` ÔÇö no collation.
- Its Postgres counterpart `findByEmailForRead` used `eq(users.email, email)`.
- `getUserWithPasswordHash` (credentials path) used a plain `findOne`.
- The `AUTHORIZED_EMAILS` / `AUDITOR_EMAILS` allow-lists compared raw strings.

For Google OAuth users this was mostly invisible because the address usually arrives lowercase; it surfaced when the stored email had mixed case (or when an identity provider returned a case that differed from what was stored).

## What Changed and Why

The fix normalizes the *lookup*, not the stored data (a full data-normalization pass is out of scope for this hotfix). On the Mongo side that is a case-insensitive collation; on the Postgres side a `lower(...) = lower(...)` comparison. Both paths keep an enabled-first ordering so that, when a mixed-case and a lowercase record both match, the login lands on the enabled one; the pre-existing post-fetch `disabled` guard stays the final gate.

The Postgres branch deliberately keeps its existing `orderBy(disabled, mongoId)` fail-open tiebreak rather than switching to a hard `disabled = false` filter, and the Mongo branch mirrors that shape ÔÇö so the two read backends behave identically.

## Deployment / operational note

There is **no data migration** in this change. The three known mixed-case duplicate accounts are disabled **manually by the PO** out of band. The code is self-sufficient without that step: the case-insensitive read plus the prefer-enabled ordering already resolves a live case-variant pair to the enabled account.
