## Summary

This PR introduces the new `location_name` field across the application, replacing the previous pattern of using `facility_name` as the display name for locations. The goal is to give each location a dedicated human-friendly name (e.g. "Boston LI") that is separate from the facility name.

### What changed

1. **API layer now returns `location_name`** — The patient and intake API routes were updated to include `location_name` in location lookups and prefer it as the display value.
2. **Patient-facing components use the new field** — `PatientInfoCard` now reads from `location.location_name` (with fallback to `location.city`) instead of the old flat fields (`location_city`, `location_name` at root level, `homeLocationId`).
3. **Admin locations table shows the new column** — A "Location Name" column was added to the locations list page, and the search service includes `location_name` in its query.
4. **Admin location edit form includes the new field** — The edit form now has a "Location Name" input and the form layout was reorganized into a cleaner 3-row structure.
5. **CSS grid columns extended** — New column span classes (`col-5` through `col-11`) were added to `DynamicForm` styles to support flexible layouts.

### Form layout (location edit page)

| Row | Fields |
|-----|--------|
| 1 | WeInfuse ID (6 cols) &#124; Location Name (6 cols) |
| 2 | Facility Name (6 cols) &#124; Servicing Address (6 cols) |
| 3 | City (4 cols) &#124; Callback Number (4 cols) &#124; Fax Number (4 cols) |
| 4+ | Unchanged (Group NPI, Tax ID, Pharmacy Eligible, etc.) |
