# [MLID-2838] docs(claude-md): document MM/DD/YYYY as the default date format

**Branch:** `feature/MLID-2838-claude-md-date-format` → `develop`
**Jira:** [MLID-2838](https://localinfusion.atlassian.net/browse/MLID-2838)

## Summary

LISA uses the US date format `MM/DD/YYYY` as its default across the UI and the rest of the application, but this convention was never written down. Without a single source of truth in `CLAUDE.md`, new code and AI-assisted changes had no rule to follow and drifted toward whatever format the surrounding file happened to use.

This PR adds a `Date Format` section to `CLAUDE.md`. Documentation only — no runtime code is affected.

## What changed

One file, 18 added lines. The new section sits between `Import Paths` and `Git Commits`, alongside the other cross-cutting conventions:

- States `MM/DD/YYYY` as the default everywhere, overridable only when a specific requirement calls for it.
- Points numeric date formatting at `formatUTCDate` from `@/utils/formatters` instead of hand-rolled formatters.
- Covers long form dates (`Feb 26, 2026`) via `formatDate`, keeping US ordering — month before day.
- Requires an explicit `en-US` locale on `toLocaleDateString` / `Intl.DateTimeFormat` calls.
- Warns against setting `adapterLocale` on `LocalizationProvider`.
- Carves out `YYYY-MM-DD` as storage and transport only, not a display format.

## Why the section covers more than the numeric format

A codebase check while drafting the section turned up two things that shaped the wording. Both are worth knowing when reviewing:

1. **Around twelve files call bare `toLocaleDateString()` with no locale argument.** These render `DD/MM/YYYY` on a non-US browser today. The explicit `en-US` rule names existing drift rather than stating a hypothetical. Examples: `apps/web/app/admin/holidays/page.tsx`, `apps/web/components/Modules/api-keys/ApiKeysTable.tsx`, several `apps/web/app/admin/integrations/*/page.tsx`.

2. **Long form dates (`Feb 26, 2026`) are slightly more common than numeric ones** in the review and chat UIs. Documenting only numeric `MM/DD/YYYY` would have contradicted a good chunk of the existing UI, so the section covers both shapes and ties them together under US ordering.

3. **MUI date pickers pass no `adapterLocale`.** They render `MM/DD/YYYY` only because date-fns defaults to `enUS` implicitly. That default is unpinned, hence the warning.

No code was changed to address any of these — the drift is pre-existing and out of scope here. If we want the bare-locale calls cleaned up, that is worth a separate ticket.

## Testing

Documentation-only change; no tests apply. Prettier reports the file as already conforming.
