# [MLID-2355] feat(auth-review): Needs Review counter badge on Prior Auth tab

## Summary

- Add a green numeric badge to the **Prior Auth** sub-tab inside `OrderDetailsReviewsPanel`. The badge shows the count of prior-auth documents for the open order that are in **Needs Review** status, and is hidden when the count is zero.
- The count comes from a new focused hook (`useAuthReviewNeedsReviewCount`) that calls the existing order-scoped auth-review endpoint with `reviewStatus=needs_review&page=1&pageSize=1` and reads the server-computed `total`. No new API route, no SignalR, no per-user read state, no new collections, no new feature flags, and no new env vars.
- Fix a counting defect discovered during browser testing: the badge (and the list/table status filter) **undercounted** Needs Review documents because the shared list pipeline defined "needs review" differently from the table display rule. The badge showed 2 while the table showed 3.

## Root Cause

The application decided "needs review" in two places that did not agree:

- **Table display** (`services/priorAuth/transforms.ts`): a row is shown as Needs Review whenever the stored `reviewStatus` is **anything other than the string `reviewed`** — this includes documents where the field is **absent**, null, or explicitly the string `needs_review`.
- **List/badge filter** (`services/priorAuth/query.ts`, `buildPriorAuthListPipeline`): the `needs_review` branch matched only `{ $exists: false }` (field absent). A document whose field is explicitly set to the string `needs_review` matched **neither** the `needs_review` branch nor the `reviewed` branch — it fell into a gap.

Because the two branches were not complements of each other, any document explicitly stamped `needs_review` was silently dropped from the count. The badge faithfully reused this filter, so it inherited the undercount.

**Fix:** change the `needs_review` branch from `{ $exists: false }` to `{ $ne: 'reviewed' }`, making it the exact complement of the `reviewed` branch and matching the table display rule. The badge now counts absent, null, and explicit `needs_review` documents alike.

## What Changed and Why

### 1. The badge feature (`feat`)

`OrderDetailsReviewsPanel` now calls `useAuthReviewNeedsReviewCount(orderId)` and spreads the badge onto the Prior Auth tab item, following the existing documents-tab badge pattern (hide when zero):

```ts
...(isAuthReviewEnabled
  ? [
      {
        label: 'Prior Auth',
        value: 'auth',
        ...(authReviewNeedsReviewCount > 0
          ? { badge: authReviewNeedsReviewCount }
          : {}),
      },
    ]
  : []),
```

The hook is intentionally minimal — it reuses the existing order-scoped endpoint and reads only the `total` field (a count, no PHI). The badge refreshes on navigation: after a document is reviewed, the detail page redirects back to the order dashboard, which re-mounts the panel and refetches.

### 2. The counting fix (`fix`)

In `buildPriorAuthListPipeline`:

```js
// before — misses documents explicitly stamped 'needs_review'
baseMatch['additionalInfo.priorAuthData.reviewStatus'] = { $exists: false };

// after — complement of the reviewed branch (absent, null, or 'needs_review')
baseMatch['additionalInfo.priorAuthData.reviewStatus'] = { $ne: 'reviewed' };
```

**Verified against live data** for a real order (4 in-scope documents: 2 with the field absent, 1 explicitly `needs_review`, 1 `reviewed`):
- Old filter returned **2** (the explicit `needs_review` doc fell into the gap).
- New filter returns **3**, matching the table — and the `reviewed` filter still returns its documents unchanged.

**Blast radius (intended):** `buildPriorAuthListPipeline` is shared. The same fix also corrects:
- the order auth-review table's own "Needs Review" status-dropdown filter, and
- the patient-level prior-auth list route's "Needs Review" filter.

Both are now consistent with the table coloring. This is a deliberate behavior change to the shared prior-auth filter (originally introduced under MLID-2306) and is the reason the patient-route test assertions were updated.

**Alternative considered:** fixing only the badge hook (for example, post-filtering client-side) was rejected — it would have left the list/table filter inconsistent with the table coloring and would not satisfy the badge's own acceptance criterion of matching the displayed Needs Review rows.
