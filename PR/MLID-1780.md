# [MLID-1780] fix(intakes): search input loses focus after each keystroke

## Summary

- Fixed the search input on the Intakes page losing focus after every keystroke, forcing users to re-click the field for each character
- Added 300ms debounced search to prevent unnecessary API calls on every keystroke

## Root Cause

`Controls` and `TableSection` were defined as **arrow functions inside the page component**. Every state update (including each keystroke) redefined these functions with new references. React interpreted `<Controls />` as a different component type on each render, causing it to **unmount the old DOM tree and mount a new one** — destroying focus on the search input.

Additionally, there was **no debounce** on the search input. Every keystroke immediately triggered `setFilters()` + `router.push()` + `loadIntakes()` (API call), compounding the re-render problem.

## What Changed and Why

### 1. JSX variables instead of inline component functions

**Before (broken):**
```tsx
const Controls = () => (<div>...</div>);
<Controls />  // new function identity every render → unmount/remount
```

**After (fixed):**
```tsx
const controls = (<div>...</div>);
{controls}  // plain JSX, React reconciles in place → preserves DOM + focus
```

Same change applied to `TableSection` → `tableSection` for consistency.

**Alternative considered:** Extracting `Controls` and `TableSection` into standalone components in a `components/` folder (pattern used by `/intakes/[id]/review`). This would also fix the bug, but requires defining large props interfaces (13 props for Controls, ~8 for TableSection) with no added value for this fix. Documented in the plan file as a potential follow-up refactor.

### 2. Debounced search (300ms)

Follows the established pattern from `hospital-system-agreements/page.tsx`:
- `searchInput` state updates immediately on every keystroke (input stays responsive)
- `searchTimerRef` debounces `updateFilters()` (API call + URL update) by 300ms
- Timer is cleaned up on unmount
