# [MLID-2447] fix(orders): backfill document notifications on maintenance order creation

**Branch:** `hotfix/MLID-2447-maintenance-order-notification-backfill` → `develop`

## Summary

The unread-notification badge on the maintenance orders table did not appear (or was undercounted) for newly-created maintenance orders. This PR adds the missing order-creation notification backfill to the maintenance orders route, mirroring the behavior the new-orders route already had. One change fixes every maintenance QA case reported so far.

## Problem

Notifications for an order are created by several triggers, all routed through `propagateDocumentNotificationToOrder`. One of them is the **order-creation backfill** (`notifyOrderOfRecentDocuments`): when an order is created, it fans the patient's existing documents onto the new order so documents that pre-date the order still produce a badge.

`POST /api/orders/new` fires this backfill right after auto-assignment. `POST /api/orders/maintenance` did not — the call was absent. Across the codebase the backfill was wired only into the new-orders route and the reassignment helper, never into maintenance creation.

Because the intake flow POSTs maintenance orders to this same route, the gap affected **every** maintenance order — both manually-created and intake-created — regardless of change type.

Observed symptoms:

- **Manually-created maintenance order:** badge count 0, and no "document added" entries in Order History.
- **Intake-created maintenance order:** badge count 1 (only the Order-category document, which is linked at creation and notifies through a separate trigger) instead of the full count of the patient's documents.

## Fix

In `apps/web/app/api/orders/maintenance/route.ts` (POST handler), after `autoAssignOrders`, loop the created orders and fire `notifyOrderOfRecentDocuments` (fire-and-forget), guarding on a resolvable patient object — identical to the new-orders route:

```ts
for (const createdOrder of maintenanceOrders) {
  if (
    !createdOrder.patient ||
    typeof createdOrder.patient !== 'object' ||
    !('_id' in createdOrder.patient)
  ) {
    continue;
  }
  notifyOrderOfRecentDocuments({
    orderId: String(createdOrder._id),
    assignedTo: createdOrder.assignedTo,
    patientId: String(createdOrder.patient._id),
  }).catch(() => {});
}
```

Plus the corresponding import. No new infrastructure — this reuses the existing shared entry point.

## Tests

Added a backfill test block to `apps/web/app/api/orders/maintenance/route.test.ts` covering:

- a single created order → one backfill call with the correct payload;
- the assignee resolved by `autoAssignOrders` is the one passed to the backfill;
- a protocol pair → one call per order;
- an order whose patient is not a resolvable object → skipped;
- `createOrder` returns null → no call;
- backfill rejects → route still returns 200 (fire-and-forget guard).

Full maintenance route suite: 63 passing. `tsc --noEmit` clean on the changed route. Prettier clean.

## Manual verification (staging data)

A maintenance order created after the fix, for a patient with 6 existing documents, produced **5** notifications — one per eligible document — written within a few seconds of order creation (the backfill firing on the POST). The 6th document is an Order-category document linked to a different order, so the eligibility gate correctly excludes it. Before the fix the same order would have had 0.

## Risk & rollout

- **Scope:** one added loop on the maintenance create path; the new-orders path is untouched.
- **Feature flag:** gated by `ORDER_DOCUMENT_NOTIFICATIONS` (unchanged) — off = no behavior change.
- **Fire-and-forget:** failures are swallowed and never affect the create response.
- **Not retroactive:** orders created before deploy are not backfilled; the fix applies to orders created afterward.
