## 🔴 Primary finding — notification stranded on a previous owner

The single notification on this order is a **`type: "user"`** document-upload notification whose `targetUserId` is
`69791ace5c552f7bf48f053f` (the **previous owner**) — which is **neither** the current assignee, **nor** the
assigner, **nor** `createdBy`.

**Timeline:**
- 2026-07-10 19:10:23 — order created, document uploaded, notification created targeting the previous owner.
- 2026-07-13 14:28:54 — order re-assigned (`lastAssigned`) to the new owner by the assigner.

**Hypothesis:** the notification was correctly created for whoever owned the order on 2026-07-10, but when the
order was reassigned on 2026-07-13 the notification's `targetUserId` was **not** moved to the new owner. The
codebase has `reassignUserNotificationsForOrder(orderId, fromUserId, toUserId)` in
`apps/web/services/mongodb/notification.ts` that is supposed to migrate `type: "user"` notifications to the new
assignee on reassignment. Either it did not run for this reassignment, or it ran with a `fromUserId` that did not
match `69791ace…`. Net effect: the new owner never sees the document notification, and the previous owner
(`69791ace…`) holds a notification for an order they no longer own.

## Root Cause Analysis

### Confirmed facts
1. One `type: "user"` document-upload notification exists on the order, created **2026-07-10** at order/document
   creation, targeting the **previous owner** (`69791ace…`).
2. The order was later re-assigned to the **new owner** (`6a359b…`) by the **assigner** on **2026-07-13**
   (`lastAssigned`).
3. The notification's `targetUserId` still points at the previous owner — it did **not** follow the
   reassignment. Result: the new owner has no document notification; the previous owner holds one for an order
   they no longer own; nobody has read it.

### The intended mechanism (and why it should have fixed this)
On reassignment, `PATCH /api/orders/maintenance` computes `previousAssignedTo` (the order's `assignedTo` before
the write) and `reassignedTo`, and when they differ calls `handleOrderReassignment(order, previousAssignedTo,
reassignedTo)`. For a non-initial reassignment that calls
`reassignUserNotificationsForOrder(orderId, fromUserId, toUserId)`, whose filter is:

```js
{ type: 'user', targetUserId: fromUserId, entities: { $elemMatch: { entityType: 'order', entityId: orderId } } }
```

With `fromUserId = previous owner` and `toUserId = new owner`, that filter **would** have matched this exact
notification and moved it to the new owner. So in the *current* code the retarget is correct and
collection-agnostic (wired for maintenance orders under MLID-2454 / D4-T5, in `handleOrderReassignment.ts` and
`api/orders/maintenance/route.ts`).

### Why it did not happen here — leading hypotheses (to confirm)
- **H1 — Reassignment did not go through the PATCH route.** If the 2026-07-13 assignment change was applied by a
  path that does not call `handleOrderReassignment` (bulk reassign, linked-order cascade, a WeInfuse sync — note
  `weInfuseOrderSeriesId: 1617224`, or a direct DB/admin edit), the retarget never fired.
- **H2 — Timing / deploy gap.** If the maintenance-order retarget wiring (MLID-2454 / D4-T5) reached production
  *after* 2026-07-13, the reassignment predated the fix and the notification was never migrated.
- **H3 — Feature flag.** `handleOrderReassignment` early-returns unless `ORDER_DOCUMENT_NOTIFICATIONS` is
  enabled. The creation of the 7-10 notification proves the flag was on then, but if it was toggled off around
  7-13 the retarget would have silently skipped. (Least likely.)

The audit log for this order distinguishes H1 vs H2/H3 by showing exactly how and by whom the `assignedTo` field
changed on 2026-07-13, and the `source` of that change.

## ✅ Confirmed Root Cause — deploy-timing gap (H2)

The maintenance-order notification-retarget cascade did not run because **the code that performs it was not yet
in production when the reassignment happened.**

- The retarget for maintenance orders was added by commit `abd89596e`
  *"[MLID-2454] - feat(orders): cascade notifications on maintenance order reassignment"* (authored 2026-06-26 on
  the `epic/MLID-2417-document-notifications-maintenance` branch).
- That commit first reached `main` **only** via the `v2026.07.14` release (tag `v2026.07.14`, cut
  2026-07-13 16:41 UTC, deployed as the 2026-07-14 release). No earlier tag/release contains it.
- The reassignment occurred **2026-07-13 14:28 UTC** — roughly two hours before the release was even tagged, and
  about a day before it was deployed to production.

So at 2026-07-13 14:28 UTC, production had the **audit-logging** and **reassignment** code (older features) but
**not** the `handleOrderReassignment` retarget wiring on `PATCH /api/orders/maintenance`. The order's
`assignedTo` moved previous owner → new owner and was audited, but nothing retargeted the existing document
notification. It stayed on the previous owner.

**Other hypotheses:**
- **H1 (non-user path):** ruled out — audit `source: "user"`.
- **H3 (feature flag off):** moot / ruled out — the 2026-07-10 notification exists (so
  `ORDER_DOCUMENT_NOTIFICATIONS` was on), and regardless the retarget code was not deployed, so the flag never
  came into play for this reassignment.

**Why the current code is correct (no code bug to fix):** `currentOrder` is loaded by a raw `findOne` on
`MaintenanceOrders`, so `assignedTo` is the stored string (previous owner's id). `previousAssignedTo` resolves to
the previous owner, `reassignedTo` to the new owner, they differ, `result` is an array — so the retarget branch
fires and `reassignUserNotificationsForOrder(orderId, previousOwner, newOwner)` matches this exact notification. A
reassignment done today would be handled correctly. This was purely a pre-deploy occurrence.

## Impact
- **Reported case:** 1 order (`6a51439ffa8305fc476c285a` / displayId `01wE`), 1 notification stranded on the
  previous owner, never delivered to the current owner, never read.
- **Actual scope (measured):** the detection sweep below returned **230 stranded `type: "user"` notifications**
  on `maintenanceorders` — order-scoped notifications whose `targetUserId` no longer matches the order's current
  `assignedTo`. (An initial run under-read the count as 20; the full count is 230.) The reported case was one of
  230.
- **Open verification:** split the 230 by the 2026-07-14 deploy boundary. Reassignments **before** the deploy are
  the expected H2 backfill. Any **on/after** the deploy would mean the shipped fix is not holding — a second,
  active defect rather than history. Bucketing query in the Detection section.
- **Why they don't self-heal:** a *future* reassignment retargets from the *then-current* owner, so a
  notification stranded on an earlier owner is never picked up and stays stranded.
- **Not yet swept:** `neworders` (its retarget cascade shipped earlier under MLID-2391, so a smaller pre-fix
  population may exist there — run the same pipeline with `from: "neworders"`).

## Detection — finding all affected notifications

**Data signature of a stranded notification:** a `type: "user"` notification, scoped to an order, whose
`targetUserId` differs from that order's current (non-empty) `assignedTo`. In healthy state these are always
equal; the deploy-timing gap left them out of sync.

Run as an **aggregation pipeline on the `notifications` collection** (Atlas aggregation editor). Enable
**Allow Disk Use** if it is slow. Swap `from: "maintenanceorders"` for `"neworders"` to sweep the other
collection.

```json
[
  { "$match": { "type": "user", "entities.entityType": "order" } },
  { "$addFields": {
      "_orderId": {
        "$let": {
          "vars": { "oe": { "$first": { "$filter": {
            "input": "$entities", "as": "e",
            "cond": { "$eq": ["$$e.entityType", "order"] } } } } },
          "in": "$$oe.entityId"
        }
      }
  } },
  { "$addFields": { "_orderObjId": { "$convert": { "input": "$_orderId", "to": "objectId", "onError": null } } } },
  { "$lookup": {
      "from": "maintenanceorders",
      "localField": "_orderObjId",
      "foreignField": "_id",
      "as": "_order"
  } },
  { "$addFields": { "_order": { "$first": "$_order" } } },
  { "$match": { "$expr": { "$and": [
      { "$ne": ["$_order", null] },
      { "$ne": ["$_order.deleted", true] },
      { "$eq": [{ "$type": "$_order.assignedTo" }, "string"] },
      { "$ne": ["$_order.assignedTo", ""] },
      { "$ne": ["$targetUserId", "$_order.assignedTo"] }
  ] } } },
  { "$project": {
      "_id": 0,
      "notificationId": "$_id",
      "orderId": "$_orderId",
      "notificationType": "$type",
      "strandedTargetUserId": "$targetUserId",
      "currentAssignedTo": "$_order.assignedTo",
      "notificationCreatedAt": "$createdAt",
      "orderLastAssigned": "$_order.lastAssigned",
      "isRead": "$isRead"
  } }
]
```

Each result row is a ready-made remediation instruction: on `notificationId`, move `targetUserId` from
`strandedTargetUserId` to `currentAssignedTo`.

**Variants:**
- Count only: replace the final `$project` with `{ "$count": "strandedCount" }`.
- Confirm the H2 timing gap for a given row: a genuine pre-release stranding has `orderLastAssigned` **after**
  `notificationCreatedAt` and **before** the 2026-07-14 deploy. A row whose `orderLastAssigned` is after the
  deploy came from a different cause and needs a separate look.

**Result on 2026-07-14 run:** 230 stranded notifications on `maintenanceorders` (an initial run under-read this
as 20). Spread across a smaller number of orders — one notification per `(order, document)` pair, so orders with
several documents contribute several rows.

## Pre-remediation safety checks

Two facts about the notification model, confirmed in code, that make a bulk retarget safe:
- **Reassignment retargets in place, never deletes.** `reassignUserNotificationsForOrder` is a single
  `updateMany` doing `$set: { targetUserId }`. It does not delete or recreate rows, so the gap did not create
  duplicates — the 230 are distinct, just stranded on the wrong user.
- **One notification per `(order, document)` pair is by design.** `writeOrderDocumentNotification` writes one
  row per document; an order with N documents legitimately has N notifications. Multiple notifications per order
  are expected, not duplicates.

**Duplicate check (run 2026-07-14):** grouped all order-scoped user notifications by `(orderId, documentId)` and
filtered to `count > 1`. **No rows returned** — no `(order, document)` pair has more than one notification.
Therefore retargeting all 230 to each order's current `assignedTo` cannot produce two identical notifications for
the same `(order, document, user)`. The bulk retarget is safe.

**Still open before executing:** the deploy-boundary bucketing (Detection variants) — confirm all 230 have
`orderLastAssigned` **before** the 2026-07-14 deploy. Any on/after the deploy would indicate the shipped MLID-2454
fix is not holding (an active defect), and those rows must be investigated rather than backfilled.

## Remediation

**1. Fix this specific notification** — retarget it from the previous owner to the current owner. Verify first
with a `find`, then update. In Atlas, on `notifications`:

Find (confirm exactly one doc):
```json
{ "_id": ObjectId("6a51439ffa8305fc476c2865"), "targetUserId": "69791ace5c552f7bf48f053f" }
```
Update (set `targetUserId` to the new owner):
```json
{ "$set": { "targetUserId": "6a359b7537e6874664f8bffe" } }
```
Note: `notifications` is dual-written to Postgres in the app, but a **manual Atlas edit updates Mongo only** —
the Postgres mirror will drift. Prefer driving this through an app-side path / script that dual-writes, or flag
the Postgres row for the same update.

**2. Sweep for other affected orders (optional, recommended).** Identify maintenance orders whose latest
`assignedTo` no longer matches the `targetUserId` of their `type: "user"` notifications, for reassignments dated
before the 2026-07-14 deploy. Best done as a script using `reassignUserNotificationsForOrder` (dual-write safe)
rather than manual edits.

**3. Process takeaway.** This was a benign race between a normal user action and a same-day release — no code or
config defect. If document-notification correctness on reassignment is considered high-value, a one-time backfill
after deploying such a cascade (mirroring the MLID-2391 initial-assignment backfill approach) would close the
pre-deploy gap automatically.

## Remediation — implemented (2026-07-14)

Rather than the manual one-off edits above, the fix shipped as a **one-time data migration** applying a frozen
230-row mapping (`{ notificationId, oldTargetUserId, newTargetUserId }`) computed from the detection sweep. Two
coordinated halves, because `notifications` is dual-written:

- **Mongo** — `apps/web/db-update/migration/043.reassign-stranded-notifications.mjs` (+ its test and
  `migration-data/043.reassign-stranded-notifications.json`).
- **Postgres** — `packages/db/migrations/0083_mlid_2799_reassign_stranded_notifications.sql` (mirror update:
  `target_user_mongo_id` + resolved `target_user_id` FK).

The migration is idempotent (retargets a notification only while it is still stranded on its recorded old owner,
so a re-run — or a row already moved by the deployed MLID-2454 code — is a no-op), backs up previous
`targetUserId` values, and supports `DRY_RUN`.

**How it reached each environment:**
- **develop / staging:** merged via PR into `develop`; the deploy pipeline's migration job applies it
  automatically on staging. Both `043` (Mongo) and `0083` (Postgres) are on develop.
- **production:** Ruslan applied the migration **directly against production from his local machine** (he has
  prod database access), rather than waiting for the develop → main release cycle.
- **Cancelled:** a separate Mongo-only hotfix branch off `main` was created and then deleted. It was redundant
  and would have caused harm — it would have delivered only the Mongo half to prod and split how prod vs develop
  received the Postgres side. The fix now lives on develop (flowing to prod via the normal release) and prod is
  already corrected by Ruslan's direct run.

## Summary (one paragraph)
Order `01wE` (`6a51439ffa8305fc476c285a`, a maintenance order) was created 2026-07-10 assigned to the previous
owner, generating a "New Document: Order" user notification targeting them. On 2026-07-13 14:28 UTC the assigner

to the current owner, never read. Current code handles this correctly; remediation is a one-off data fix for this
notification plus an optional sweep for other orders reassigned before the 2026-07-14 deploy.
