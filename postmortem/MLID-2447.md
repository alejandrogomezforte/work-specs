## Root cause, fix, and verification (Case 1 + Case 4)

Cases 1 and 4 are the **same defect**. Case 1 (comment 37178, 2026-07-01) reported a manually-created maintenance order with no counter and no documents-added history; Case 4 (comment 37199, 2026-07-02) reported an intake-created maintenance order counting 1 instead of 6. Both trace to one missing call.

### The triggers (design)

Notifications are created by several triggers in `apps/web/services/notifications/documentNotification.ts`, all routed through `propagateDocumentNotificationToOrder`:

1. `notifyDocumentCreated` ŌĆö a document is created ŌåÆ fan out to all the patient's new + maintenance orders.
2. `notifyOrderOfRecentDocuments` ŌĆö an order is created ŌåÆ fan out all the patient's existing documents onto the new order (the "order-creation backfill").
3. `handleOrderLink` ŌĆö a document is linked to orders (the Order-category case) ŌåÆ notify the newly-linked orders.
4. Initial-assignment backfill ŌĆö `handleOrderReassignment` re-runs trigger 2 when a previously-unassigned order gets its first assignee.

### Root cause (confirmed)

**`POST /api/orders/maintenance` never called the order-creation backfill (`notifyOrderOfRecentDocuments`).** The new-orders route calls it (`apps/web/app/api/orders/new/route.ts`, right after `autoAssignOrders`), but the maintenance route did not. Across the whole codebase the backfill was wired only in `new/route.ts` and `handleOrderReassignment.ts` ŌĆö never for maintenance creation.

Intake does not have its own order-creation logic: `apps/web/components/Modules/document-intake/components/submission-api.ts` POSTs to `/api/orders/new` for new orders and `/api/orders/maintenance` for maintenance orders. So the gap affected **every** maintenance order ŌĆö manual and intake alike.

That single gap explains both counts:

- **Case 1 (manual maintenance order): 0.** No creation backfill, and nothing was linked afterward ŌåÆ zero notifications. (For order 015K / `6a46935e46603415d8f558f5` the later DanaŌåÆAlejandro reassignment used the retarget path, which had no existing notifications to move.)
- **Case 4 (intake maintenance order): 1.** No creation backfill either ŌĆö the lone notification came from the document-link trigger (`handleOrderLink`) when the Order document was linked. The other 5 documents were created before the order existed, so their `notifyDocumentCreated` fan-out found no order and was not counted. Net = 1.

### The fix

Mirror the new-orders wiring in the maintenance POST handler. In `apps/web/app/api/orders/maintenance/route.ts`, after `autoAssignOrders`, loop each created order and fire the backfill (fire-and-forget), guarding on a resolvable patient object:

```ts
for (const createdOrder of maintenanceOrders) {
  if (
    !createdOrder.patient ||
    typeof createdOrder.patient !== 'object' ||
    !('_id' in createdOrder.patient)
  ) {
    continue;
  }
  notifyOrderOfRecentDocuments({
    orderId: String(createdOrder._id),
    assignedTo: createdOrder.assignedTo,
    patientId: String(createdOrder.patient._id),
  }).catch(() => {});
}
```

Plus the `notifyOrderOfRecentDocuments` import. Tests were added to `apps/web/app/api/orders/maintenance/route.test.ts` (single order, post-auto-assign assignee, protocol pair ŌåÆ 2 calls, unresolvable-patient skip, null-createOrder no-call, and 200-on-reject). Full suite: 63 passing; `tsc --noEmit` clean on the changed routes.

**Fix branch:** `hotfix/MLID-2447-maintenance-order-notification-backfill` (off `develop`).

**Files changed:**

- `apps/web/app/api/orders/maintenance/route.ts` ŌĆö import + backfill loop in POST.
- `apps/web/app/api/orders/maintenance/route.test.ts` ŌĆö mock + 6-test backfill block.

## Conclusion ŌĆö coverage of all QA-reported cases

QA (Daria Pelevaniuk) reported two failing scenarios on this ticket, and the second was clarified into two "bugs". All of them are the **same single defect** ŌĆö `POST /api/orders/maintenance` never fired the order-creation backfill (`notifyOrderOfRecentDocuments`) ŌĆö and all are resolved by this one hotfix. There is no separate second root cause.

| QA label | Scenario | Symptom | Example order | Resolution |
|----------|----------|---------|---------------|------------|
| "Case 1" (comment 37178) | Second maintenance order created **manually** (Drug/Dose/Frequency) for a patient who already has documents | No counter; no "document added" history entries | 014u (`6a4542bc1cc638571236e36e`) ŌĆö confirmed **0** notifications, assignee present | Backfill now runs at creation ŌåÆ notifications for the patient's existing documents |
| "Bug 1" (comments 37199 / 37205) | Maintenance order via **intake**, non-Maintenance type | Counter shows **1** instead of **6** | 015G (`6a4632553e7044dde7bcbb68`) ŌĆö this postmortem's Case 4 | Backfill creates the 5 non-Order docs; the 6th (Order-category) is created by the document-link trigger at link time ŌåÆ 5 + 1 = 6 |
| "Bug 2" (comment 37205) | Same order ŌĆö only one "Mark as Read" button | Notifications created only for the Order-category document, not the other types | 015G | The maintenance route is change-type-agnostic, so the backfill fires for every change type and every eligible document |

Passed scenarios (Actemra, Unassigned, Skyrizi+SQ ŌĆö all via intake) already worked and required no change.

### Why "Case 1" and "Case 4" produced different numbers (0 vs 1)

Both are the missing backfill, but they differ by whether any document happened to be linked to the order afterward:

- **Manual order (014u / 015K):** no document was linked after creation, and no backfill ŌåÆ **0**.
- **Intake order (015G):** the backfill was still missing, but the intake links the Order-category document to the order, and that link trigger (`handleOrderLink`) created its single notification ŌåÆ **1**.

With the fix, the intake order now backfills the 5 non-Order documents in addition to the 1 Order-document-link notification ŌåÆ the full **6**.

### Verified vs. to-verify-after-deploy

- **Verified in data:** the backfill fires on maintenance creation and produces one notification per eligible existing document ŌĆö order 015L, 5 notifications (see the Verification section above). This alone proves the root-cause fix.
- **To verify after deploy (QA):** a genuine intake maintenance order lands on exactly **6** (5 backfilled + 1 from the Order-document link), not 5. The standalone test could not exercise the "+1" because it reused a patient whose Order document was already linked to a different order. This is the single remaining end-to-end check and is called out in the PR as the post-deploy QA step.
