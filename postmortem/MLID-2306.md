# MLID-2306 — Prior Auth Table: Missing Document Postmortem

This file tracks cases found during QA where a prior-auth intake document is expected to
appear in the order's "Prior Auth" tab but does not. For each case we record the raw
collection data, identify which filter condition is responsible for the exclusion, and note
the root cause.

The two filters applied by `buildOrderScopedPriorAuthMatch` are:

1. **Date filter** — `effectiveDecisionDate >= order.createdAt`
   - `effectiveDecisionDate` = parsed `additionalInfo.ocrData.authorization.dateOfDecision.value`
     (YYYY-MM-DD string), falling back to `intake.createdAt` when absent or malformed.
   - `order.createdAt` = `neworders.createdAt` (LISA record creation timestamp — marked
     provisional in the plan; the business date is `neworders.received`).

2. **Drug filter** — show the document when:
   - `priorAuthData.liDrugId.value` == order drug id (hard match), OR
   - `liDrugId` is null/empty AND (`brandDrugName` or `genericDrugName` contains the order
     drug name, case-insensitive substring), OR
   - `liDrugId` is null/empty AND both name fields are also empty (passthrough).

## Case 1

**Order:** `6a2fd3deff38201a9d83f36d`
**Intake:** `697120ac33541f20296e7c6c`

### Intro

This order was received by the clinic on **2026-01-21** (stored in `neworders.received`),
but the LISA record for it was only created on **2026-06-15** (stored in `neworders.createdAt`).
The prior-auth intake document was also processed on **2026-01-21** and carries a
`dateOfDecision.value` of `"2026-01-21"`.

The date filter compares the intake's effective decision date against `neworders.createdAt`.
Since `2026-01-21 < 2026-06-15`, the condition fails and the document is excluded — even
though the intake is clearly associated with this order by date and patient.

This is a concrete instance of the **provisional `neworders.createdAt` problem** flagged in
the plan (Open Question 2): the field used as the order date is the LISA row creation
timestamp, not the real business received date. When a LISA record is created months after
the order was actually received, every prior-auth document from that period is hidden.

so the document is excluded — even though it is clearly associated with this order by
business date.

received date and matches the decision date exactly. Switching from `neworders.createdAt`
to `neworders.received` in the `orderCreatedAt` argument passed to
`buildOrderScopedPriorAuthMatch` would fix this case.

## Case 1-B

**Order:** `6a2fe5ecff38201a9d8403cd`
**Intake:** `6970eb99d9fd5cbe059f8bbd`

### Intro

Unlike Case 1, here the order's `createdAt` and the intake's `dateOfDecision.value` fall on
the **same calendar day** (`2026-01-21`). The reason the document is still excluded is a
time-of-day precision issue: `$dateFromString("2026-01-21")` produces
`2026-01-21T00:00:00.000Z` (midnight), while the order's `createdAt` is
`2026-01-21T11:45:48.696Z` (11:45 AM). Midnight is before 11:45 AM, so the `>=` condition
fails even though both timestamps share the same date.

Notably, `neworders.received` (`2026-01-21T11:40:03.975Z`) is also within the same day and
also has a time component later than midnight, so switching to `received` alone would **not**
fix this case. The comparison must be date-only (truncated to midnight on both sides) for
same-day documents to pass.

### Filter evaluation

| Field | Value |
|-------|-------|
| Decision date (OCR `dateOfDecision.value`) | `"2026-01-21"` |
| Parsed by `$dateFromString` | `2026-01-21T00:00:00.000Z` (midnight — no time in the string) |
| Order anchor date (`neworders.createdAt`) | `2026-01-21T11:45:48.696Z` |
| Condition: `decisionDate >= orderCreatedAt` | `2026-01-21T00:00:00Z >= 2026-01-21T11:45:48Z` → **FALSE** |
| Drug filter | not reached — date filter already excludes the document |

### Conclusion

The decision date and the order creation date are on the same calendar day, but the
comparison fails because `$dateFromString` on a date-only string (`"2026-01-21"`) produces
midnight (`T00:00:00.000Z`), while the order's `createdAt` has a time of 11:45 AM. Midnight
is strictly before 11:45 AM, so `>=` evaluates to false.

This is a **time-of-day precision issue**, distinct from Case 1's months-long gap. The fix
requires the comparison to be performed on **truncated dates** (both sides normalized to
midnight), so that a decision date of `2026-01-21` matches any order whose anchor date falls
on the same calendar day, regardless of the time component.

## QA Disposition — Cases 1 and 1-B

Both cases were reported by QA as bugs but are **false bugs**. The filter is behaving
correctly per the current implementation spec:

- **Case 1:** The decision date (`2026-01-21`) is before the LISA order creation date
  (`2026-06-15`). Per spec, only documents whose decision date is on or after the LISA order
  creation date are shown. The exclusion is correct.
- **Case 1-B:** The decision date string (`"2026-01-21"`) and the LISA order creation date
  (`2026-01-21T11:45:48Z`) fall on the same calendar day, but the datetime comparison is
  strict: midnight is before 11:45 AM, so the document is excluded. The filter is working
  exactly as implemented.

The QA reporter was notified of this explanation. No code change is required for these cases
under the current spec.

## Case 2

**Order:** `6a2fdc92ff38201a9d83fe75`
**Intake:** `6972643d21a76dbc70ff5bdc`

### Intro

This case has a different profile from Cases 1 and 1-B. The date filter **passes** — the
intake's decision date (`2026-01-21`) is after the order's creation date (`2026-01-12`). The
document is excluded by the **drug filter**.

The intake has no `brandDrugName` in the OCR service object, only a `genericDrugName`
of `"vedolizumab"` (Entyvio). The order is for **Gammaked** (`lidrugs._id =
6938978c9879dc15c6a1de6b`, `name = "Gammaked"`). The fuzzy match checks whether the
order's drug name is a case-insensitive substring of `brandDrugName` or `genericDrugName`.
`"gammaked"` is not contained in `"vedolizumab"`, so the match fails. The passthrough
(which would admit documents with no drug info at all) also does not apply because
`genericDrugName` is present. The document is correctly excluded.

The intake's `documentDefinitions[0].description.value` field (filled manually by the
person who processed this intake) reads `"Decision date after order creation date + drug
name is null"`, which confirms the date passed but the drug name was identified as the
blocking factor at processing time.

### Conclusion

is for **vedolizumab (Entyvio)** while the order is for **Gammaked** — two entirely different
drugs. The fuzzy match finds no substring overlap, and the passthrough did not apply because
`genericDrugName` was present.

However, this case motivated a fix: the drug filter was revised to use a **not-enough-evidence
passthrough** that fires when the intake has no `brandDrugName` and no `liDrugId`, and the
order has no `genericName` (so the generic-vs-generic comparison cannot be made). With no
basis for either comparison, the document is shown rather than hidden.
