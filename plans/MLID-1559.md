# MLID-1559 - Add Hospital System entry to main navigation menu

## Jira Task

**Summary:** Add Hospital System entry to main navigation menu
**Story Points:** 2
**Epic:** MLID-1021 (Hospital System Agreements)
**Deliverable:** D1 (Hospital List View)

## Context

The sidebar navigation is built in `apps/web/components/Layout/Sidebar/Sidebar.tsx`. Menu items are structured in 3 groups:

1. **Base links** (always visible): Home, Communications, Appointments, Users & Locations
2. **Conditional links** (feature-flag or permission gated): Drug Admin Panel, Orders Tracker, Chat, Intake, RxPreferred Claims
3. **Auditor/Admin links** (role-based): AI Chat Dashboard, Provider Management, Content Library, Insurance Plans, Administration

The current visual order renders as: `[...baseNavLinks, ...conditionalLinks, ...auditorLinks, ...adminLinks]`

### Existing patterns for reference

| Nav Item | Visibility Control | Uses Feature Flag? | Uses Permission Check? |
|----------|-------------------|:------------------:|:---------------------:|
| Drug Admin Panel | Feature flag only | Yes (`DRUG_ADMIN_PANEL`) | No (sidebar level) |
| Chat | Feature flag + admin role check | Yes (`CHAT_BUTTON`, `CHAT_BUTTON_ADMIN_ONLY`) | `isPrivileged` |
| Provider Management | Feature flag + admin/auditor role | Yes (`PROVIDER_MANAGEMENT`) | `isPrivileged \|\| isAuditor` |
| Insurance Plans | Admin/auditor role | No | `isPrivileged` |

### What Hospital Systems needs

- **Permission-based visibility** using `canManageHospitalSystems(userObj)` from `@/utils/permissions`
- Admin role, Auditor role, or User with Corporate position = **visible**
- All other users = **completely hidden** (not disabled)
- **No feature flag** needed (the epic branch strategy ensures it won't reach `develop` until all D1 tasks are complete)

### Session data availability

The session (`useSession()`) provides:
```typescript
session.user = {
  id: string;
  name: string;
  email: string;
  image: string;
  role: UserRole;
  userObj?: UserDTO;  // <-- has positions: UserPosition[]
}
```

`canManageHospitalSystems(userObj)` requires a `UserDTO`, which includes `positions`. The session callback (`apps/web/app/api/auth/[...nextauth]/route.ts`, line 137) populates `userObj` with the full user document from MongoDB.

If `userObj` is undefined (edge case), the nav item should be hidden (fail-closed).

#### 2. Sidebar changes (`Sidebar.tsx`)

**New imports:**


**Extract userObj from session (add after `isAuditor` line):**
```typescript
const userObj = session?.user?.userObj as UserDTO | undefined;
```

**Add Hospital Systems entry right after Drug Admin Panel block (line ~100):**
```typescript
if (userObj && canManageHospitalSystems(userObj)) {
  conditionalLinks.push({
    name: 'Hospital Systems',
    path: ROUTES.HOSPITAL_SYSTEM_AGREEMENTS,
    icon: '🏥',
  });
}
```

**Position rationale:** The Jira task says "between Drug Admin Catalog and Manage Users." In the current sidebar, Drug Admin Panel is the first conditional link. Placing Hospital Systems right after it puts it before Orders Tracker, Chat, and the admin section — which matches the intended position.

**Icon:** `🏥` (hospital/medical building emoji). The Jira task specifies "hospital/medical building icon." Note: `🏥` is also used for Provider Management in the auditor section, but they live in different permission groups and will never appear adjacent to each other for the same user type.

#### 3. Test updates (`Sidebar.test.tsx`)

**Mock updates needed:**
- Add `HOSPITAL_SYSTEM_AGREEMENTS: '/hospital-system-agreements'` to ROUTES mock
- Import/mock `canManageHospitalSystems` from `@/utils/permissions`

**New `describe('Hospital Systems Navigation')` block with tests:**

| # | Test | Session Mock | Permission Mock | Expected |
|---|------|-------------|-----------------|----------|
| 1 | Shows for admin user | `{ role: 'admin', userObj: { role: 'admin', ... } }` | returns `true` | Link present |
| 2 | Shows for auditor user | `{ role: 'auditor', userObj: { role: 'auditor', ... } }` | returns `true` | Link present |
| 3 | Shows for corporate user | `{ role: 'user', userObj: { role: 'user', positions: ['Corporate'] } }` | returns `true` | Link present |
| 4 | Hidden for regular user (no corporate) | `{ role: 'user', userObj: { role: 'user', positions: [] } }` | returns `false` | Link absent |
| 5 | Hidden when userObj is undefined | `{ role: 'user' }` (no userObj) | n/a | Link absent |
| 6 | Correct URL | admin session | returns `true` | href = `/hospital-system-agreements` |
| 7 | Correct icon | admin session | returns `true` | `🏥` present |
| 8 | Active state when on route | admin session, pathname = `/hospital-system-agreements` | returns `true` | `activeNavLink` class |

**Mock approach for `canManageHospitalSystems`:**

Option A: Mock the `@/utils/permissions` module directly.
Option B: Use real function + mock the session data with appropriate role/positions.

**Decision: Option B** — Use the real `canManageHospitalSystems` function. This tests the actual integration between the sidebar and the permission system. The function is pure (no side effects, no async) so there's no reason to mock it. We just need to provide the right `userObj` in the session mock.

However, looking at the existing sidebar tests, they don't import from `@/utils/permissions` at all — the permission checks currently use `hasAdminPermissions` and `isAdmin` from `@/utils/roles`, which the sidebar imports directly. Since `canManageHospitalSystems` is a new import that calls both `hasAdminPermissions` and `isCorporateEmployee`, we can let it run with the real implementation.

We do need to make sure the test doesn't fail because `isCorporateEmployee` or `hasAdminPermissions` aren't properly resolved. Since the sidebar test already uses the real `@/utils/roles` (it's not mocked), the real permissions module should work too.
