## Summary

Add shared infrastructure for the 340B and Pharmacy Eligibility features: two optional string fields on the `NewOrderDTO` type, the `ORDER_ELIGIBILITY` feature flag (default: `false`), and a migration script that creates 4 indexes on the `neworders` collection. This is pure foundation work — no business logic, no UI changes.

---

## Codebase Analysis

### 1. `NewOrderDTO` — `apps/web/types/orders.ts:65–73`

`NewOrderDTO` is a type alias extending `OrderBaseDTO`:

```typescript
type NewOrderDTO = OrderBaseDTO & {
  category: 'new';
  orderType: NewOrderType;
  patient: PatientDetails['_id'];
  drug: LIDrug['_id'] | string;
  provider: (ICmsNpiProvider['_id'] | undefined) | string;
  providerPracticeFax?: string;
  orderProtocol?: OrderProtocol['_id'];
};
```

The two new fields (`hospital340BEligible?: string`, `pharmacyEligible?: string`) are optional, so existing code compiles without changes. `neworders` is a raw MongoDB collection (no Mongoose model), so there is no schema file to update — the DTO is the source of truth for typing.

### 2. Feature flag — `apps/web/utils/featureFlags/featureFlags.ts`

Pattern:
- Add entry to `FeatureFlag` enum (line ~18–66)
- Add default value to `DEFAULT_FEATURE_FLAGS` record (line ~72–125)
- Default: `false` (disabled until explicitly enabled)

Existing test file: `apps/web/utils/featureFlags/featureFlags.test.tsx` — follows a pattern of testing `getDefaultFeatureFlagValue()` for specific flags. Add one test for `ORDER_ELIGIBILITY`.

### 3. Migration script — `apps/web/db-update/migration/`

Established pattern from `017.add-missing-indexes.mjs`:
- Numbered `.mjs` file, next available: **020**
- Uses `runDBUpdater` from `../db-update-runner/index.mjs`
- `trxOff: true` (MongoDB doesn't support transactions on index operations)
- Idempotent: checks existing indexes by name before creating
- Handles `IndexOptionsConflict` (error code 85) gracefully
- Includes `down()` rollback function using `dropIndex`
- Named indexes with `background: true`
- Summary logging with created/skipped/failed counts

No existing tests for migration scripts (confirmed: `apps/web/db-update/` has no test files). Migration correctness is validated by running it — the script itself is idempotent and has rollback.

### 4. Indexes to create on `neworders`

| Index | Name | Purpose |
|-------|------|---------|
| `{ provider: 1 }` | `provider_1` | 340B: hospital NPI triggers, patient insurance webhook |
| `{ patient: 1 }` | `patient_1` | 340B + Pharmacy: patient insurance triggers |
| `{ drug: 1 }` | `drug_1` | Pharmacy: drug flag trigger |
| `{ site: 1 }` | `site_1` | Pharmacy: location flag trigger |

Note: `neworders` already has a compound index `{ provider: 1, providerPracticeFax: 1 }` (from `ensureFaxMatchingIndexes`), but not a single-field `{ provider: 1 }` index. The single-field index is needed for queries that filter only by `provider`.
