## Summary

D1-T1–T3 established the 340B foundation, core calculation service, and inline triggers (order create/update). This task adds **background Pulse jobs** for two hospital-system-related triggers that cause fan-out recalculation across multiple orders:

1. **Trigger 3 — Hospital NPI list bulk update**: When NPIs are added/removed, recalculate 340B only for orders tied to *changed* NPIs (diff strategy).
2. **Trigger 5 — Hospital system rename**: When the system name changes, recalculate 340B for all orders tied to that system's NPIs (the stored value includes the hospital name).

---

## Codebase Analysis

- **Pulse job patterns**: Jobs are defined in `services/jobs/definitions/`, with handlers following the `(job, done)` signature. Job types are in `services/jobs/types.ts`. Registration happens in `definitions/index.ts` via `defineAllJobs()`. Immediate scheduling uses `pulse.now()` from `services/jobs/pulse.ts`.
- **`bulkUpdateProviders()`** (`services/mongodb/hospitalSystem.ts:~298`): Validates NPIs, deduplicates, then `findOneAndUpdate` replaces the entire providers array. Does NOT currently return the previous NPI list — we need to add a `findOne` before the update.
- **`updateHospitalSystem()`** (`services/mongodb/hospitalSystem.ts:~201`): Uses `findOneAndUpdate({ new: true })` — returns the updated doc, not the old one. The PUT route must fetch old state before calling this.
- **Hospital PUT route** (`app/api/hospitals/[id]/route.ts`): Already imports `getHospitalSystemById` — we can use it to fetch old state.
- **Bulk providers route** (`app/api/hospitals/[id]/providers/bulk/route.ts`): Calls `bulkUpdateProviders()` and returns updated hospital.
- **`compute340BForOrder()`** (`services/orderEligibility/orderEligibilityService.ts`): Existing orchestrator that fetches data and computes the 340B result. Takes `{ provider, patient }`.
- **`neworders` collection**: Raw MongoDB (no Mongoose model). Has `{ provider: 1 }` index from D1-T1 for efficient NPI-based queries.
- **`COLLECTION` constants** (`utils/constants.ts`): Has `NewOrders` constant for collection name.

## Implementation Steps

### Step 1 — Add job data types

- **File**: `apps/web/services/jobs/types.ts`
- **What**: Add `Recalculate340BByNpiJobData` (`affectedNpis: string[]`, `hospitalSystemId: string`) and `Recalculate340BRenameJobData` (`systemNpis: string[]`, `hospitalSystemId: string`, `newName: string`). Both extend `BaseJobData`.
- **No tests**: Pure type definitions (legitimate TDD exception).

### Step 2 — Create NPI update trigger function (TDD)

- **Files**: `apps/web/services/orderEligibility/triggers/hospitalNpiUpdate.ts` (new), `triggers/__tests__/hospitalNpiUpdate.test.ts` (new)
- **What**: `triggerHospitalNpiUpdate({ hospitalSystemId, oldNpis, newNpis })`:
  1. Compute diff: `added = new − old`, `removed = old − new`, `affected = added ∪ removed`
  2. If affected is empty → return `{ scheduled: false, affectedNpiCount: 0 }`
  3. Check `ORDER_ELIGIBILITY` flag → skip if disabled
  4. Call `scheduleImmediateJob('recalculate340BByNpi', { affectedNpis, hospitalSystemId })`
- **Tests**: Same list → no-op; added-only; removed-only; mixed; flag disabled → no schedule; correct job data

### Step 3 — Create hospital rename trigger function (TDD)

- **Files**: `apps/web/services/orderEligibility/triggers/hospitalRename.ts` (new), `triggers/__tests__/hospitalRename.test.ts` (new)
- **What**: `triggerHospitalRename({ hospitalSystemId, oldName, newName, systemNpis })`:
  1. If `oldName === newName` → no-op
  2. If `systemNpis` empty → no-op
  3. Check flag → skip if disabled
  4. Schedule `recalculate340BRename` job
- **Tests**: Same name → no-op; empty NPIs → no-op; name changed → job scheduled; flag disabled → skip

### Step 4 — Create Pulse job handlers (TDD)

- **Files**: `apps/web/services/jobs/definitions/orderEligibility.ts` (new), `definitions/__tests__/orderEligibility.test.ts` (new)
- **What**:
  - Shared helper `recalculate340BForOrders(npis, context)`: queries `neworders` with `{ provider: { $in: npis } }`, processes in batches of 100, calls `compute340BForOrder()` per order, `$set` result, per-order error handling
  - `recalculate340BByNpiHandler`: reads `affectedNpis` from job data, calls shared helper
  - `recalculate340BRenameHandler`: reads `systemNpis` from job data, calls shared helper
  - Both check feature flag at execution time (double-check: trigger checks before scheduling, handler checks before processing)
  - `defineOrderEligibilityJobs()`: registers both with Pulse
- **Tests**: Flag disabled → `done()` without processing; orders found and recalculated; per-order error → logged, continues; no orders → clean completion; job error → `done(error)`

### Step 5 — Register jobs

- **File**: `apps/web/services/jobs/definitions/index.ts`
- **What**: Import and call `defineOrderEligibilityJobs()` inside `defineAllJobs()`

### Step 6 — Modify `bulkUpdateProviders()` to return `previousNpis` (TDD)

- **Files**: `apps/web/services/mongodb/hospitalSystem.ts`, `__tests__/hospitalSystem.test.ts`
- **What**: Before the `findOneAndUpdate`, add a `findOne` to fetch current providers. Return `previousNpis: string[]` in the success result variant of `BulkUpdateProvidersResult`.
- **Tests**: Update existing tests for new return shape; verify `previousNpis` is populated correctly

### Step 7 — Wire NPI trigger into bulk providers API route (TDD)

- **Files**: `apps/web/app/api/hospitals/[id]/providers/bulk/route.ts`, `route.test.ts`
- **What**: After successful `bulkUpdateProviders()`, call `triggerHospitalNpiUpdate()` with `oldNpis: result.previousNpis`, `newNpis: sanitizedProviders.map(p => p.npi)`. Wrap in try-catch — trigger failure must not fail the API response.
- **Tests**: Trigger called with correct args; trigger error swallowed

### Step 8 — Wire rename trigger into hospital PUT route (TDD)

- **Files**: `apps/web/app/api/hospitals/[id]/route.ts`, `route.test.ts`
- **What**: In PUT handler, before `updateHospitalSystem()`, fetch current state with `getHospitalSystemById()` (already imported). After update, call `triggerHospitalRename()` with old name, new name, and system NPIs. Wrap in try-catch.
- **Tests**: Trigger called when name changes; not called when same name; trigger error swallowed

### Manual / E2E Verification

This task introduces Pulse background jobs. To test end-to-end locally, you need to run the worker process alongside the web app.

**Local setup** (`.env.local` already has `PULSE_COLLECTION=pulseJobsAGomez`):

```bash
# Terminal 1 — web app (already running)
npm run dev:web

# Terminal 2 — worker (reads same .env.local)
cd apps/web && npm run worker
```

Both processes load `.env.local`, so both use the `pulseJobsAGomez` collection. The Azure staging workers use `pulseJobs` — no interference between your local testing and the shared environment.


**Test steps:**

1. Start both terminals (web app + worker)
2. Toggle `ORDER_ELIGIBILITY` feature flag on (admin panel)
3. **NPI update trigger**: Go to a hospital system → update its NPI list (add or remove an NPI)
   - Worker logs: `"Scheduled 340B recalculation for NPI changes"` → `"recalculate340BByNpi: completed"`
   - Verify affected orders in `neworders` have updated `hospital340BEligible`
4. **NPI no-op**: Re-upload the same NPI list without changes
   - Worker should NOT log any recalculation (diff is empty)
5. **Rename trigger**: Edit a hospital system name
   - Worker logs: `"Scheduled 340B recalculation for hospital rename"` → `"recalculate340BRename: completed"`
   - Verify affected orders show the new hospital name in `hospital340BEligible`
6. **Rename no-op**: Save hospital system without changing the name
   - Worker should NOT log any recalculation
7. **Feature flag off**: Disable `ORDER_ELIGIBILITY` and repeat steps 3/5
   - No jobs should be scheduled
