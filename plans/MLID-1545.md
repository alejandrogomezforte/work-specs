## Files to Create

### 1. `apps/web/app/api/hospitals/[id]/providers/bulk/route.ts`

**PUT handler** following the exact same pattern as the existing `[id]/route.ts` and `[id]/providers/route.ts`:

```
1. Auth check       → 401 if no session
2. Permission check → 403 if !canManageHospitalSystems
3. connectToDatabase()
4. Validate :id     → 400 if not valid ObjectId
5. Parse body       → 400 if providers is not an array
6. Validate items   → 400 if any item missing `npi` field or npi is not a string
7. Call bulkUpdateProviders(id, providers, userEmail)
8. Handle result:
   - { success: false, invalidNPIs } → 400 with invalidNPIs
   - { success: true, data: null }   → 404 not found
   - { success: true, data }         → 200 with { data }
9. catch → 500
```

**Request body:**
```json
{
  "providers": [
    { "npi": "1234567890", "name": "Dr. Smith" },
    { "npi": "0987654321" }
  ]
}
```

**Response cases:**
| Status | Condition | Body |
|--------|-----------|------|
| 200 | Success | `{ data: HospitalSystemDocument }` |
| 400 | Invalid ObjectId | `{ error: "Invalid hospital system ID" }` |
| 400 | Body not array | `{ error: "providers must be an array" }` |
| 400 | Item missing npi | `{ error: "Each provider must have a valid 'npi' string field" }` |
| 400 | NPI format invalid | `{ error: "Invalid NPI format", invalidNPIs: ["123", "abc"] }` |
| 401 | No session | `{ error: "Unauthorized" }` |
| 403 | No permission | `{ error: "Forbidden: Admin, Auditor, or Corporate access required" }` |
| 404 | Hospital not found | `{ error: "Hospital system not found" }` |
| 500 | Server error | `{ error: "Failed to bulk update providers" }` |

## Validation Strategy

**Two-layer validation:**
1. **Route layer** (structural): Checks that `providers` is an array and each item has `npi` as a string. This catches malformed requests before hitting the service.
2. **Service layer** (business): `bulkUpdateProviders` validates NPI format (`/^\d{10}$/`) and returns specific `invalidNPIs`. Already tested in service tests.

This separation means the route returns early for obviously bad input (missing field, wrong type) without touching the DB, while the service handles domain-specific NPI validation.
