## Jira Task

**Deliverable:** D1 (Hospital List View)

## Context

The hospital list page (`apps/web/app/hospital-system-agreements/page.tsx`) already has a working paginated table with URL-based filter state. The backend **already fully supports search** — the API route accepts a `search` query param, the MongoDB service filters by `name` using case-insensitive regex with `escapeRegex()`, and the client service (`hospitalSystemService.ts`) already includes `search` in `HospitalSystemsListParams`.

The only missing piece is the **frontend**: the page component has no search input field, the `Filters` interface doesn't include `search`, and `getFiltersFromUrl()` doesn't read a `search` param.

## Pattern Analysis

Before designing the implementation, we compared search patterns from two existing pages with similar functionality:

| Aspect | Locations (`/users-locations?tab=location`) | Intakes (`/intakes`) | Hospital Systems (this plan) |
|--------|----------------------------------------------|----------------------|------------------------------|
| **Input component** | Native `<input>` + MUI `SearchIcon` | `StyledInput` (native input wrapper) | Native `<input>` + MUI `SearchIcon` |
| **Clear button** | None | None | None (per Figma design) |
| **Debounce** | Yes — inline `setTimeout` (500ms) | None — fires every keystroke | Yes — inline `setTimeout` (500ms) |
| **Search state** | Custom `useLocations` hook (`searchTerm` + `debouncedSearchTerm`) | `filters.search` in page state | `searchInput` state + inline debounce in page component |
| **URL persistence** | No | Yes — `?search=` in URL | Yes — `?search=` in URL |
| **Pagination reset** | `useEffect` on debounced term change | Explicit `page: 1` in `updateFilters` | Explicit `page: 1` in `updateFilters` |
| **Placeholder** | "Search by location name, facility name, city, NPI or Tax ID" | "Search by patient name or ID..." | "Search" |
| **Layout position** | Full-width row above table | Full-width, first in controls column | Inside Card, row with "+ Add new" button |
| **Input width** | `1807px` (max-width: 100%) | `100%` (via StyledInput) | Flexible (flex: 1), shares row with button |
| **Styling** | CSS modules (custom border, focus states) | CSS modules (purple focus ring) | CSS modules (matching locations pattern) |
| **State update flow** | `setSearchTerm` → debounce → `debouncedSearchTerm` → `loadLocations` | `updateFilters` → `updateUrl` + `loadIntakes` | `setSearchInput` → debounce → `updateFilters` → `updateUrl` + `loadHospitalSystems` |

**Decision:** Follow the **locations tab** pattern for the input component and debounce (native input + SearchIcon, 500ms inline setTimeout), and the **intakes page** pattern for URL persistence and state flow (search in URL params, `updateFilters` with explicit `page: 1` reset).

## Implementation Details

### 1. Update `Filters` interface (`page.tsx`)

Add `search` field:
```typescript
interface Filters {
  page: number;
  pageSize: number;
  sortBy: string;
  sortOrder: 'asc' | 'desc';
  search: string;
}
```

### 2. Update `getFiltersFromUrl()` (`page.tsx`)

Read `search` param from URL:
```typescript
function getFiltersFromUrl(searchParams: URLSearchParams): Filters {
  return {
    page: parseInt(searchParams.get('page') || '1'),
    pageSize: parseInt(searchParams.get('pageSize') || '25'),
    sortBy: searchParams.get('sortBy') || 'name',
    sortOrder: (searchParams.get('sortOrder') as 'asc' | 'desc') || 'asc',
    search: searchParams.get('search') || '',
  };
}
```

### 3. Update `updateUrl()` (`page.tsx`)

Include `search` in URL (only when non-empty and trimmed, to keep URLs clean):
```typescript
if (newFilters.search && newFilters.search.trim()) {
  params.set('search', newFilters.search.trim());
}
```

### 4. Add search input with debounce (`page.tsx`)

**Component approach:** Native HTML `<input>` with MUI `SearchIcon` positioned absolutely inside (same pattern as the locations tab in `/users-locations`). This keeps visual consistency with existing pages and avoids introducing MUI `TextField` into a page that uses CSS modules throughout.

**Debounce approach:** Inline `setTimeout` with cleanup (500ms delay), matching the locations tab pattern. The component maintains a local `searchInput` state for immediate UI response, and a debounced effect triggers the API call.

**Flow:**
1. User types in search input → `searchInput` state updates immediately (responsive UI)
2. `useEffect` with `setTimeout` (500ms) debounces the value
3. After 500ms of inactivity, call `updateFilters({ search: debouncedValue, page: 1 })`
4. This updates URL + triggers API call with `search` param
5. Pagination resets to page 1 on every new search

### 5. Add toolbar layout inside the Card (`page.tsx` + `styles.module.css`)

Add a toolbar row as the first child inside the Card, containing the search input and the future "+ Add new" button (MLID-1550).

CSS additions:
- `.toolbar` — flex container, `align-items: center`, `gap: 16px`, `margin-bottom: 16px`, `padding: 16px 16px 0`
- `.searchWrapper` — `position: relative`, `flex: 1` (takes remaining width)
- `.searchInput` — native input styles matching locations pattern: full width, `height: 40px`, `padding-left: 40px` (room for icon), border, border-radius, focus states
- `.searchIcon` — `position: absolute`, `left: 12px`, `top: 50%`, `transform: translateY(-50%)`, color `#71717a`

### 6. Update "no results" message

Keep current message "No hospital systems found." — works for both empty state and empty search results.

### 7. Sync initial URL search param

On page load, if URL has `?search=foo`, pre-fill the search input with that value. This is handled by initializing `searchInput` from `getFiltersFromUrl()`.
