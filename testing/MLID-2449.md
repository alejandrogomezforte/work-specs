## Why seed `auditLogs` directly

The History tab renders rows from the `auditLogs` collection, queried by
`{ entityType: 'order', entityId: '<orderId>' }`. Those rows are normally
written by `writeOrderDocumentAuditEntry` (step 3 of the propagation
orchestrator). The maintenance triggers that call it are delivered later in
**D4 (MLID-2450–2454)**, so today there is **no live maintenance audit data**.
Until D4 lands, verification requires seeding `auditLogs` rows by hand.

## Persisted `auditLogs` shape (authoritative)

`writeAuditLog` (`apps/web/services/mongodb/auditLog.ts`) inserts exactly:

```js
{
  entityType: 'order',
  entityId: '<orderId string>',
  action: 'update',
  actorId: '<userId string, or "" for system>',
  source: '<audit source: user | system | webhook | job>',
  timestamp: <Date>,
  changes: [
    { field: 'documents', from: <obj|null>, to: <obj|null> },
  ],
}
```

Notes that the seed below honors:

- The sort/render field is **`timestamp`** (a `Date`), not `createdAt`.
- Top-level **`source`** is the **audit** source (`system` / `user`). The
  **document** source (`Upload` / `E-Order` / `Fax`) lives inside the change at
  **`to.source`** (`upload` / `e-order` / `fax`).
- The "Change" column text is produced server-side by the `documents` formatter
  in `auditLog.config.ts` from the document `category` inside `to` / `from`:
  - `category === 'Order'`, `from === null` → **A Document has been added (Document type = Order)**
  - `category === 'Order'`, `to === null` → **A Document has been removed (Document type = Order)**
  - any other category → **New Document received (Document type = <category>)**

The seed rows match the persisted shape exactly — no extra fields. Cleanup is
scoped by the order's `entityId` plus `changes.field: 'documents'`. This is safe
because the maintenance triggers (D4) have not landed, so this order currently
has **no** real `documents` audit rows; the only rows that match the cleanup
filter are the ones seeded here. Once D4 lands and produces real document audit
history on this order, narrow the cleanup (for example by the specific seeded
document ids) before running it.
