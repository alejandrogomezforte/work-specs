# MLID-2452 — Propagate Notifications when a Document is Linked to a Maintenance Order — Testing

Feature: when a document is **linked** to a **maintenance** order (via the
Link Orders flow), the notification propagation must fire for that maintenance
order — a notification row targeting the order's assignee, an `auditLogs` entry,
and a `documentAdded` SignalR broadcast. D4-T3 of epic MLID-2417.

This is the link-time trigger, the counterpart to the upload-time trigger tested
in MLID-2450. It exercises `PUT /api/documents/[documentId]/link-orders` →
`handleOrderLink`.

## What changed (under test)

**No production code change.** `handleOrderLink`
(`apps/web/services/notifications/documentNotification.ts`) already resolves each
linked order with a fallback — `getOrderById(id, 'new') ?? getOrderById(id, 'maintenance')`
— and hands it to the collection-agnostic
`propagateDocumentNotificationToOrder`. So linking to a maintenance order already
propagates. MLID-2452 added the missing **test coverage** that proves the full
contract fires when the resolved order lives in `maintenanceorders`.

This manual run confirms the same path end-to-end in the UI.

> Note: a later task (D5-T1 / MLID-2628) refactors away the `'new' ?? 'maintenance'`
> guess and threads the order category through the request. This test documents
> the current fallback behavior.

## Prerequisites

1. **Feature flag ON.** `ORDER_DOCUMENT_NOTIFICATIONS` must be **ON**
   (`/admin/feature-flags`). With it OFF, `handleOrderLink` returns before any
   lookup and nothing propagates — that is the OFF case in Test 4.

2. **The user who clicks "Link" must NOT be the maintenance order's assignee.**
   There is a pre-existing rule in `propagateDocumentNotificationToOrder`
   (MLID-2434, commit `9ad4efc8d`, epic MLID-2444): when the actor performing the
   action is the order's assignee, the in-app **notification** is suppressed (the
   audit entry and SignalR broadcast still run). Product added this so a user is
   not notified about a document they handled themselves.

   Consequence for this test: the notification always targets the order's
   **assignee**. To see a "New" chip appear, perform the link as a user who is
   **not** the assignee, then check the result while logged in as the assignee.
   If you link as the assignee, you will not see a chip — that is the suppression
   working, not a failure.
