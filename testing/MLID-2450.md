# MLID-2450 — Propagate Notifications to Maintenance Orders on Intakes Upload — Testing

Feature: when a document is uploaded through the **intakes-review** flow
(split-pdf path), notifications must fan out to every **maintenance** order on
the document's patient — not just new orders. D4-T1 of epic MLID-2417.

Unlike MLID-2449 (which seeded `auditLogs` rows by hand), this is a **true
end-to-end test of the new code path**: the upload itself must produce the
`notifications` row. Nothing is seeded — we trigger the real flow and confirm
the row the code wrote.

## What changed (under test)

- `getOrdersByPatient` lean path (`apps/web/services/mongodb/ordersTracker.ts`)
  now accepts a `collections` option; default stays `['new']`.
- `notifyDocumentCreated` (`apps/web/services/notifications/documentNotification.ts`)
  now calls it with `collections: ['new', 'maintenance']`, so upload-time
  fan-out reaches maintenance orders.

This same change also covers D4-T2 (e-order / fax), because that background job
routes through the same `notifyDocumentCreated`.

## How to upload a document via Intake Review (detailed walkthrough)

This is the exact click-by-click flow. Tests 1 and 2 reuse it and only differ
in which document **section/Type** you choose. Substitute `[id]` with the intake
`_id` from the Prerequisites query.

The intake-review wizard is 3 steps, then a submission dialog. Source: the
`document-intake` module under `apps/web/components/Modules/document-intake/`;
page at `apps/web/app/intakes/[id]/review/page.tsx`.

**Open the review wizard**

1. Go to `/intakes`.
2. Click the row for your `spruceFax`/`eOrder` intake matched to patient `9562`
   (opens `/intakes/[id]`).
3. Click the **"Review Documents"** button (shown only for `spruceFax`/`eOrder`).
   This opens `/intakes/[id]/review`.

**Step 1 of 3 — PDF Viewer**

4. The intake PDF loads in a viewer. Note which pages hold which content.
5. Click **"Next: Assign Sections"** (footer button).

**Step 2 of 3 — Assign page ranges** (heading: "Assign page ranges to each section:")

6. You see a checklist of sections: "Referral Copy", "Order", "Insurance Cards",
   "Supporting Medical Document", "Authorization Copy", "Labs", "Demographics",
   "Other".
7. Check the section(s) you want to upload (which ones depends on the test —
   see Test 1 / Test 2 below).
8. For each checked section **except "Referral Copy"**, a page-range row appears:
   enter a start page in the **"Page"** input and an end page in the **"to"**
   input (for example `1` to `1`). Use **"+ Add page range"** if a section spans
   non-contiguous pages. ("Referral Copy" shows a **"Full Document"** badge and
   needs no page range — it uploads the whole PDF.)
   - Note: "Referral Copy" → backend **Path A** (full-document upload). Every
     other section → backend **Path B** (split-page upload). Both paths call the
     propagation; you do not press anything different — the section choice
     decides the path.
9. The footer shows progress like "N / N selected types configured". When every
   checked section has a valid page range, click **"Next: Review Data"**.

**Step 3 of 3 — Review Data** (one tab per checked section)

10. For each section tab, fill the required fields. For non-Order sections the
    form has: **"Description"** (text), **"Date Received"** (date, no future
    dates), **"Type"** (dropdown), **"From"** (text). The "Order" section shows
    an order-specific form instead.
11. The **"Type"** dropdown options (from `apps/web/constants/documentCategories.ts`)
    are: "Referral", "Order", "Insurance Card", "Medical Record", "Supporting
    Medical Document", "Insurance Authorization", "Lab Results", "Demographics",
    "Other", "Fax", "Document". Pick per the test.
12. Each tab shows "⏳ N / N fields verified" until complete, then "✅". Fill
    every checked tab until all show ✅.
13. Click **"Complete Intake"** (footer button; enabled once required sections
    are complete).

**Submission dialog — "Processing Intake"**

14. A modal titled **"Processing Intake"** opens ("Ready to Submit Intake").
15. Click **"Submit Intake"**. The steps run with ⟳ → ✓ (or ✗ on failure). The
    `POST /api/documents/split-pdf` call fires here, once per configured section.
16. When all rows show ✓, click **"Done"**. The dialog closes and the intake
    status becomes `processed`. (There is no toast — the all-green ✓ list is the
    success signal.)

> Note: "Order" and "Insurance Cards" may be required sections to complete the
> wizard. If so, a single submission produces several documents at once — that
> is fine. The verification queries below distinguish them by category, so you
> can run Test 1 and Test 2 from the same submission.

## Verification Result (2026-06-22)

✅ **Pass — maintenance fan-out on intakes upload works end-to-end.**

### Key gotcha discovered (MLID-2434 rule)

There is a pre-existing rule in `propagateDocumentNotificationToOrder` (added by
Egor under MLID-2434, commit `9ad4efc8d`, part of epic MLID-2444): **if the
uploader is the order's assignee, the in-app notification is suppressed** (the
audit entry and SignalR broadcast still run). Product added this so a user is
not notified about a document they uploaded themselves.

Consequence for testing: uploading as the test account `695c25a530f7a75a8b1ed8c8`
— which is also `0yn`'s assignee — produced **no notification row** for `0yn`
(by design). The first two upload attempts looked like a failure for exactly
this reason. **To see a maintenance notification, the uploader must be a
different user than the order's assignee.**

### What confirmed the feature

1. **Stale-server false negative first.** The initial upload ran the
   pre-change code (server not yet restarted) and produced neworders-only
   notifications. After restarting the dev server on the feature branch, the
   `[Notification Creation]` logs showed `collections: ['new','maintenance']`
   and `maintenanceorders count: 1`, with `0yn`
   (`6a34840be879c9374538c919`) in the returned order set.

2. **Assignee-uploader skip confirmed.** Uploading as the assignee
   (`695c…`) logged `skipNotificationForAssigneeUploader: true` for `0yn` —
   audit + SignalR ran, no notification row. Working as the MLID-2434 spec
   intends.

3. **Notification confirmed with a different uploader.** Re-uploaded a
   non-Order document (`6a39ef723d4549bae0be8dbb`, category Prior
   Authorization) via intake `6a39ca418c5a3e85d6c4ea02` while logged in as the
   secondary account `agomez@mylocalinfusion.com` (≠ `0yn`'s assignee). The
   `[Notification Creation]` log showed `Notification row created` for
   `orderId: 6a34840be879c9374538c919`, `targetUserId: 695c25a530f7a75a8b1ed8c8`.

4. **UI confirmed.** On `/orders-tracker/maintenance/0yn/documents` the
   Documents-tab badge showed unread count and the uploaded doc ("Copay claim
   approval for AVSOLA…") rendered the **"New" chip** and **"Mark as Read"**
   button. Screenshot: `Screenshot 2026-06-22 203229.png`.

### Notes

- D4-T2 (e-order / fax) is covered by the same change — that path routes
  through the same `notifyDocumentCreated` → `getOrdersByPatient` chain.
- The `[Notification Creation]` logs were kept (renamed from the temporary
  trace tag) as durable production observability for the notification chain.
