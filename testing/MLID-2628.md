# MLID-2628 (D5-T1) — Manual Testing Guide

**Branch:** `feature/MLID-2628-link-orders-category-passthrough`
**Change type:** Refactor (write-path). Threads the order category end-to-end through the document-link flow so `handleOrderLink` resolves the order in a single `getOrderById(id, category)` call instead of probing `new` then falling back to `maintenance`. The cross-collection guess is deleted.

Because this is a refactor, the user-visible behavior should be the **same as before**. The real proof is that linking a **maintenance** order still works and that the request now carries the category. Document `linkedOrderIds` storage stays a bare `string[]` (unchanged); the `/api/intakes/[id]/link-orders` route is untouched.

## Prerequisites

1. **Be on the branch with a fresh server.** Start the web app so it picks up the new code (e.g. `npm run dev:web`).
2. **Feature flags ON** (at `/admin/feature-flags`):
   - `ORDER_DOCUMENT_NOTIFICATIONS` — gates the link → notification propagation. Without it, linking works but no badge/chip appears.
   - `ORDER_LINK_INTAKE` — needed only for Test B (the intake order-option list).
3. **Linker must NOT be the order's assignee.** An existing rule (MLID-2434) suppresses the in-app notification when the person doing the action is the order's assignee. To see the "New" chip / badge, use an order assigned to a *different* user than the one you are logged in as.
4. Have ready:
   - a **maintenance order** that has an assignee, and
   - an **Order-category** document for that same patient (the route rejects non-Order documents with 422).

## The key thing to verify (the actual change)

Open browser **DevTools → Network** before saving. The change is the request shape sent to `PUT /api/documents/{documentId}/link-orders`. It must now be:

```json
{ "linkedOrders": [ { "id": "<orderId>", "category": "maintenance" } ] }
```

Previously it was:

```json
{ "linkedOrderIds": ["<orderId>"] }
```

Seeing `category` in the payload, plus a successful response, confirms the category is threaded through. Server-side this means the maintenance order resolves in one lookup instead of two — the payload shape is the clean client-side proof.

## Test B — Document intake submission (sender 2: `useSubmissionProcessor`)

This is the *other* place that links a document to an order. It exercises a
different code path than Test A: `useSubmissionProcessor` builds the
`{ id, category }` payload, and (for the update flow) `UpdateOrderForm`
persists the selected order's category. The intake submission is a heavier
setup than Test A — it is the document-intake review wizard, not a single
modal.

> Note: the processor + UpdateOrderForm logic is already covered by automated
> tests, so Test B is optional belt-and-suspenders. The most valuable variant
> is the **Update flow with an existing maintenance order**, because that is
> the path unique to this change.

### Preconditions for Test B

- **`ORDER_LINK_INTAKE` flag ON** — this is the master switch. With it off, the
  "Order Type & Review" section and all the LISA/link steps do not appear at all.
- **`ORDER_DOCUMENT_NOTIFICATIONS` flag ON** — only needed if you also want to
  see the notification/badge on the order side afterward (the link itself runs
  without it).
- **An intake of type `spruceFax` or `eOrder`** — the "Review Documents" button
  only renders for those types.
- **The patient must be matched** (have a WeInfuse patient id) for the Update
  flow's existing-order dropdown to populate; otherwise it shows "Patient has
  no existing orders".
- **Linker ≠ assignee** for the notification to appear (MLID-2434 rule) — log
  in as the secondary account `agomez@mylocalinfusion.com` if the target order
  is assigned to `alejandro.gomez@fortegrp.com`.

### Navigation to the submission UI

1. Go to **`/intakes`**.
2. Click an intake **row** (type `spruceFax` or `eOrder`) → lands on `/intakes/{id}`.
3. Click the **"Review Documents"** button → lands on `/intakes/{id}/review`. This
   opens the 3-step wizard ("Step N of 3").
4. **Step 1 — Patient Lookup:** match/confirm the patient.
5. **Step 2 — Document Definitions:** make sure there is a document defined with
   type **Order** (the whole link branch is gated on an Order-type document
   being present).
6. **Step 3 — Review Data:** on the left, select the **order** tab. You will see
   a card titled **"Order Type & Review"**.

### Choosing the order type (in the "Order Type & Review" card)

Under the **"Select Order Type"** label there is a row of three pills
(a tablist, not radio buttons):

- **"Create New Order"**
- **"Maintenance Order"**
- **"Update Order"**

**Variant 1 — Update flow (recommended, exercises the new code):**
1. Click the **"Update Order"** pill.
2. In the **"Select Existing Order"** dropdown, pick an existing **maintenance**
   order (e.g. `010w`). If the dropdown is empty, the patient is not matched —
   fix Step 1 first.

**Variant 2 — Create flow:**
1. Click the **"Maintenance Order"** pill.
2. Fill in the required drug selection (and any other required order fields).

### Submitting

1. At the bottom of the Review Data step, click **"Complete Intake"**.
2. A dialog titled **"Processing Intake"** opens, showing the steps that will
   run. Click **"Submit Intake"**.
3. Watch the step list. Confirm the step titled **"Link Document to LISA Order"**
   runs and completes (✓).

### What to inspect in Network (the MLID-2628 proof)

With **DevTools → Network** open (filter on `link-orders`), the submission fires:

```
PUT /api/documents/{documentId}/link-orders
```

Confirm the request **payload** carries the category:

- **Variant 1 (Update, maintenance order):**

  ```json
  { "linkedOrders": [ { "id": "<existingOrderId>", "category": "maintenance" } ] }
  ```

  This is the key proof: the category comes from the `selectedExistingOrderCategory`
  that `UpdateOrderForm` persisted when you picked the order — the new code in
  this task.

- **Variant 2 (Create, maintenance):** the payload category is also
  `"maintenance"`, sourced from the selected order type (`selectedType`).

4. (Optional) If `ORDER_DOCUMENT_NOTIFICATIONS` is on and you submitted as the
   secondary account, confirm the **"New" chip** / badge appears for the order's
   assignee, same as Test A.

## Optional — validation (negative cases)

These are covered by automated tests, but can be checked with a direct request if desired:

- A request body with the legacy `{ "linkedOrderIds": [...] }` shape (no `linkedOrders`) → **400**.
- A `linkedOrders` entry with an invalid 24-char hex `id` → **400**.
- A `linkedOrders` entry with a `category` other than `"new"` / `"maintenance"` → **400**.
- A non-Order document → **422** (unchanged behavior).
